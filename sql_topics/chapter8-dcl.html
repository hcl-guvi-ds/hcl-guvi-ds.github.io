Chapter 8

DCL — Data Control Language

Chapter 8 · Data Control Language

Who Can See What?
Control Your Database Like a Pro

DCL commands decide who can access what in your database. Think of it as the security guard of SQL — granting access to those who need it, and revoking it when they don't.

What is DCL?

DCL stands for Data Control Language. It is the set of SQL commands used to control access permissions on database objects like tables, views, and procedures.

✅
GRANT
Give specific permissions to a user or role — like allowing an analyst to SELECT from a table.
🚫
REVOKE
Take back permissions from a user or role — useful when someone leaves a team or changes role.
👤
CREATE USER
Add a new database user with a username and password so they can log in.
🎭
Roles
Group permissions under a named role (like "analyst") and assign that role to multiple users at once.
🎓
Real-world analogyAt HCL GUVI, the HR team can view student records but can't edit course content. The tech team can deploy code but can't touch payroll. DCL is exactly how these access boundaries are enforced in databases.
🗂️ DCL vs Other SQL Categories
CategoryCommandsPurpose
DDLCREATE, ALTER, DROPStructure of database
DMLSELECT, INSERT, UPDATE, DELETEData manipulation
TCLCOMMIT, ROLLBACKTransaction control
DCLGRANT, REVOKEAccess control ← You are here
💡
Why does this matter?In production databases at companies like Zomato, IRCTC, or HDFC Bank, thousands of users and applications connect to the database. Without proper access control, a junior intern could accidentally (or intentionally!) delete financial records. DCL prevents that.

👤 CREATE USER & DROP USER

Before you grant any permissions, you first need to create users in your database. Think of it as registering people into the system.

🆕 CREATE USER — Syntax

Creates a new database user with a username and password.

MySQL / MariaDB 
CREATE USER 'username'@'host' IDENTIFIED BY 'password';
🏢 HCL GUVI Example — Creating Team Users
SQL 
-- Create analyst user (read-only)
CREATE USER 'priya_analyst'@'localhost'
  IDENTIFIED BY 'SecurePass@123';

-- Create developer user (read + write)
CREATE USER 'ravi_dev'@'localhost'
  IDENTIFIED BY 'DevPass@456';

-- Create admin user (full access)
CREATE USER 'admin_guvi'@'%'
  IDENTIFIED BY 'Admin@SuperSecure!';
⚠️
The @'host' part matters! 'user'@'localhost' → can only connect from the same machine.
'user'@'%' → can connect from anywhere (use with caution!).
'user'@'192.168.1.10' → can connect only from that specific IP.
🔍 Check Existing Users
SQL 
-- See all users in MySQL
SELECT user, host FROM mysql.user;

-- Check grants for a specific user
SHOW GRANTS FOR 'priya_analyst'@'localhost';
🗑️ DROP USER — Remove a User

When an employee leaves, always drop their user account to revoke all access.

SQL 
-- Remove user completely
DROP USER 'priya_analyst'@'localhost';

-- Drop only if exists (no error if user doesn't exist)
DROP USER IF EXISTS 'ravi_dev'@'localhost';
🚨
Security Best PracticeNever use root for application connections. Always create a dedicated user with only the permissions that application needs. This is called the Principle of Least Privilege.

✅ GRANT — Give Permissions

GRANT is how you give users permission to perform specific actions on database objects. Always be specific — grant only what's needed.

📋 GRANT Syntax
Syntax 
GRANT privilege_list
  ON database_name.table_name
  TO 'user'@'host'
  [WITH GRANT OPTION];
🏢 GUVI Example — Granting Privileges
SQL 
-- Allow analyst to only read student data
GRANT SELECT
  ON guvi_db.students
  TO 'priya_analyst'@'localhost';

-- Allow developer to read and write courses
GRANT SELECT, INSERT, UPDATE, DELETE
  ON guvi_db.courses
  TO 'ravi_dev'@'localhost';

-- Allow admin full access to entire GUVI database
GRANT ALL PRIVILEGES
  ON guvi_db.*
  TO 'admin_guvi'@'%';

-- Apply immediately
FLUSH PRIVILEGES;
🔑 Available Privileges
PrivilegeTypeWhat it allows
SELECTReadRead / query data
INSERTWriteAdd new rows
UPDATEWriteModify existing rows
DELETEWriteRemove rows
CREATEDDLCreate tables/databases
DROPDDLDelete tables/databases
ALTERDDLModify table structure
INDEXDDLCreate/drop indexes
EXECUTEWriteRun stored procedures
ALL PRIVILEGESAdminEverything (use sparingly!)
GRANT OPTIONAdminCan grant their privileges to others
🔗 GRANT with GRANT OPTION

This special option lets a user pass on their privileges to other users. Use it carefully — only for trusted senior users.

SQL 
-- Ravi can now give SELECT access to others
GRANT SELECT
  ON guvi_db.students
  TO 'ravi_dev'@'localhost'
  WITH GRANT OPTION;
💡
ON *.* vs ON db.* vs ON db.table
ON *.* → all databases, all tables (global)
ON guvi_db.* → all tables in guvi_db
ON guvi_db.students → only students table

🚫 REVOKE — Take Back Permissions

REVOKE is the opposite of GRANT. It removes previously granted privileges from a user. It does not delete the user — just takes back specific access rights.

📋 REVOKE Syntax
Syntax 
REVOKE privilege_list
  ON database_name.table_name
  FROM 'user'@'host';
🏢 GUVI Example — Revoking Privileges
SQL 
-- Priya moved teams — remove her write access
REVOKE INSERT, UPDATE, DELETE
  ON guvi_db.courses
  FROM 'priya_analyst'@'localhost';

-- Ravi left company — remove ALL his privileges
REVOKE ALL PRIVILEGES
  ON guvi_db.*
  FROM 'ravi_dev'@'localhost';

-- Revoke Grant Option from Ravi
REVOKE GRANT OPTION
  ON guvi_db.students
  FROM 'ravi_dev'@'localhost';
⚠️
REVOKE doesn't cascade automatically in MySQLIf Ravi used GRANT OPTION to give Priya access, revoking Ravi's privilege does NOT remove Priya's access in MySQL. You must revoke from each user explicitly. PostgreSQL handles this differently with CASCADE.
⚖️ GRANT vs REVOKE — Side by Side
AspectGRANTREVOKE
PurposeGive permissionsTake back permissions
KeywordTO userFROM user
EffectUser gains accessUser loses access
User deleted?NoNo — only access removed
Best practiceGrant minimum neededRevoke when role changes

🎭 Roles & Role Hierarchies

Instead of granting permissions to each user individually, you can group permissions into a Role and assign that role to multiple users. Much easier to manage!

💡
Think of it like job titlesAt GUVI, the "Analyst" role can view reports. The "Developer" role can also edit data. Instead of setting 50 permissions per person, you assign them a role — and change it once if the role changes.
🆕 CREATE ROLE & Assign
MySQL 8+ / PostgreSQL 
-- Step 1: Create the roles
CREATE ROLE 'analyst';
CREATE ROLE 'developer';
CREATE ROLE 'db_admin';

-- Step 2: Grant privileges to the role
GRANT SELECT
  ON guvi_db.*
  TO 'analyst';

GRANT SELECT, INSERT, UPDATE, DELETE
  ON guvi_db.*
  TO 'developer';

GRANT ALL PRIVILEGES
  ON guvi_db.*
  TO 'db_admin';

-- Step 3: Assign roles to users
GRANT 'analyst'  TO 'priya_analyst'@'localhost';
GRANT 'developer' TO 'ravi_dev'@'localhost';
GRANT 'db_admin'  TO 'admin_guvi'@'%';

GUVI Role Architecture

🔗 Role Hierarchies — Roles within Roles

In PostgreSQL, you can even grant a role to another role — creating a hierarchy.

PostgreSQL 
-- developer inherits all analyst permissions too
GRANT analyst TO developer;

-- db_admin inherits developer (which includes analyst)
GRANT developer TO db_admin;

-- Now db_admin has everything:
-- analyst permissions + developer permissions + admin privileges
🔍 SHOW GRANTS — Check Who Has What
MySQL 
-- Check grants for a user
SHOW GRANTS FOR 'priya_analyst'@'localhost';

-- Check grants for a role
SHOW GRANTS FOR 'analyst';

-- Check current user's own grants
SHOW GRANTS;

🔐 Privilege Levels

SQL lets you grant privileges at different levels of granularity — from the entire server down to a single column.

  • 1
    Global Level — ON *.*

    Applies to ALL databases on the server. Only for super admins.

    GRANT ALL PRIVILEGES ON *.* TO 'superadmin'@'localhost';
  • 2
    Database Level — ON db.*

    Applies to all tables within a specific database.

    GRANT SELECT, INSERT ON guvi_db.* TO 'ravi_dev'@'localhost';
  • 3
    Table Level — ON db.table

    Applies to one specific table only. Most common in practice.

    GRANT SELECT ON guvi_db.students TO 'priya_analyst'@'localhost';
  • 4
    Column Level — ON specific columns

    Grant access to specific columns only — great for hiding sensitive data like salary or Aadhaar number.

    -- Priya can see name and email, but NOT salary or aadhaar
GRANT SELECT (student_name, email, course)
  ON guvi_db.students
  TO 'priya_analyst'@'localhost';
✅
Principle of Least Privilege (PoLP)Always grant only the minimum permissions a user needs to do their job. If an analyst only needs to read reports, don't give them INSERT or DELETE. This reduces the risk of accidental or malicious data changes.
🏦 Real-world Example — HDFC Bank DB
SQL 
-- Customer service rep can see name + balance but NOT account_pin
GRANT SELECT (customer_name, balance, branch)
  ON hdfc_db.accounts
  TO 'support_rep'@'localhost';

-- Auditors can read transactions but not modify anything
GRANT SELECT
  ON hdfc_db.transactions
  TO 'auditor_team'@'%';

🔒 Row-Level Security (RLS)

Column-level privileges hide columns. Row-Level Security (RLS) filters which rows a user can see. This is a PostgreSQL feature — MySQL uses views to simulate it.

📖
Real-world scenarioAt GUVI, each instructor should only see their own batch's students — not all students across all batches. Row-Level Security makes this automatic.
🐘 PostgreSQL — Enable Row-Level Security
PostgreSQL 
-- Step 1: Enable RLS on the table
ALTER TABLE students ENABLE ROW LEVEL SECURITY;

-- Step 2: Create policy — instructor sees only their batch
CREATE POLICY instructor_policy
  ON students
  FOR SELECT
  USING (instructor_id = current_user_id());

-- Step 3: Grant table access (RLS handles row filtering)
GRANT SELECT ON students TO instructor_role;
🐬 MySQL — Simulate RLS with Views

MySQL doesn't have native RLS, but you can use views to achieve the same effect.

MySQL 
-- Create a view that filters rows by instructor
CREATE VIEW my_students AS
  SELECT * FROM students
  WHERE instructor_id = CURRENT_USER();

-- Grant access to the VIEW, not the base table
GRANT SELECT ON guvi_db.my_students
  TO 'instructor_rahul'@'localhost';

-- Rahul sees ONLY his own students when he queries my_students
💡
Views are powerful for access controlUsing views, you can expose only the safe, filtered version of your data to users — hiding sensitive rows, columns, or joining data appropriately. This is widely used at companies like Zomato (restaurant owners see only their own restaurant's data).

🏢 HCL GUVI Case Study

Let's build a complete access control system for the HCL GUVI learning platform — with analyst, developer, and admin roles — from scratch.

🎯
ScenarioHCL GUVI's database (guvi_db) has tables: students, courses, enrollments, payments, instructors, reports. We need 3 types of users:
📊 Analyst (Priya) — read-only access for reports
💻 Developer (Ravi) — read-write for courses & enrollments
🛡️ Admin (Kiran) — full access to everything
📋 Complete Implementation Script
Complete DCL Script — guvi_db 
-- ═══════════════════════════════════════════
-- HCL GUVI DB — Access Control Setup
-- ═══════════════════════════════════════════

-- STEP 1: Create Users
CREATE USER 'priya_analyst'@'localhost' IDENTIFIED BY 'Analyst@2024';
CREATE USER 'ravi_dev'@'localhost'     IDENTIFIED BY 'Dev@Secure2024';
CREATE USER 'kiran_admin'@'%'         IDENTIFIED BY 'AdminK!r@n9999';

-- STEP 2: Create Roles
CREATE ROLE 'guvi_analyst';
CREATE ROLE 'guvi_developer';
CREATE ROLE 'guvi_admin';

-- STEP 3: Grant privileges to roles

-- Analyst: read-only on reports, students
GRANT SELECT ON guvi_db.students     TO 'guvi_analyst';
GRANT SELECT ON guvi_db.enrollments  TO 'guvi_analyst';
GRANT SELECT ON guvi_db.reports      TO 'guvi_analyst';
-- Column-level: no salary/aadhaar for analyst
GRANT SELECT (name, email, batch, course)
  ON guvi_db.students TO 'guvi_analyst';

-- Developer: full DML on courses & enrollments
GRANT SELECT, INSERT, UPDATE, DELETE
  ON guvi_db.courses      TO 'guvi_developer';
GRANT SELECT, INSERT, UPDATE, DELETE
  ON guvi_db.enrollments  TO 'guvi_developer';
GRANT SELECT
  ON guvi_db.students     TO 'guvi_developer';

-- Admin: everything
GRANT ALL PRIVILEGES ON guvi_db.* TO 'guvi_admin';

-- STEP 4: Assign roles to users
GRANT 'guvi_analyst'   TO 'priya_analyst'@'localhost';
GRANT 'guvi_developer' TO 'ravi_dev'@'localhost';
GRANT 'guvi_admin'     TO 'kiran_admin'@'%';

FLUSH PRIVILEGES;
