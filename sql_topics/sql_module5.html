Aggregations, GROUP BY & HAVING

This is where SQL transforms from a lookup tool into an analytics engine. Learn to summarise millions of rows into meaningful numbers that drive business decisions.

COUNT · SUM · AVG · MIN · MAX GROUP BY HAVING NULL handling ROLLUP
What Are Aggregate Functions?

Aggregate functions take a set of rows and return a single value. They collapse many rows into one summary number. Without them, you'd have to pull all data into Python and compute manually. With them, the database engine does it faster, closer to the data.

COUNT()
Counts the number of rows or non-NULL values in a column.
COUNT(*) → 1500
SUM()
Adds up all numeric values in a column. Ignores NULLs.
SUM(price) → ₹82,400
AVG()
Calculates the arithmetic mean. Ignores NULLs in numerator AND denominator.
AVG(salary) → ₹68,200
MIN()
Returns the smallest value. Works on numbers, dates, and text (alphabetically).
MIN(join_date) → 2018-06-30
MAX()
Returns the largest value. Extremely useful for "most recent" or "highest" queries.
MAX(order_total) → ₹12,499
🎓 Analogy — The Cricket Scorecard

Imagine a table with every ball bowled in a cricket match — 300 rows. COUNT(*) tells you how many balls were bowled. SUM(runs) gives the team's total. AVG(runs) is the run rate per ball. MAX(runs) is the highest run in a single delivery. MIN(runs) is the lowest. The entire scorecard is built from five aggregate functions applied to the raw delivery log. That's exactly what you'll do with business data.

The Query Execution Order — Why It Matters

SQL is not executed in the order it's written. Understanding the actual execution order explains why you can use WHERE with regular columns but HAVING with aggregates — they run at different stages.

💡 Why does this matter?

WHERE runs at Step 2 — before aggregation. So you can't write WHERE COUNT(*) > 5 — the count doesn't exist yet!
HAVING runs at Step 4 — after GROUP BY and aggregation. That's why HAVING is where you filter aggregate results. This single concept eliminates 80% of "why doesn't my query work?" confusion.

WHERE vs HAVING — The Golden Rule
📋 WHERE — Filters Rows (before grouping)
• Runs on individual rows
• Executes before GROUP BY
• Can filter on any column
• ❌ Cannot use aggregate functions

Use when: "I only want to include orders from 2024" or "only active customers"
🔍 HAVING — Filters Groups (after grouping)
• Runs on grouped result sets
• Executes after GROUP BY
• Can only reference grouped/aggregated columns
• ✅ Can use aggregate functions

Use when: "Only show me departments with more than 3 employees" or "cities with avg salary > 70K"
🎓 One-Line Mnemonic to Teach

"WHERE filters rows. HAVING filters groups." — Write this on the board. Repeat it every time a student confuses them. After three classes, students will say it automatically when they see each clause.

GROUP BY — How It Works Internally

GROUP BY partitions your data into buckets based on unique values of the specified column(s). Then aggregate functions are applied to each bucket separately, returning one row per group.

📥
Step 1 — Raw Rows
50,000 order rows. Each row has customer_id, city, amount. One row per order.
🗂️
Step 2 — GROUP BY city
Database sorts and groups all rows by city. All Chennai rows together. All Mumbai rows together. Etc.
📊
Step 3 — Aggregate
COUNT(*) and SUM(amount) applied to each city bucket. 50,000 rows collapse to one row per city.
⚠️ The GROUP BY Rule — Most Common Beginner Error

Every column in SELECT that is NOT inside an aggregate function MUST appear in GROUP BY.

You're essentially telling SQL: "for each unique combination of these columns, compute the aggregates." If a column isn't grouped or aggregated, SQL doesn't know which row's value to show for that group — and rightfully throws an error (or returns random values in MySQL's permissive mode). Always SELECT only what you GROUP BY plus your aggregates.

Example 1 — COUNT(): Every Variation You'll Ever Need
COUNT(*) vs COUNT(col) vs COUNT(DISTINCT col)Ex 1
-- COUNT(*): counts ALL rows including NULLs — the most common form
SELECT COUNT(*) AS total_employees
FROM   employees;

-- COUNT(column): counts only NON-NULL values in that column
SELECT COUNT(salary)     AS employees_with_salary,
       COUNT(manager_id) AS employees_with_manager,
       COUNT(*)          AS total_employees
FROM   employees;
-- salary has 9 non-NULLs (Sana has NULL salary)
-- manager_id has 7 non-NULLs (3 people have no manager)

-- COUNT(DISTINCT col): counts unique non-NULL values
SELECT COUNT(DISTINCT department) AS num_departments,
       COUNT(DISTINCT city)       AS num_cities
FROM   employees;

-- COUNT with WHERE: count matching rows only
SELECT COUNT(*) AS engineers
FROM   employees
WHERE  department = 'Engineering';

-- COUNT with CASE: conditional counting (powerful pattern!)
SELECT
    COUNT(*)                                                    AS total_orders,
    COUNT(CASE WHEN status = 'Delivered'  THEN 1 END)          AS delivered,
    COUNT(CASE WHEN status = 'Cancelled'  THEN 1 END)          AS cancelled,
    COUNT(CASE WHEN status = 'Pending'    THEN 1 END)          AS pending,
    ROUND(COUNT(CASE WHEN status='Cancelled' THEN 1 END) * 100.0
          / COUNT(*), 1)                                        AS cancellation_rate_pct
FROM   orders;
COUNT(CASE WHEN ...) Output — Pivot in a single row
total_orders | delivered | cancelled | pending | cancellation_rate_pct -------------+-----------+-----------+---------+----------------------- 10 | 6 | 1 | 1 | 10.0
COUNT(CASE WHEN ... THEN 1 END) is one of the most useful SQL patterns. When the CASE evaluates to NULL (condition not met), COUNT ignores it. When it evaluates to 1, COUNT counts it. This effectively performs conditional counting — no GROUP BY needed, all categories in one row. Used constantly in BI dashboards and reporting queries.
Example 2 — SUM(): Totals, Running Totals & Weighted Averages
SUM in all its forms — including advanced patternsEx 2
-- Basic SUM: total revenue
SELECT SUM(amount)                    AS total_revenue,
       SUM(amount * discount_pct/100) AS total_discount_given,
       SUM(amount) - SUM(amount * discount_pct/100) AS net_revenue
FROM   orders
WHERE  status != 'Cancelled';

-- SUM with CASE: conditional sum (revenue only from Electronics)
SELECT
    SUM(CASE WHEN category = 'Electronics' THEN amount ELSE 0 END) AS electronics_rev,
    SUM(CASE WHEN category = 'Furniture'   THEN amount ELSE 0 END) AS furniture_rev,
    SUM(amount)                                                       AS total_rev
FROM   orders;
-- This is called "pivoting" — all categories as separate columns

-- SUM vs COUNT: different questions, different answers
SELECT
    COUNT(*)      AS number_of_orders,   -- how many orders?
    SUM(amount)   AS total_order_value,  -- what's their total value?
    SUM(amount) / COUNT(*) AS avg_order_value -- same as AVG(amount)
FROM   orders;

-- Weighted average using SUM (true weighted mean)
-- What is the real average discount rate, weighted by order size?
SELECT
    AVG(discount_pct)                               AS simple_avg_discount,
    SUM(amount * discount_pct) / SUM(amount)        AS weighted_avg_discount
FROM   orders;
-- Weighted avg is more accurate: big orders drag the average more than small ones
SUM(CASE WHEN...) is the horizontal pivoting pattern — turning row values into columns. The difference between AVG() and SUM()/COUNT() is crucial when some values are NULL — AVG skips NULLs in both numerator and denominator, while SUM()/COUNT(*) only skips NULL in the numerator. Weighted averages using SUM are more accurate than simple AVGs when your data has very different-sized groups.
📊 AVG · MIN · MAX

AVG(), MIN() & MAX()

Example 3 — AVG, MIN, MAX: Complete Patterns
Statistics that tell the full story of your dataEx 3
-- Basic stats on salary (remember: Sana has NULL salary — watch how each handles it)
SELECT
    AVG(salary)           AS avg_salary,      -- ignores the NULL row
    MIN(salary)           AS min_salary,      -- ignores NULL, returns 48000
    MAX(salary)           AS max_salary,      -- ignores NULL, returns 95000
    MAX(salary) - MIN(salary) AS salary_range,  -- spread between top and bottom
    ROUND(AVG(salary), 0)  AS avg_salary_rounded
FROM   employees;

-- MIN/MAX on text: alphabetical ordering
SELECT
    MIN(name) AS first_name_alphabetically,
    MAX(name) AS last_name_alphabetically
FROM   employees;

-- MIN/MAX on dates: earliest and most recent
SELECT
    MIN(join_date) AS earliest_hire,
    MAX(join_date) AS most_recent_hire,
    DATEDIFF(MAX(join_date), MIN(join_date)) AS days_span
FROM   employees;

-- AVG with CASE: average salary of active employees only
SELECT
    AVG(CASE WHEN is_active = TRUE THEN salary END) AS avg_active_salary
FROM   employees;

-- Percentile approximation using AVG + WHERE trick
-- Employees earning above the average salary
SELECT name, salary
FROM   employees
WHERE  salary > (SELECT AVG(salary) FROM employees)
ORDER BY salary DESC;

-- Five-number summary: Min, Q1 approx, Median approx, Q3 approx, Max
SELECT
    MIN(salary)                                              AS minimum,
    AVG(salary)                                              AS mean,
    MAX(salary)                                              AS maximum,
    MAX(salary) - MIN(salary)                                AS range_val,
    ROUND(STDDEV(salary), 2)                                AS std_dev
FROM   employees;
-- STDDEV() and VARIANCE() are also aggregate functions in MySQL
Basic stats output
avg_salary | min_salary | max_salary | salary_range | avg_salary_rounded -------------+------------+------------+--------------+------------------- 70111.11 | 48000 | 95000 | 47000 | 70111
← AVG computed on 9 rows (Sana's NULL excluded). Range = 95000-48000 = 47000.
🎓 Teaching Moment — NULL in AVG

Ask students: "If 9 employees have salaries and 1 has NULL, and we AVG(salary) — does SQL use 9 or 10 as the denominator?" Answer: 9. SQL skips NULL entirely. This is why you should use COALESCE(salary, 0) if you want NULL to count as 0 in your average. These two queries produce very different results, and data scientists need to know which they actually want.

🗂️ GROUP BY

GROUP BY — Aggregate Per Category

GROUP BY is the engine of analytical SQL. It answers "show me aggregates broken down by each category."

Example 4 — GROUP BY Basics: Single & Multi-Column
Grouping by one column, then multiple columnsEx 4
-- GROUP BY single column: stats per department
SELECT
    department,
    COUNT(*)           AS headcount,
    SUM(salary)        AS total_salary_bill,
    AVG(salary)        AS avg_salary,
    MIN(salary)        AS min_salary,
    MAX(salary)        AS max_salary
FROM   employees
GROUP BY department
ORDER BY total_salary_bill DESC;

-- GROUP BY multiple columns: stats per department PER city
SELECT
    department,
    city,
    COUNT(*)    AS headcount,
    AVG(salary) AS avg_salary
FROM   employees
GROUP BY department, city
ORDER BY department, avg_salary DESC;

-- GROUP BY with alias (can reference SELECT alias in MySQL)
SELECT
    YEAR(join_date) AS join_year,
    COUNT(*)        AS new_hires
FROM   employees
GROUP BY join_year
ORDER BY join_year;
GROUP BY department output
department | headcount | total_salary_bill | avg_salary | min_salary | max_salary -------------+-----------+-------------------+--------------+------------+----------- Engineering | 4 | 350000 | 87500.00 | 78000 | 95000 Sales | 2 | 113000 | 56500.00 | 52000 | 61000 Marketing | 2 | 122000 | 61000.00 | 55000 | 67000 HR | 2 | 48000 | 48000.00 | 48000 | 48000 ← HR avg uses only 1 salary (Sana's is NULL) — important!
The GROUP BY rule in practice: In the first query, department is in GROUP BY, so it's safe in SELECT. COUNT(*), SUM(salary) etc. are aggregates — safe. You can NEVER add name to SELECT here without also adding it to GROUP BY or wrapping it in an aggregate like MAX(name).
Example 5 — GROUP BY with WHERE, ORDER BY & LIMIT
Filtering before grouping, then sorting and slicing resultsEx 5
-- WHERE filters rows BEFORE grouping (only consider delivered/shipped orders)
SELECT
    category,
    COUNT(*)        AS order_count,
    SUM(amount)     AS revenue,
    AVG(amount)     AS avg_order_value
FROM   orders
WHERE  status IN ('Delivered', 'Shipped')   -- filter BEFORE grouping
GROUP BY category
ORDER BY revenue DESC;

-- Group by computed expression: group by month
SELECT
    DATE_FORMAT(order_date, '%Y-%m') AS order_month,
    COUNT(*)                           AS orders_placed,
    SUM(amount)                        AS monthly_revenue
FROM   orders
GROUP BY order_month
ORDER BY order_month;

-- Top categories by revenue (GROUP BY + ORDER BY + LIMIT)
SELECT
    category,
    SUM(amount) AS total_revenue
FROM   orders
GROUP BY category
ORDER BY total_revenue DESC
LIMIT  3;   -- top 3 categories by revenue

-- Group by employee with their order performance
SELECT
    e.name           AS sales_rep,
    e.department,
    COUNT(o.order_id) AS orders_handled,
    SUM(o.amount)    AS total_sales,
    AVG(o.amount)    AS avg_deal_size
FROM   employees e
LEFT JOIN orders o ON e.emp_id = o.emp_id
GROUP BY e.emp_id, e.name, e.department
ORDER BY total_sales DESC;

The DATE_FORMAT + GROUP BY pattern generates every monthly/weekly report you'll ever build. Every business dashboard — Flipkart sales by month, Swiggy orders by day of week, GUVI learners by enrollment month — uses exactly this pattern. Master it and you can build 70% of all BI reports from scratch.

Example 6 — GROUP BY with ROLLUP (Subtotals & Grand Totals)
WITH ROLLUP — automatic subtotals at each grouping levelEx 6
-- WITH ROLLUP: adds subtotal rows automatically at each group level
SELECT
    COALESCE(department, '── GRAND TOTAL') AS department,
    COUNT(*)                                  AS headcount,
    SUM(salary)                               AS total_payroll,
    ROUND(AVG(salary), 0)                    AS avg_salary
FROM   employees
GROUP BY department WITH ROLLUP;

-- Multi-level ROLLUP: subtotals per department AND per city
SELECT
    COALESCE(department, 'ALL DEPARTMENTS') AS dept,
    COALESCE(city,       'ALL CITIES')      AS city,
    COUNT(*)                                  AS count,
    SUM(salary)                               AS payroll
FROM   employees
GROUP BY department, city WITH ROLLUP
ORDER BY department, city;
WITH ROLLUP — subtotals + grand total automatically added
department | headcount | total_payroll | avg_salary --------------------+-----------+---------------+----------- Engineering | 4 | 350000 | 87500 HR | 2 | 48000 | 48000 Marketing | 2 | 122000 | 61000 Sales | 2 | 113000 | 56500 ── GRAND TOTAL | 10 | 633000 | 70333
← ROLLUP adds the last row automatically. COALESCE replaces NULL with a label.
WITH ROLLUP is essential for financial reports and Excel-like summaries. The grand total row has NULL for the grouping column — that's why we use COALESCE(department, 'GRAND TOTAL') to replace it with a readable label. Used in every financial statement, P&L summary, and management report.
🔍 HAVING

HAVING — Filter the Groups

HAVING is WHERE for grouped data. It runs after aggregation — so you can filter on COUNT(), SUM(), AVG() results.

Example 7 — HAVING: From Simple to Real Analytics
HAVING — filtering groups using aggregate conditionsEx 7
-- Basic HAVING: departments with more than 1 employee
SELECT
    department,
    COUNT(*) AS headcount
FROM   employees
GROUP BY department
HAVING   COUNT(*) > 1;

-- HAVING with AVG: departments where avg salary exceeds threshold
SELECT
    department,
    ROUND(AVG(salary), 0) AS avg_salary
FROM   employees
GROUP BY department
HAVING   AVG(salary) > 60000
ORDER BY avg_salary DESC;

-- Combining WHERE (row filter) + HAVING (group filter)
-- Q: Which categories had 3+ delivered/shipped orders AND total revenue > ₹10,000?
SELECT
    category,
    COUNT(*)    AS order_count,
    SUM(amount) AS revenue
FROM   orders
WHERE  status IN ('Delivered', 'Shipped')  -- ← filter rows FIRST
GROUP BY category
HAVING   COUNT(*) >= 3
    AND  SUM(amount) > 10000               -- ← filter groups AFTER
ORDER BY revenue DESC;

-- HAVING with SUM: find employees whose total sales exceed a target
SELECT
    e.name,
    COUNT(o.order_id)  AS deals_closed,
    SUM(o.amount)      AS total_sales
FROM   employees  e
JOIN   orders     o ON e.emp_id = o.emp_id
WHERE  o.status = 'Delivered'
GROUP BY e.emp_id, e.name
HAVING   SUM(o.amount) > 50000   -- only show reps who exceeded target
ORDER BY total_sales DESC;
Example 8 — WHERE vs HAVING: Side-by-Side Comparison
The exact same-looking questions — very different answersEx 8
━━━━━ QUESTION 1: Employees in Engineering department ━━━━━

-- Use WHERE: filter individual rows by department column
SELECT department, COUNT(*) AS headcount
FROM   employees
WHERE  department = 'Engineering'
GROUP BY department;
-- Result: Engineering → 4  (only Engineering rows entered the grouping)

━━━━━ QUESTION 2: Departments that have exactly 4 employees ━━━━━

-- Use HAVING: filter groups by their COUNT aggregate
SELECT department, COUNT(*) AS headcount
FROM   employees
GROUP BY department
HAVING   COUNT(*) = 4;
-- Result: Engineering → 4  (all depts grouped first, then filtered by count)

━━━━━ THE COMMON MISTAKE ━━━━━

-- ❌ WRONG: Can't use aggregate in WHERE
-- SELECT department, COUNT(*) FROM employees
-- WHERE COUNT(*) > 2 GROUP BY department;
-- Error: Invalid use of group function

-- ✅ CORRECT: aggregate condition goes in HAVING
SELECT department, COUNT(*) AS headcount
FROM   employees
GROUP BY department
HAVING   COUNT(*) > 2;

━━━━━ PERFORMANCE TIP ━━━━━

-- Put as much filtering in WHERE as possible — it reduces rows BEFORE grouping
-- This is faster (fewer rows to group):
SELECT department, AVG(salary)
FROM   employees
WHERE  is_active = TRUE       -- filter 20% inactive rows BEFORE grouping
GROUP BY department
HAVING   AVG(salary) > 60000; -- filter groups AFTER
✅ Decision Rule — Teach This to Students

Ask yourself: "Am I filtering individual rows, or am I filtering the result of an aggregation?"
— Individual rows → WHERE
— Aggregation result (COUNT, SUM, AVG etc.) → HAVING

If you're not sure, try writing the equivalent English sentence: "Only include orders where status is Delivered" → WHERE. "Only include departments that have more than 3 employees" → HAVING (headcount is a count — an aggregate).

⚠️ NULLs & Advanced

NULL Handling & Advanced Patterns

Example 9 — NULL Handling in Every Aggregate Function
How each function treats NULL — critical for accurate reportingEx 9
-- Our employees table: Sana has NULL salary, 3 people have NULL manager_id

-- 1. COUNT behaviour with NULL
SELECT
    COUNT(*)          AS total_rows,          -- 10 (counts everything)
    COUNT(salary)     AS salary_not_null,     -- 9  (skips Sana's NULL)
    COUNT(manager_id) AS has_manager         -- 7  (3 people without manager)
FROM   employees;

-- 2. SUM and AVG: both IGNORE NULLs
SELECT
    SUM(salary)                        AS sum_ignores_null,
    AVG(salary)                        AS avg_ignores_null,        -- divides by 9
    SUM(COALESCE(salary, 0))           AS sum_null_as_zero,
    SUM(COALESCE(salary, 0)) / COUNT(*) AS avg_null_as_zero       -- divides by 10
FROM   employees;

-- 3. NULL in GROUP BY: NULLs form their own group!
SELECT
    manager_id,                        -- NULL = "employees with no manager"
    COUNT(*) AS reports
FROM   employees
GROUP BY manager_id
ORDER BY manager_id;

-- 4. COALESCE — handle NULL gracefully in output
SELECT
    name,
    COALESCE(salary, 0)              AS salary_display,
    COALESCE(city, 'Unknown')        AS city_display,
    IFNULL(manager_id, 'No Manager') AS manager_display
FROM   employees;

-- 5. NULLIF — turn a value INTO NULL (useful for division)
-- Prevents division by zero errors:
SELECT
    category,
    SUM(amount) / NULLIF(COUNT(*), 0) AS safe_avg
    -- NULLIF returns NULL if count is 0 → division gives NULL, not error
FROM   orders
GROUP BY category;
AVG(salary) — NULLs excluded
Rows: 10
Non-NULL salaries: 9
SUM = 633,000
AVG = 633,000 / 9 = 70,333
AVG using COALESCE — NULLs = 0
Rows: 10
All 10 rows included
SUM = 633,000
AVG = 633,000 / 10 = 63,300
⚠️ Real-World Impact — Wrong AVG Leads to Wrong Decisions

If you're calculating average salary for payroll budgeting and 20% of records have NULL salary (data not filled in yet), your AVG is 20% higher than reality — you'll underestimate the budget. Always decide consciously: should NULL salary count as ₹0 (use COALESCE) or should that employee be excluded from the calculation (use AVG directly)? The answer depends on your business question.

Example 10 — Advanced Analytics: Full Dashboard Query
A real-world sales dashboard query combining everything in this chapterCapstone
-- Business Question:
-- "Give me a sales performance summary by category and month
--  showing: order volume, revenue, avg order, cancellation rate,
--  discount impact — for orders with meaningful revenue only"

SELECT
    -- Grouping dimensions
    category,
    DATE_FORMAT(order_date, '%Y-%m')                               AS month,

    -- Volume metrics
    COUNT(*)                                                          AS total_orders,
    COUNT(CASE WHEN status = 'Delivered' THEN 1 END)                AS delivered,
    COUNT(CASE WHEN status = 'Cancelled' THEN 1 END)                AS cancelled,

    -- Revenue metrics
    SUM(amount)                                                       AS gross_revenue,
    SUM(amount * (1 - discount_pct / 100))                          AS net_revenue,
    ROUND(AVG(amount), 0)                                            AS avg_order_value,

    -- Discount analysis
    ROUND(AVG(discount_pct), 1)                                       AS avg_discount_pct,
    SUM(amount * discount_pct / 100)                                 AS total_discount_given,

    -- Rates (as percentages)
    ROUND(
        COUNT(CASE WHEN status = 'Cancelled' THEN 1 END) * 100.0
        / NULLIF(COUNT(*), 0)
    , 1)                                                               AS cancellation_rate_pct,

    -- Min/Max for range insight
    MIN(amount) AS smallest_order,
    MAX(amount) AS largest_order

FROM  orders

WHERE order_date >= '2024-01-01'            -- row-level filter: 2024 orders only

GROUP BY
    category,
    DATE_FORMAT(order_date, '%Y-%m')

HAVING
    SUM(amount) > 5000                        -- group filter: only meaningful revenue
    AND COUNT(*) >= 2                          -- at least 2 orders in the group

ORDER BY
    month,
    net_revenue DESC;
-- This single query powers a complete sales analytics dashboard
This query demonstrates every concept from Chapter 5 in one place: aggregate functions (COUNT, SUM, AVG, MIN, MAX), COUNT(CASE WHEN) for conditional counting, computed columns with expressions, DATE_FORMAT for temporal grouping, WHERE for pre-group row filtering, GROUP BY on multiple columns, HAVING for post-group filtering, NULLIF for safe division, and ORDER BY on a derived column. This is the kind of query a senior data analyst writes every day.

Fraud analysts write: find accounts where COUNT(transactions in last 1 hour) > 20 OR SUM(amount in last 10 min) > 50000. That's HAVING on a GROUP BY account_id, time_window. Every fraud alert is an aggregate threshold query. Banks run these continuously on streaming data.

"Which batch has the lowest average quiz score?" → GROUP BY batch_id, ORDER BY AVG(quiz_score). Before training a churn model: SELECT customer_id, COUNT(orders) AS order_frequency, AVG(order_value) AS avg_spend, MAX(order_date) AS last_purchase, DATEDIFF(NOW(), MAX(order_date)) AS days_since_last_order FROM orders GROUP BY customer_id. ❓ FAQ

Questions Students Always Ask

Q: Why do I get "1140 – In aggregated query without GROUP BY, expression #1 of SELECT list contains nonaggregated column"?▾
This means you have a column in SELECT that is neither in GROUP BY nor inside an aggregate function. SQL doesn't know which row's value to show for that column within the group.

-- ❌ ERROR: name is not in GROUP BY and not aggregated SELECT department, name, COUNT(*) FROM employees GROUP BY department; -- ✅ FIX 1: Add name to GROUP BY (now grouping by dept+name combo) SELECT department, name, COUNT(*) FROM employees GROUP BY department, name; -- ✅ FIX 2: Aggregate name (e.g., pick any one name from the group) SELECT department, MAX(name) AS sample_name, COUNT(*) FROM employees GROUP BY department; -- ✅ FIX 3: Remove name from SELECT SELECT department, COUNT(*) FROM employees GROUP BY department;
Note: MySQL has a "permissive" mode (sql_mode without ONLY_FULL_GROUP_BY) where it won't throw an error — but returns an arbitrary row's value, which is almost always wrong. Always fix the query properly.
Q: Can I use a SELECT alias in HAVING?▾
It depends on the database:

MySQL: Yes — MySQL evaluates aliases from SELECT before HAVING, so this works:
SELECT department, COUNT(*) AS headcount FROM employees GROUP BY department HAVING headcount > 2; -- works in MySQL
PostgreSQL / standard SQL: No — HAVING runs before SELECT aliases are resolved. Use the full expression:
HAVING COUNT(*) > 2; -- always works in all databases
Best practice: always use the full expression in HAVING for maximum portability.
Q: What's the difference between COUNT(*) and COUNT(1)? Which is faster?▾
COUNT(*) and COUNT(1) both count all rows including those with NULLs. They produce identical results and modern query optimisers treat them identically — there is no performance difference.

Historically, people believed COUNT(1) was faster because it didn't need to look at column values. This was a myth. Today's optimisers recognise both forms and execute them the same way. Use COUNT(*) — it's the ANSI standard and clearly communicates "count all rows." COUNT(column_name) is different — it only counts non-NULL values.
Q: How do I find duplicates in a table using GROUP BY?▾
Classic interview question! GROUP BY + HAVING COUNT > 1 is the standard approach:
-- Find duplicate emails in customers SELECT email, COUNT(*) AS occurrences FROM customers GROUP BY email HAVING COUNT(*) > 1 ORDER BY occurrences DESC; -- Find all rows that are duplicates (with full details) SELECT * FROM customers WHERE email IN ( SELECT email FROM customers GROUP BY email HAVING COUNT(*) > 1 ) ORDER BY email;
Q: Can I GROUP BY and HAVING without a SELECT aggregate? What happens?▾
Yes, technically — but it's unusual. HAVING uses the aggregate for filtering even if it's not in SELECT:
-- HAVING without showing the aggregate in SELECT SELECT department FROM employees GROUP BY department HAVING COUNT(*) > 2; -- Shows only department name, but filtering was done using COUNT -- This is valid — SELECT only shows what you need
Useful when you want to filter based on an aggregate but don't want to show it in the output.
Q: How do I calculate a percentage / ratio within a group?▾
Two common patterns:

Pattern 1: Use a subquery or scalar subquery in SELECT:
SELECT department, COUNT(*) AS headcount, ROUND(COUNT(*) * 100.0 / (SELECT COUNT(*) FROM employees), 1) AS pct_of_total FROM employees GROUP BY department;
Pattern 2: Window functions (covered in Chapter 13 — SUM OVER PARTITION BY) — more elegant for complex cases.
