Topic 1

Why Do We Need Data Warehousing?

In Chapter 1, you learned about databases — they store operational data (orders, users, payments). But a regular database is built for speed of transactions, not speed of analysis. As companies grow, they hit a wall when trying to analyze their own data.

🇮🇳 Real-World Problem
Imagine you're a Data Analyst at Flipkart
Your manager asks: "Give me the top 10 product categories by revenue, state-wise, for the last 24 months, compared to the previous 24 months."

You run this query on the live orders database... and wait.
5 minutes later — still running.
20 minutes later — the database server is now slow for all 10 million shoppers because your analysis query is hogging resources.

This is the exact problem Data Warehousing solves. 🎯
⚡ Operational DB (Day-to-Day)
Handles thousands of small transactions per second
Optimized for INSERT, UPDATE, DELETE speed
Data is current — real-time
Running an analysis query SLOWS IT DOWN
Stores data for weeks/months (then archived)
Example: MySQL, PostgreSQL
📊 Data Warehouse (Analysis)
Handles complex analytical queries across billions of rows
Optimized for SELECT, GROUP BY, aggregations
Data is historical — refreshed in batches
Analysis queries never affect live users
Stores years of historical data
Example: BigQuery, Snowflake, Redshift
💡
The core idea
Think of it like this — your operational database is the kitchen where food is prepared (live, fast, hectic). Your data warehouse is the restaurant review website where all the historical data about meals, ratings, and revenue is stored and analyzed at leisure — without disturbing the kitchen.

Topic 2

OLTP vs OLAP — The Core Difference

These two terms define how a system processes data. You will hear them constantly in data engineering and analytics roles.

⚡ OLTP — Online Transaction Processing
Purpose: Run the business day-to-day
Operations: INSERT, UPDATE, DELETE constantly
Query type: Simple — "Get order #12345"
Data volume: Current data, KB to MB per query
Users: Developers, applications, end users
Speed: Milliseconds per transaction
Examples: Booking a train on IRCTC, placing Zomato order, ATM withdrawal
📊 OLAP — Online Analytical Processing
Purpose: Analyze the business historically
Operations: Mostly SELECT, aggregations
Query type: Complex — "Sales by region last 3 years"
Data volume: Historical, GB to TB per query
Users: Analysts, data scientists, management
Speed: Seconds to minutes is acceptable
Examples: Sales trend dashboards, ML feature stores, business intelligence reports
🎮 Interactive: OLTP or OLAP? Classify each scenario!
Normalization (OLTP)
OLTP databases are highly normalized — data is split into many tables to avoid redundancy and make writes fast.
Students, Enrollments, Courses in separate tables
Denormalization (OLAP)
OLAP warehouses often denormalize — combine data into fewer, wider tables to make reads/aggregations faster.
One big "sales_fact" table with all dimensions joined in
HTAP
Hybrid Transactional/Analytical Processing — newer systems (like Google Spanner) do both OLTP and OLAP in one.
Used by some modern cloud databases
Latency
OLTP needs low latency (ms). OLAP tolerates higher latency (seconds/minutes) because queries are complex.
ATM = ms. Monthly report = minutes is fine

Topic 3

What is a Data Warehouse?

A Data Warehouse (DWH) is a centralized repository that collects, stores, and organizes large volumes of historical data from multiple sources — specifically designed for analysis and reporting. It is the backbone of Business Intelligence (BI).

📖
Bill Inmon's Definition (Father of Data Warehousing)
A data warehouse is a Subject-Oriented, Integrated, Non-Volatile, and Time-Variant collection of data in support of management's decision-making process. — Bill Inmon, 1990

👆 The 4 Core Properties of a Data Warehouse — click each to expand:

🎯
Property 1
Subject-Oriented
Organized around key business subjects
›
A DWH is organized around business subjects — Sales, Customers, Products, Finance — not around operational processes (like placing an order or processing a payment). This makes it easy for analysts to ask "Give me all data about our Sales" without knowing which 12 tables that spans.

Example: Instead of having orders, payments, returns, shipping tables separately — a DWH might have a "Sales" subject area that consolidates all of it.
🔗
Property 2
Integrated
Data from ALL systems, unified into one format
›
A DWH pulls data from multiple source systems — MySQL, Oracle, Excel files, APIs, third-party apps — and transforms them all into one consistent format. Inconsistencies are resolved during loading.

Example: One system calls it "Gender: M/F", another says "sex: Male/Female", a third uses "0/1". The DWH standardizes all three to "gender: Male/Female" during ETL.
🔒
Property 3
Non-Volatile
Data is never updated or deleted — only added
›
Once data enters the warehouse, it is never changed or deleted. Analysts need to trust that historical data is stable — if you correct a mistake, you ADD a new corrected record, not modify the old one.

Example: If a customer's city was recorded wrong as "Bangalor" and later corrected to "Bangalore" in the source system — the DWH keeps the old record AND adds the corrected one with a timestamp. You can always see the full history.
⏰
Property 4
Time-Variant
Every record has a time dimension — history matters
›
Every record in a DWH is tagged with a time element. This allows you to compare data across different time periods — this month vs last month, this year vs last year, etc.

Example: A customer's record stores when they were acquired, when they first purchased, their purchase history over time. You can track their entire lifecycle as a customer — not just their current state.
🇮🇳 Real-World Example
HDFC Bank's Data Warehouse
HDFC Bank has millions of customers with transactions across Credit Cards, Home Loans, Savings Accounts, Net Banking, and Mobile App — each running on different systems.

Their DWH:
✅ Integrates all these systems into one unified customer view
✅ Stores 10+ years of transaction history (time-variant)
✅ Never modifies historical records (non-volatile)
✅ Organized by subject: Customer, Product, Risk, Compliance

This allows risk teams to detect fraud patterns, product teams to identify upsell opportunities, and compliance teams to generate regulatory reports — all from one place.

Topic 4

ETL — Extract, Transform, Load

ETL is the process that moves data from source systems into the data warehouse. It's the pipeline that runs (usually nightly or in near-real-time) to keep the warehouse fresh. Understanding ETL is essential for data engineering roles.

1️⃣ Extract — Get the raw data
Pull data from all source systems — RDBMS databases, REST APIs, flat files (CSV/Excel), third-party SaaS tools. The extraction must be done carefully to not slow down the source system. Common patterns: full extract (copy everything) or incremental extract (only new/changed records since last run).
Example: Pull all orders from MySQL where order_date > last_run_timestamp
2️⃣ Transform — Clean and standardize
This is the heavy lifting. Raw data is messy — different formats, missing values, duplicates, inconsistent naming. Transform rules: standardize date formats, fix encoding issues, deduplicate records, handle NULLs, join related data, apply business logic (e.g., categorize orders by value tier).
Example: Convert "01-15-2024" → "2024-01-15", "M" → "Male", NULL city → "Unknown"
3️⃣ Load — Write to the warehouse
Insert the cleaned, transformed data into the warehouse tables. Two strategies: Full Load (truncate and reload everything — simple but slow) or Incremental Load (append only new records — fast but complex). Most production systems use incremental load with upsert logic.
Example: INSERT INTO fact_sales SELECT ... FROM staging_orders
📌
ELT — The modern twist
Modern cloud warehouses (BigQuery, Snowflake) are so powerful that teams now use ELT — Extract, Load, then Transform. Raw data is loaded first, then transformed using SQL inside the warehouse itself. Tools like dbt (Data Build Tool) are built on this approach.

Topic 5

Fact Tables & Dimension Tables

The two most important concepts in Data Warehouse design. Every DWH is built around Fact Tables and Dimension Tables. Understanding these will help you design any warehouse schema.

📊 Fact Table

Contains the measurable, quantitative data — the actual business events or transactions. It's what you aggregate and analyze.

  • Stores numeric measures (amount, quantity, score)
  • Has many rows — can be billions of records
  • References dimension tables via Foreign Keys
  • Rows are added constantly (non-volatile)
  • Almost never updated once inserted
🇮🇳 Examples: fact_sales, fact_orders, fact_transactions, fact_page_views
📐 Dimension Table

Contains the descriptive, contextual data — the "who, what, where, when" that gives meaning to the facts.

  • Stores descriptive attributes (name, city, category)
  • Has far fewer rows than fact tables
  • Referenced by fact table via Foreign Keys
  • Can be updated (e.g., customer changes city)
  • Provides the "slice and dice" ability
🇮🇳 Examples: dim_customer, dim_product, dim_date, dim_location
🎯
The Cricket Analogy — Fact vs Dimension
Fact table (fact_match): runs_scored=85, wickets=3, fours=6, sixes=2 — these are the NUMBERS
dim_player: Virat Kohli, India, Right-handed, Mumbai Indians — WHO scored
dim_match: IPL 2024, Mumbai vs Delhi, Wankhede Stadium — CONTEXT
dim_date: April 2024, Sunday, Season 17 — WHEN

You analyze: "How many runs did Kohli score in home matches in 2023 vs 2024?" — fact + multiple dimensions!

Topic 6

Star Schema vs Snowflake Schema

These are the two most common warehouse schema designs. They differ in how dimension tables are structured — and each has clear tradeoffs.

⭐ Star Schema — Simple and Fast

The fact table sits in the center, surrounded by denormalized dimension tables — like a star. Each dimension is one single table with all attributes, even if there's some redundancy. This makes queries fast and simple.

📐 dim_date

│
📐 dim_customer
—
📊 fact_orders
—
📐 dim_product
│
📐 dim_location
Fact table in center · Flat dimension tables around it · Simple joins
✅
When to use Star Schema
Most production data warehouses use Star Schema. Fewer JOINs = faster queries. BI tools like Tableau, Power BI, Metabase work best with star schemas. Preferred choice for analytics teams.

❄️ Snowflake Schema — Normalized Dimensions

The fact table is in the center, but dimension tables are further split into sub-dimension tables — like a snowflake. Less redundancy but more JOINs required.

📐 dim_product

↓
📎 dim_category

↓
📎 dim_subcategory
—
📊 fact_orders
—
📐 dim_location

↓
📎 dim_city

↓
📎 dim_state
Dimensions further normalized into sub-tables · Less storage · More JOINs
⚠️
When to use Snowflake Schema
Use when storage space is critical, or when dimension attributes change frequently (like product categories being reorganized). The extra normalization reduces redundancy but requires more complex JOINs. Less common in modern cloud warehouses where storage is cheap.
⭐ Star Schema
Dimension tables are denormalized (flat)
Fewer JOINs — simpler, faster queries
Some data redundancy in dimensions
Easy to understand and use
Best for BI tools and dashboards
Used by: most production DWHs
❄️ Snowflake Schema
Dimension tables are normalized (split further)
More JOINs — complex queries
No redundancy — saves storage
More complex to maintain
Better for frequently-updated dimensions
Used by: legacy warehouses, strict storage
💡
Industry Reality
In modern cloud data warehouses (BigQuery, Snowflake, Redshift), storage is cheap. Most companies use Star Schema because query speed and simplicity matter more than saving a few GBs. When in doubt — go with Star Schema.

Topic 7

Data Mart, Data Lake & Data Lakehouse

As companies grow, their data architecture evolves. Here are three terms you'll encounter constantly in the industry — each is a different approach to storing and using big data.

🏪 Data Mart

A subset of a data warehouse focused on one business unit or department.

📌 Think of the full DWH as a supermarket
📌 A data mart is one aisle (like "Sales Mart" or "Finance Mart")
📌 Each team gets their own focused slice
📌 Faster queries since less data to scan
Example: Marketing team has a "Customer Analytics Mart" — only customer + campaign data, not finance data
🏞️ Data Lake

A massive raw storage of ALL data — structured, semi-structured, and unstructured — in its original format.

📌 Dump everything in first, structure it later
📌 Stores: SQL data + JSON + images + logs + audio
📌 Very cheap storage (object storage like S3)
📌 Risk: becomes a "data swamp" if not managed
Example: AWS S3, Azure Data Lake, Google Cloud Storage — all raw data dumped in, processed later by Spark or dbt
🏠 Data Lakehouse

The best of both worlds — lake storage + warehouse query capabilities. The modern standard.

📌 Raw data stored cheaply (like a lake)
📌 SQL queries run directly on raw files
📌 ACID transactions supported
📌 Unified platform for BI + ML + streaming
Example: Databricks Delta Lake, Apache Iceberg, BigLake — increasingly adopted by large companies
📌
Which one will you encounter?
Startups/SMEs: Often just a Data Warehouse (Bigquery or Redshift)
Mid-size companies: DWH + Data Marts per team
Large enterprises (Flipkart, Swiggy, Ola): Data Lake or Lakehouse architecture
As a Data Scientist: You'll query the DWH or Lakehouse using SQL — this chapter gives you the mental model to understand what's happening behind the scenes.

Topic 8

The Modern Data Stack

The "Modern Data Stack" is the set of tools companies use today to build their data infrastructure. Every tool in this stack uses SQL. This is why SQL is the #1 skill for any data role.

📥
Layer 1 — Ingestion
Fivetran, Airbyte, Kafka, custom Python scripts
Pull data from 100s of sources (Salesforce, MySQL, Stripe, Google Ads) into your warehouse
↓
🏭
Layer 2 — Storage & Warehouse
BigQuery, Snowflake, Amazon Redshift, Databricks
Centralized, scalable, cloud-native SQL warehouses — query petabytes in seconds
↓
⚙️
Layer 3 — Transformation
dbt (Data Build Tool), SQL scripts, Spark SQL
Write SQL models to clean and transform raw data into analysis-ready tables — version controlled like code
↓
📊
Layer 4 — BI & Visualization
Tableau, Power BI, Looker, Metabase, Superset
Connect to warehouse, write SQL queries, create dashboards and reports for business teams
↓
🤖
Layer 5 — ML & Data Science
Python + SQL, Jupyter notebooks, MLflow, Feature stores
Feature engineering from warehouse using SQL → feed into ML models → predictions stored back in warehouse
