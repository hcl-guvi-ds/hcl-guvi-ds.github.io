Views, Indexes & Stored Procedures

Chapter 14 · Views, Indexes & Stored Procedures

Abstraction, Speed,
and Reusable Logic.

Views hide complexity. Indexes make queries fly. Stored procedures package business logic inside the database itself. These three tools are what separate "I can write a SELECT" from "I can architect a database."

What You'll Master

👁️ Views

Virtual tables built from a saved query. Simplify complex joins, hide sensitive columns, present a clean interface.

🌳 Indexes

Data structures that let the database find rows in milliseconds instead of scanning millions of rows.

⚙️ Stored Procedures

Reusable, parameterized SQL logic stored inside the database — callable with a single CALL statement.

🎓
Why this chapter mattersA query that takes 8 seconds on 100,000 rows can take 8 milliseconds with the right index. A messy 5-table JOIN that confuses your team can become one clean view: SELECT * FROM student_report. This chapter is where "it works" becomes "it works fast and clean."

👁️ Views — Virtual Tables

A VIEW is a saved SQL query that behaves like a table. It doesn't store data itself — every time you query a view, the underlying query runs fresh.

How a View Works

📖 CREATE VIEW Syntax
SQL
CREATE VIEW view_name AS
SELECT columns
FROM   table(s)
WHERE  condition;
🟢 Simple View — One Table
SQL — GUVI
-- A simple view: active learners only
CREATE VIEW active_learners AS
SELECT student_id, name, city, batch_id, quiz_score
FROM   guvi_students
WHERE  DATEDIFF(CURDATE(), last_login) <= 30;

-- Now query it just like a table!
SELECT * FROM active_learners WHERE city = 'Chennai';
🔵 Complex View — Multi-Table JOIN
SQL — GUVI
-- Complex view: full student report (hides the JOIN complexity)
CREATE VIEW student_full_report AS
SELECT
  s.name           AS student,
  s.city,
  b.batch_name,
  c.course_name,
  i.name           AS instructor,
  s.quiz_score,
  CASE WHEN s.quiz_score >= 75 THEN 'Pass' ELSE 'Needs Improvement' END AS status
FROM       students    s
JOIN       batches     b ON s.batch_id      = b.batch_id
JOIN       courses     c ON b.course_id     = c.course_id
JOIN       instructors i ON b.instructor_id = i.instructor_id;

-- Now the whole team can do this, instead of rewriting the 4-table JOIN:
SELECT * FROM student_full_report WHERE status = 'Needs Improvement';
💡
Views shine for reuse.Once student_full_report exists, every analyst, dashboard, and report just queries the view — nobody needs to remember the 4-table JOIN logic ever again.
🛠️ Managing Views — DROP, ALTER, CREATE OR REPLACE
SQL
-- Modify a view's definition without dropping dependent objects
CREATE OR REPLACE VIEW active_learners AS
SELECT student_id, name, city, batch_id, quiz_score, last_login
FROM   guvi_students
WHERE  DATEDIFF(CURDATE(), last_login) <= 14;  -- changed from 30 to 14 days

-- ALTER VIEW (PostgreSQL/SQL Server syntax — same effect as CREATE OR REPLACE)
ALTER VIEW active_learners AS
SELECT * FROM guvi_students WHERE quiz_score IS NOT NULL;

-- Remove a view entirely
DROP VIEW IF EXISTS active_learners;

🔐 Views for Security & Updatable Views

Two powerful uses of views: hiding sensitive columns from certain users, and writing data back through a view.

🔐 Using Views for Column-Level Security

Instead of granting access to the full table, grant access to a view that exposes only safe columns. This combines beautifully with the DCL concepts from Chapter 8.

SQL — HDFC Bank example
-- Real table has sensitive columns: salary, aadhaar, bank_balance
-- Create a view that hides them
CREATE VIEW employee_public_info AS
SELECT employee_id, name, department, designation, city
FROM   employees;
-- salary, aadhaar, bank_balance are NOT in the view — invisible!

-- Grant access to the VIEW, not the base table
GRANT SELECT ON employee_public_info TO 'hr_intern'@'localhost';
-- hr_intern can NEVER see salary, even with SELECT * — the column doesn't exist in the view!
✅
This is true security, not just convention.Even a malicious SELECT * on the view physically cannot return salary data — the column simply isn't part of the view's definition.
✏️ Updatable Views — Writing Through a View

Some views allow INSERT/UPDATE/DELETE directly — these changes pass through to the underlying table. Views become "non-updatable" once they include JOINs, GROUP BY, DISTINCT, or aggregate functions.

SQL
-- ✅ This view IS updatable — single table, no aggregation
CREATE VIEW chennai_students AS
SELECT student_id, name, email, quiz_score
FROM   students
WHERE  city = 'Chennai';

-- Update through the view — updates the REAL students table!
UPDATE chennai_students SET quiz_score = 88 WHERE student_id = 'S01';

-- ❌ This view is NOT updatable — it has a JOIN and aggregate
CREATE VIEW batch_summary AS
SELECT b.batch_name, COUNT(s.student_id) AS total
FROM   batches b JOIN students s ON b.batch_id = s.batch_id
GROUP BY b.batch_name;
-- UPDATE batch_summary ... → ERROR: view is not updatable
📋 Updatable vs Non-Updatable — Quick Rules
View has...Updatable?
Single table, no aggregation✅ Usually updatable
JOIN across multiple tables❌ Not updatable
GROUP BY / aggregate functions❌ Not updatable
DISTINCT❌ Not updatable
Subqueries in SELECT⚠️ Depends on DB engine

💾 Materialized Views — PostgreSQL

A regular view re-runs its query every time. A materialized view stores (caches) the actual result physically on disk — much faster to read, but data goes stale until refreshed.

👁️ Regular VIEW

Runs the query fresh every time. Always up-to-date. Slower for expensive queries (joins, aggregates on large tables).

💾 MATERIALIZED VIEW

Stores results physically. Very fast to read. Data can be stale — needs manual or scheduled REFRESH.

📖 CREATE MATERIALIZED VIEW Syntax (PostgreSQL)
PostgreSQL
-- Create — runs once, stores the result physically
CREATE MATERIALIZED VIEW guvi_monthly_summary AS
SELECT
  batch_id,
  COUNT(*)            AS total_learners,
  ROUND(AVG(quiz_score),1) AS avg_score,
  MAX(quiz_score)        AS top_score
FROM   guvi_learners
GROUP BY batch_id;

-- Query it — instant, because data is pre-computed
SELECT * FROM guvi_monthly_summary WHERE avg_score > 75;

-- Refresh when underlying data changes (run nightly via cron)
REFRESH MATERIALIZED VIEW guvi_monthly_summary;

-- Refresh without locking reads (PostgreSQL — needs unique index)
REFRESH MATERIALIZED VIEW CONCURRENTLY guvi_monthly_summary;
🎯 When to Use Materialized Views
  • 1
    Expensive aggregations

    Dashboard queries that scan millions of rows and compute heavy aggregates — compute once, read many times.

  • 2
    Data that doesn't need real-time freshness

    Monthly reports, analytics dashboards — refreshing nightly or hourly is perfectly acceptable.

  • 3
    Reducing load on production database

    Heavy analytical queries can slow down the live transactional database — materialized views isolate that cost.

⚠️
MySQL doesn't support materialized views natively.Common workarounds: scheduled events that write results into a real summary table, or use a tool like a cron job + INSERT INTO ... SELECT to simulate the behaviour.

🌳 Indexes — How a B-Tree Works

An index is a separate, sorted data structure that lets the database jump directly to matching rows instead of scanning every single one. Most databases use a B-Tree (balanced tree) structure.

📖
The book index analogyWithout an index, finding "SQL Joins" in a 1000-page textbook means reading every page. With an index at the back of the book, you jump straight to page 247. A database index works exactly the same way.
🌳 B-Tree Structure — Visual
M
D
T
A,B,C
E,F,G
P,Q,R
U,V,W

Searching for 'R' (highlighted): Root(M) → R>M go right → T → R<T go left → Found in leaf [P,Q,R]. Only 3 comparisons instead of scanning all 23 letters!

⚡ Full Table Scan vs Index Seek
Aspect🔴 Full Table Scan (no index)🟢 Index Seek (with index)
How it worksReads EVERY row, checks each oneJumps directly to matching rows via B-tree
Time complexityO(n) — linearO(log n) — logarithmic
100,000 rows searchUp to 100,000 comparisons~17 comparisons (log₂ 100,000)
When it happensNo index on filtered column, or query can't use oneWHERE/JOIN/ORDER BY uses an indexed column
Best forSmall tables, or returning most of the table anywayLarge tables, selective filters (few matching rows)
💡
Indexes aren't always faster!If a query returns 90% of a table's rows, a full scan can actually be faster than using an index (because jumping around the index + fetching rows individually has overhead). The query optimizer decides automatically based on statistics.

📑 CREATE INDEX & Composite Indexes

Creating the right index — on the right column(s), in the right order — is one of the highest-leverage skills a backend developer can have.

📖 CREATE INDEX Syntax
SQL
-- Single column index
CREATE INDEX idx_students_city
ON students(city);

-- Unique index — also enforces uniqueness
CREATE UNIQUE INDEX idx_students_email
ON students(email);

-- Drop an index
DROP INDEX idx_students_city ON students;   -- MySQL
DROP INDEX idx_students_city;             -- PostgreSQL
🔗 Composite Index — Multiple Columns

A composite (multi-column) index speeds up queries that filter on multiple columns together — but column ORDER matters!

SQL
-- Composite index on (batch_id, quiz_score)
CREATE INDEX idx_batch_score
ON guvi_learners(batch_id, quiz_score);

-- ✅ Uses the index fully — filters batch_id first, then quiz_score
SELECT * FROM guvi_learners
WHERE  batch_id = 'B01' AND quiz_score > 75;

-- ✅ Still uses the index — just batch_id (leftmost column)
SELECT * FROM guvi_learners WHERE batch_id = 'B01';

-- ❌ CANNOT use this index efficiently — skips batch_id (leftmost column)
SELECT * FROM guvi_learners WHERE quiz_score > 75;
⚠️
Leftmost prefix rule!A composite index on (A, B, C) can be used for queries filtering on A, or A+B, or A+B+C — but NOT for queries filtering only on B or only on C. Always put the most selective / most commonly filtered column first.
🎯 Which Columns Should You Index?
  • 1
    Columns in WHERE clauses

    If you frequently filter WHERE city = 'Chennai', index the city column.

  • 2
    Columns used in JOINs

    Foreign key columns like batch_id, course_id should almost always be indexed.

  • 3
    Columns in ORDER BY

    Sorting large result sets is much faster when the column is already indexed (pre-sorted).

  • 4
    High-cardinality columns

    Columns with many unique values (email, student_id) benefit more than low-cardinality ones (gender: M/F only).

🚨
Don't over-index!Every index speeds up reads but SLOWS DOWN writes (INSERT/UPDATE/DELETE must update every index too). Index only columns that are actually queried frequently.

🔍 EXPLAIN & EXPLAIN ANALYZE — Reading Query Plans

EXPLAIN shows you exactly HOW the database engine plans to execute your query — whether it uses an index, scans the full table, or something else.

📖 EXPLAIN Syntax
SQL
-- Show the query plan (doesn't actually run the query)
EXPLAIN SELECT * FROM guvi_learners WHERE batch_id = 'B01';

-- EXPLAIN ANALYZE — actually runs the query AND shows real timing
EXPLAIN ANALYZE SELECT * FROM guvi_learners WHERE batch_id = 'B01';
🔴 EXPLAIN Output — Without Index (Full Table Scan)
idselect_typetabletypepossible_keyskeyrowsExtra
1SIMPLEguvi_learnersALLNULLNULL100,000Using where

type: ALL means a FULL TABLE SCAN — all 100,000 rows checked one by one. This is the slowest possible plan.

🟢 EXPLAIN Output — With Index (Index Seek)
idselect_typetabletypepossible_keyskeyrowsExtra
1SIMPLEguvi_learnersrefidx_batch_ididx_batch_id312Using index condition

type: ref means the index is being used efficiently — only 312 rows checked instead of 100,000!

📋 Key EXPLAIN Columns to Watch
ColumnWhat to look for
typeconst, eq_ref, ref = good (uses index). ALL = bad (full scan)
keyWhich index is actually being used. NULL = no index used
rowsEstimated rows the engine will examine — lower is better
Extra"Using filesort" or "Using temporary" are warning signs of inefficiency

⚙️ Stored Procedures — Reusable SQL Blocks

A stored procedure is a named, saved block of SQL logic stored inside the database. Instead of repeating the same complex SQL from every application, you write it once and CALL it.

📖 CREATE PROCEDURE Syntax
MySQL
DELIMITER //

CREATE PROCEDURE procedure_name(parameters)
BEGIN
    -- SQL statements here
END //

DELIMITER ;
💡
Why DELIMITER //?Inside a procedure, you use semicolons to separate statements. Changing the DELIMITER to // temporarily tells MySQL "don't end the whole command at the first semicolon" — so the procedure body can have multiple statements.
🏫 GUVI Example — Simple Procedure (No Parameters)
SQL
DELIMITER //

CREATE PROCEDURE get_batch_summary()
BEGIN
    SELECT
      batch_id,
      COUNT(*)            AS total_learners,
      ROUND(AVG(quiz_score),1) AS avg_score
    FROM   guvi_learners
    GROUP BY batch_id
    ORDER BY avg_score DESC;
END //

DELIMITER ;

-- Call it anytime — no need to rewrite the query!
CALL get_batch_summary();
📥 Procedure with IN Parameter
SQL
DELIMITER //

CREATE PROCEDURE get_students_by_city(IN p_city VARCHAR(50))
BEGIN
    SELECT student_id, name, quiz_score
    FROM   guvi_learners
    WHERE  city = p_city
    ORDER BY quiz_score DESC;
END //

DELIMITER ;

-- Call with different cities — reusable!
CALL get_students_by_city('Chennai');
CALL get_students_by_city('Mumbai');
CALL get_students_by_city('Bangalore');
🎯 Why Use Stored Procedures?
✅ Pros
• Logic lives in one place, not duplicated across apps
• Reduces network round-trips (1 CALL vs many queries)
• Can enforce business rules at the DB level
• Often faster — pre-compiled execution plan
⚠️ Trade-offs
• Harder to version-control than application code
• Logic split between app and database layers
• Less portable across different database engines
• Debugging can be less convenient than app-side code

📞 IN / OUT / INOUT Parameters & CALL

Procedures can accept input, return output, or both — using three parameter types: IN, OUT, and INOUT.

IN

Input only. Pass a value INTO the procedure. Default if not specified. Cannot be modified to return a value.

OUT

Output only. The procedure sets this value, and it's returned to the caller after CALL completes.

INOUT

Both. Pass a value in, the procedure can modify it, and the new value is returned.

📤 OUT Parameter Example
SQL
DELIMITER //

CREATE PROCEDURE get_batch_avg(
    IN  p_batch_id  VARCHAR(10),
    OUT p_avg_score DECIMAL(5,2)
)
BEGIN
    SELECT AVG(quiz_score) INTO p_avg_score
    FROM   guvi_learners
    WHERE  batch_id = p_batch_id;
END //

DELIMITER ;

-- Call and capture the OUT value into a session variable
CALL get_batch_avg('B01', @result);
SELECT @result AS batch_average;
-- batch_average: 79.40
🔄 INOUT Parameter Example
SQL
DELIMITER //

CREATE PROCEDURE apply_bonus_marks(INOUT p_score INT)
BEGIN
    SET p_score = p_score + 5;       -- add 5 bonus marks
    IF p_score > 100 THEN
        SET p_score = 100;            -- cap at 100
    END IF;
END //

DELIMITER ;

-- Set initial value, call, see the modified result
SET @score = 97;
CALL apply_bonus_marks(@score);
SELECT @score;   -- 100 (97+5=102, capped at 100)
🏢 Real-World GUVI Procedure — Enroll Student with Validation
SQL
DELIMITER //

CREATE PROCEDURE enroll_student(
    IN  p_name      VARCHAR(100),
    IN  p_email     VARCHAR(100),
    IN  p_batch_id  VARCHAR(10),
    OUT p_status    VARCHAR(50)
)
BEGIN
    DECLARE batch_exists INT DEFAULT 0;
    DECLARE seat_count   INT DEFAULT 0;

    -- Check if batch exists
    SELECT COUNT(*) INTO batch_exists FROM batches WHERE batch_id = p_batch_id;

    IF batch_exists = 0 THEN
        SET p_status = 'ERROR: Batch does not exist';
    ELSE
        -- Check seat capacity
        SELECT COUNT(*) INTO seat_count FROM students WHERE batch_id = p_batch_id;

        IF seat_count >= 50 THEN
            SET p_status = 'ERROR: Batch is full';
        ELSE
            INSERT INTO students (name, email, batch_id)
            VALUES (p_name, p_email, p_batch_id);
            SET p_status = 'SUCCESS: Student enrolled';
        END IF;
    END IF;
END //

DELIMITER ;

CALL enroll_student('Kavya Nair', 'kavya@guvi.in', 'B01', @status);
SELECT @status;  -- 'SUCCESS: Student enrolled'
✅
This procedure encapsulates validation logic (batch exists? seats available?) directly in the database — any application calling enroll_student() automatically gets this protection, without duplicating the logic in every app.
