Window Functions — Ranking & Analytics

Chapter 17 · Window Functions

See Each Row
In Context of Its Neighbours.

Window functions let you rank, compare, and accumulate values across rows — without collapsing them like GROUP BY does. Rank learners within their batch, compare this month's revenue to last month's, calculate running totals — all while keeping every row visible.

Window Functions vs GROUP BY — The Key Difference

GROUP BY collapses many rows into one summary row per group. Window functions calculate something ACROSS a group of rows, but keep every individual row in the output.

📦 GROUP BY — Collapses Rows
SELECT batch_id, AVG(score)
FROM   guvi_learners
GROUP BY batch_id;

Input: 100 rows → Output: 3 rows (one per batch). Individual learner rows are GONE.

🪟 Window Function — Keeps Rows
SELECT name, batch_id, score,
       AVG(score) OVER (PARTITION BY batch_id)
FROM   guvi_learners;

Input: 100 rows → Output: 100 rows — EVERY learner still appears, now WITH their batch average attached.

🎓
Real-world analogyGROUP BY is like a class photo replaced by just the class average height — you lose every individual student. A window function is like printing each student's photo WITH "class average: 5'6"" written next to them — you see both the individual and the context.
🥇 Ranking

ROW_NUMBER, RANK, DENSE_RANK, NTILE — order and rank rows within groups.

⏪ Offset

LAG, LEAD — look at the previous or next row's value, perfect for comparisons over time.

📈 Aggregate

SUM, AVG OVER — running totals, moving averages, cumulative metrics.

🪟 The OVER() Clause — PARTITION BY + ORDER BY

Every window function ends with OVER(...) — this is what makes it a "window" function. Inside the parentheses, two clauses control its behaviour: PARTITION BY and ORDER BY.

🧬 Anatomy of OVER()
RANK() OVER (PARTITION BY batch_id ORDER BY score DESC)
Function — what to calculate
PARTITION BY — group boundaries (like GROUP BY, but keeps rows)
ORDER BY — sort order within each partition
📋 PARTITION BY — Restarting the Calculation per Group

PARTITION BY divides rows into groups. The window function resets/recalculates separately for each partition — exactly like GROUP BY, but without collapsing rows.

SQL
SELECT
  name, batch_id, quiz_score,
  RANK() OVER (PARTITION BY batch_id ORDER BY quiz_score DESC) AS rank_in_batch
FROM   guvi_learners;
-- Rank resets to 1 at the start of EACH batch

🔄 No PARTITION BY = Whole Table is One Window
SQL
SELECT name, quiz_score,
       RANK() OVER (ORDER BY quiz_score DESC) AS overall_rank
FROM   guvi_learners;
-- No PARTITION BY → ranks across ALL learners, ignoring batch

🥇 ROW_NUMBER, RANK, DENSE_RANK — The Three Ranking Functions

All three assign a position number, but they behave differently when there are ties (equal values). This is the #1 most-asked SQL interview question.

📊 Side-by-Side Comparison — Watch What Happens with Ties

Notice Priya and Arjun are TIED at 85. Watch how each function handles it differently:

namescoreROW_NUMBER()RANK()DENSE_RANK()
Meera Iyer93111
Priya Sharma85222
Arjun Patel85322
Sneha Nair7844 ← skips 3!3 ← no skip
Ravi Kumar68554
ROW_NUMBER()

Always unique, sequential: 1,2,3,4,5. Ties get DIFFERENT numbers, in arbitrary (or insertion) order.

RANK()

Ties get the SAME rank, but the next rank SKIPS — 1,2,2,4,5. Matches "Olympic" ranking style.

DENSE_RANK()

Ties get the SAME rank, NO skip in the next rank — 1,2,2,3,4. No gaps ever.

🏫 GUVI Example — All Three Together
SQL
SELECT
  name, batch_id, quiz_score,
  ROW_NUMBER() OVER (PARTITION BY batch_id ORDER BY quiz_score DESC) AS row_num,
  RANK()       OVER (PARTITION BY batch_id ORDER BY quiz_score DESC) AS rank_pos,
  DENSE_RANK() OVER (PARTITION BY batch_id ORDER BY quiz_score DESC) AS dense_rank_pos
FROM   guvi_learners
ORDER BY batch_id, quiz_score DESC;
💡
When to use which?
ROW_NUMBER() → when you need a guaranteed unique number per row (e.g., pagination, "pick exactly one per group")
RANK() → leaderboard / competition style ranking, where tied scores should show the same position
DENSE_RANK() → when you want clean, gap-free tier numbers (e.g., "top 3 distinct scores" — even with ties, you still get tiers 1, 2, 3)

📊 NTILE(n) — Percentile Bucketing

NTILE divides rows into N roughly equal-sized buckets — perfect for quartiles, performance tiers, or percentile groupings.

📖 NTILE Syntax
SQL
NTILE(n) OVER (ORDER BY column DESC)
-- Divides rows into n equal-ish groups, numbered 1 to n
🏫 GUVI Example — Quartile Performance Tiers
SQL
-- Divide all learners into 4 performance quartiles
SELECT
  name, quiz_score,
  NTILE(4) OVER (ORDER BY quiz_score DESC) AS quartile,
  CASE NTILE(4) OVER (ORDER BY quiz_score DESC)
    WHEN 1 THEN '🏆 Top 25%'
    WHEN 2 THEN '👍 Above Average'
    WHEN 3 THEN '📊 Below Average'
    ELSE '⚠️ Needs Support'
  END AS performance_tier
FROM   guvi_learners
WHERE  quiz_score IS NOT NULL;

🏆 Top N Per Group — The Classic Pattern

One of the most common real-world SQL tasks: "give me the top 3 per category." This is impossible with plain GROUP BY + LIMIT — but trivial with window functions.

📖 The Pattern — Window Function + Outer Filter
SQL
-- Top 2 scorers in EACH batch
WITH ranked AS (
  SELECT
    name, batch_id, quiz_score,
    ROW_NUMBER() OVER (PARTITION BY batch_id ORDER BY quiz_score DESC) AS rn
  FROM   guvi_learners
)
SELECT name, batch_id, quiz_score
FROM   ranked
WHERE  rn <= 2;        ← the filter that makes "Top N" work
🚨
Critical rule!You CANNOT use window functions directly inside a WHERE clause — WHERE ROW_NUMBER() OVER(...) <= 2 is a SQL error. Window functions are evaluated AFTER WHERE in the execution order. You must wrap them in a CTE or subquery, then filter the OUTER query.

🚂 IRCTC Example — Most Recent Booking Per Customer
SQL
-- For each customer, get only their MOST RECENT booking
WITH latest AS (
  SELECT
    customer_id, pnr_number, journey_date, source_city, destination_city,
    ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY journey_date DESC) AS rn
  FROM   irctc_bookings
)
SELECT * FROM latest WHERE rn = 1;

⏪ LAG() & LEAD() — Compare to Previous/Next Row

LAG looks BACKWARD to a previous row's value. LEAD looks FORWARD to the next row's value. Both are essential for time-series comparisons like month-over-month growth.

📖 Syntax
SQL
LAG(column, offset, default) OVER (ORDER BY ...)
LEAD(column, offset, default) OVER (ORDER BY ...)

-- offset (optional, default 1) = how many rows back/forward
-- default (optional) = value to use when there's no previous/next row
📈 Month-over-Month Growth — The Classic LAG Use Case
SQL
SELECT
  month_name,
  revenue,
  LAG(revenue, 1) OVER (ORDER BY month_num) AS prev_month_revenue,
  revenue - LAG(revenue, 1) OVER (ORDER BY month_num) AS change,
  ROUND(
    100.0 * (revenue - LAG(revenue, 1) OVER (ORDER BY month_num))
    / NULLIF(LAG(revenue, 1) OVER (ORDER BY month_num), 0),
    1
  ) AS pct_growth
FROM   monthly_revenue
ORDER BY month_num;
month_namerevenueprev_month_revenuechangepct_growth
January₹12,40,000NULLNULLNULL
February₹13,80,000₹12,40,000+1,40,000+11.3%
March₹11,90,000₹13,80,000-1,90,000-13.8%

💡
January's prev_month_revenue is NULL — there's no row before it. Always wrap LAG-based division in NULLIF to avoid divide-by-zero errors on the first row or any zero-revenue month.
📉 Detecting Dropoff Months — GUVI Enrollment Trend
SQL
-- Flag months where enrollment DROPPED compared to the previous month
SELECT
  month_name,
  new_enrollments,
  LAG(new_enrollments) OVER (ORDER BY month_num) AS prev_enrollments,
  CASE
    WHEN new_enrollments < LAG(new_enrollments) OVER (ORDER BY month_num)
    THEN '📉 Dropoff'
    ELSE '📈 Growth'
  END AS trend
FROM   monthly_enrollment_stats
ORDER BY month_num;
⏩ LEAD() — Looking Forward
SQL
-- For each batch start date, show when the NEXT batch starts
SELECT
  batch_name,
  start_date,
  LEAD(start_date) OVER (ORDER BY start_date) AS next_batch_start,
  DATEDIFF(
    LEAD(start_date) OVER (ORDER BY start_date),
    start_date
  ) AS days_until_next_batch
FROM   batches
ORDER BY start_date;

📈 Running Totals & Moving Averages

SUM() OVER and AVG() OVER let you compute cumulative or rolling metrics — essential for financial dashboards, growth charts, and trend smoothing.

💰 Running Revenue Total — SUM() OVER PARTITION BY
SQL
SELECT
  order_date,
  daily_revenue,
  SUM(daily_revenue) OVER (ORDER BY order_date) AS running_total
FROM   zomato_daily_sales
ORDER BY order_date;
order_datedaily_revenuerunning_total
2024-06-01₹45,000₹45,000
2024-06-02₹52,000₹97,000
2024-06-03₹38,000₹1,35,000
2024-06-04₹61,000₹1,96,000
💡
Why does SUM() OVER (ORDER BY ...) accumulate?When ORDER BY is present without an explicit frame, the default window frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — meaning each row sums everything from the start up to itself. This is exactly what creates a "running total."
📊 SUM() OVER with PARTITION BY — Running Total Per Group
SQL
-- Running total of fees collected, PER BATCH (resets at each batch)
SELECT
  batch_id,
  payment_date,
  amount,
  SUM(amount) OVER (
    PARTITION BY batch_id
    ORDER BY payment_date
  ) AS running_total_per_batch
FROM   payments
ORDER BY batch_id, payment_date;
📉 Moving Average — Smoothing Out Daily Noise
SQL
-- 3-day moving average of daily orders — smooths spikes
SELECT
  order_date,
  daily_orders,
  ROUND(
    AVG(daily_orders) OVER (
      ORDER BY order_date
      ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ), 1
  ) AS moving_avg_3day
FROM   zomato_daily_sales
ORDER BY order_date;

🪟 ROWS BETWEEN — Window Frames

The frame clause precisely controls WHICH rows are included in the calculation relative to the current row. This is what gives you fine control over running totals vs moving windows.

📋 Common Frame Patterns
Frame clauseMeaning
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROWFrom the very start through the current row — classic running total
ROWS BETWEEN 2 PRECEDING AND CURRENT ROWCurrent row + 2 rows before it = 3-row moving window
ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWINGCentred window — 1 before, current, 1 after = 3-row centred average
ROWS BETWEEN CURRENT ROW AND UNBOUNDED FOLLOWINGFrom current row to the end — useful for "remaining total"
ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWINGThe ENTIRE partition, every time — same as a plain aggregate per partition

📖 Default Frame Behaviour — Important Gotcha
⚠️
If you use ORDER BY without specifying ROWS BETWEEN, the default frame is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — which creates a running total automatically. If you DON'T use ORDER BY at all, the default frame is the WHOLE partition — giving you the same total on every row (like a plain aggregate).
SQL — Comparison
-- WITH ORDER BY → running total (frame = start to current row)
SUM(amount) OVER (PARTITION BY batch_id ORDER BY payment_date)

-- WITHOUT ORDER BY → same total repeated on every row (frame = whole partition)
SUM(amount) OVER (PARTITION BY batch_id)

🎯 FIRST_VALUE(), LAST_VALUE(), NTH_VALUE()

These functions grab a specific value from within the window frame — the first row, the last row, or the Nth row, based on the ORDER BY.

📖 Syntax & Use Cases
SQL
FIRST_VALUE(column) OVER (PARTITION BY ... ORDER BY ...)
LAST_VALUE(column)  OVER (PARTITION BY ... ORDER BY ... ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING)
NTH_VALUE(column, n) OVER (PARTITION BY ... ORDER BY ...)
🚨
LAST_VALUE gotcha!By default, the window frame stops at the CURRENT row — so LAST_VALUE without an explicit frame just returns the current row's own value (not actually the "last" value of the partition). You MUST add ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING to get the true last value across the whole partition.
🏫 GUVI Example — Compare Each Score to the Top Scorer in Their Batch
SQL
SELECT
  name, batch_id, quiz_score,
  FIRST_VALUE(name) OVER (
    PARTITION BY batch_id ORDER BY quiz_score DESC
  ) AS batch_top_scorer,

  FIRST_VALUE(quiz_score) OVER (
    PARTITION BY batch_id ORDER BY quiz_score DESC
  ) - quiz_score AS points_behind_leader

FROM   guvi_learners
ORDER BY batch_id, quiz_score DESC;

🥉 NTH_VALUE — Get the 2nd Highest Score
SQL
SELECT DISTINCT
  batch_id,
  NTH_VALUE(quiz_score, 2) OVER (
    PARTITION BY batch_id ORDER BY quiz_score DESC
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
  ) AS second_highest_score
FROM   guvi_learners;

🇮🇳 Real-World Recipes

Three complete, exam-style problems combining everything from this chapter.

📊 Recipe 1 — Rank Learners by Score Per Batch (with tier labels)
SQL
SELECT
  name, batch_id, quiz_score,
  RANK() OVER (PARTITION BY batch_id ORDER BY quiz_score DESC) AS batch_rank,
  CASE
    WHEN RANK() OVER (PARTITION BY batch_id ORDER BY quiz_score DESC) = 1 THEN '🥇 Topper'
    WHEN RANK() OVER (PARTITION BY batch_id ORDER BY quiz_score DESC) <= 3 THEN '🏅 Top 3'
    ELSE ''
  END AS badge
FROM   guvi_learners
ORDER BY batch_id, batch_rank;
📉 Recipe 2 — Detect Enrollment Dropoff Months
SQL
WITH monthly AS (
  SELECT DATE_FORMAT(enrolled_on, '%Y-%m') AS month,
         COUNT(*) AS new_learners
  FROM   guvi_learners
  GROUP BY DATE_FORMAT(enrolled_on, '%Y-%m')
)
SELECT
  month, new_learners,
  LAG(new_learners) OVER (ORDER BY month) AS prev_month,
  ROUND(100.0 * (new_learners - LAG(new_learners) OVER (ORDER BY month))
        / NULLIF(LAG(new_learners) OVER (ORDER BY month),0), 1) AS pct_change,
  CASE WHEN new_learners < LAG(new_learners) OVER (ORDER BY month) THEN '⚠️ DROPOFF' ELSE '✅ OK' END AS flag
FROM   monthly
ORDER BY month;
💰 Recipe 3 — Running Revenue Totals with Daily + Cumulative
SQL
SELECT
  payment_date,
  SUM(amount)                                            AS daily_revenue,
  SUM(SUM(amount)) OVER (ORDER BY payment_date)          AS running_total,
  ROUND(
    AVG(SUM(amount)) OVER (
      ORDER BY payment_date
      ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ), 0
  )                                                       AS rolling_7day_avg
FROM   payments
GROUP BY payment_date
ORDER BY payment_date;
-- Note: SUM(SUM(amount)) OVER(...) — nesting an aggregate inside a window function
-- is valid because GROUP BY runs first, THEN window functions apply to the grouped result
