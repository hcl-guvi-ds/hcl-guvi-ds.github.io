Triggers & Events

Chapter 15 · Triggers & Events

SQL That Runs Itself.
Automatic Reactions to Data Changes.

A trigger is a piece of SQL logic that fires automatically whenever data changes — no application code required. Audit logs, timestamp updates, business rule enforcement — all handled silently by the database itself.

What is a Trigger?

A trigger is SQL code bound to a table that automatically executes when a specific event happens — INSERT, UPDATE, or DELETE. You never call a trigger directly; it "triggers" itself.

➕ INSERT trigger

Fires when a new row is added. Great for setting default values, validation, or logging new records.

✏️ UPDATE trigger

Fires when an existing row changes. Perfect for audit logs, auto-timestamps, change tracking.

🗑️ DELETE trigger

Fires when a row is removed. Used to archive deleted data, or block deletion of critical records.

🎓
Real-world analogyThink of a trigger like a motion-sensor light. You don't manually turn it on — it activates automatically the instant something happens (someone walks by). A database trigger works the same way: the instant an INSERT/UPDATE/DELETE happens, the trigger "senses" it and fires.
⚖️
Triggers vs Stored ProceduresA stored procedure (Chapter 14) must be explicitly CALLed by someone. A trigger fires automatically and invisibly — you can't call it directly, and you often don't even know it ran unless you check the logs it produces.

🧬 CREATE TRIGGER — Syntax & Anatomy

Every trigger has 5 essential parts: name, timing (BEFORE/AFTER), event (INSERT/UPDATE/DELETE), the table it watches, and the logic that runs.

📖 Full Syntax
MySQL
DELIMITER //

CREATE TRIGGER trigger_name
BEFORE | AFTER  ← TIMING: before or after the change
INSERT | UPDATE | DELETE  ← EVENT: which operation triggers it
ON table_name             ← TABLE: which table is being watched
FOR EACH ROW             ← SCOPE: runs once per affected row
BEGIN
    -- your SQL logic here
END //

DELIMITER ;
🏫 GUVI Example — Annotated Trigger
SQL — Annotated
DELIMITER //

CREATE TRIGGER trg_before_student_insert   ← descriptive name
BEFORE INSERT                       ← runs BEFORE the row is saved
ON students                       ← watching the students table
FOR EACH ROW                      ← once per inserted row
BEGIN
    -- Auto-set enrollment date if not provided
    IF NEW.enrolled_on IS NULL THEN
        SET NEW.enrolled_on = CURDATE();
    END IF;

    -- Standardise email to lowercase
    SET NEW.email = LOWER(TRIM(NEW.email));
END //

DELIMITER ;

-- This now happens AUTOMATICALLY on every insert — no app code needed!
INSERT INTO students (name, email) VALUES ('Kavya Nair', '  KAVYA@GUVI.IN  ');
-- Stored as: email = 'kavya@guvi.in', enrolled_on = today's date
✅
The power of triggersEvery application that inserts into students — the website, the mobile app, an admin tool, a bulk import script — automatically gets this email cleanup and default date logic. The rule lives in ONE place: the database.

⏱️ BEFORE vs AFTER Triggers

Timing determines whether your trigger runs before the change is committed (so you can still modify or block it) or after (so you can react to what already happened).

Execution Order

BEFORE trigger fires
→
Row is actually INSERTED/UPDATED/DELETED
→
AFTER trigger fires
⏪ BEFORE Trigger

Runs before the change happens. Can MODIFY the incoming data (via NEW) or CANCEL the operation entirely by raising an error.

  • Validate or clean data before saving
  • Set default/computed values
  • Block invalid operations
⏩ AFTER Trigger

Runs after the change is saved. Cannot modify the row that just changed, but can update OTHER tables (logs, audit trails, counters).

  • Write to an audit/history table
  • Update related summary tables
  • Send notifications (via app polling)
🛡️ BEFORE Trigger — Validation & Blocking
SQL
DELIMITER //

CREATE TRIGGER trg_validate_score
BEFORE INSERT ON guvi_learners
FOR EACH ROW
BEGIN
    IF NEW.quiz_score < 0 OR NEW.quiz_score > 100 THEN
        -- Block the insert entirely with a custom error
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Quiz score must be between 0 and 100';
    END IF;
END //

DELIMITER ;

-- This will FAIL with our custom error message:
INSERT INTO guvi_learners (name, quiz_score) VALUES ('Test', 150);
-- ERROR 1644 (45000): Quiz score must be between 0 and 100
📝 AFTER Trigger — Logging to Another Table
SQL
DELIMITER //

CREATE TRIGGER trg_log_new_student
AFTER INSERT ON students
FOR EACH ROW
BEGIN
    -- The row already exists in 'students' — now log it elsewhere
    INSERT INTO enrollment_log (student_id, name, action, logged_at)
    VALUES (NEW.student_id, NEW.name, 'NEW_ENROLLMENT', NOW());
END //

DELIMITER ;

🔀 NEW and OLD Row References

Inside a trigger, you can access the row's data using two special keywords: NEW (the incoming/updated values) and OLD (the previous values before the change).

📋 NEW vs OLD Availability by Event Type
EventOLD available?NEW available?Meaning
INSERT❌ No✅ YesRow didn't exist before — only NEW data exists
UPDATE✅ Yes✅ YesOLD = before change, NEW = after change
DELETE✅ Yes❌ NoRow is being removed — only OLD data exists

🏫 GUVI Example — Only Log If Score Actually Changed
SQL
DELIMITER //

CREATE TRIGGER trg_score_change_log
AFTER UPDATE ON guvi_learners
FOR EACH ROW
BEGIN
    -- Only log if the score actually changed (avoid noise from unrelated updates)
    IF OLD.quiz_score != NEW.quiz_score OR
       (OLD.quiz_score IS NULL AND NEW.quiz_score IS NOT NULL) THEN

        INSERT INTO score_change_log
            (student_id, old_score, new_score, changed_at)
        VALUES
            (NEW.student_id, OLD.quiz_score, NEW.quiz_score, NOW());
    END IF;
END //

DELIMITER ;
💡
Important: in a DELETE trigger, you can only use OLD.column — there is no NEW because the row is being removed. In an INSERT trigger, you can only use NEW.column — there is no OLD because the row didn't exist before.
🚫 Prevent Deletion of Critical Records
SQL
DELIMITER //

CREATE TRIGGER trg_prevent_admin_delete
BEFORE DELETE ON students
FOR EACH ROW
BEGIN
    IF OLD.role = 'admin' THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Cannot delete admin accounts!';
    END IF;
END //

DELIMITER ;

-- This will be BLOCKED:
DELETE FROM students WHERE student_id = 'ADMIN01';
-- ERROR 1644 (45000): Cannot delete admin accounts!

📐 FOR EACH ROW — Row-Level Triggers

MySQL only supports row-level triggers — meaning the trigger fires once for EVERY row affected, not once per statement. This matters a lot for bulk operations.

🔁 What "Once Per Row" Really Means
SQL
-- This single UPDATE statement affects 50 rows
UPDATE guvi_learners
SET    quiz_score = quiz_score + 5
WHERE  batch_id = 'B01';   -- 50 students in B01

-- If a trigger exists: AFTER UPDATE ON guvi_learners FOR EACH ROW
-- → it fires 50 TIMES, once per affected row
-- → NOT once for the whole UPDATE statement
⚠️
Performance implication!A trigger that does expensive work (like another INSERT) will run that work 50 times for a 50-row bulk update. For very large bulk operations, this can slow things down significantly — be mindful of what logic you put inside triggers.
🔢 Multiple Triggers on the Same Table

You can have multiple triggers on the same table, even for the same event — MySQL lets you control their execution order.

SQL — MySQL 5.7.2+
-- Trigger 1: runs first
CREATE TRIGGER trg_validate_score
BEFORE INSERT ON guvi_learners
FOR EACH ROW
BEGIN
    IF NEW.quiz_score < 0 THEN SET NEW.quiz_score = 0; END IF;
END //

-- Trigger 2: explicitly placed AFTER trigger 1
CREATE TRIGGER trg_set_grade
BEFORE INSERT ON guvi_learners
FOR EACH ROW
FOLLOWS trg_validate_score    ← controls ordering!
BEGIN
    SET NEW.grade = CASE
        WHEN NEW.quiz_score >= 90 THEN 'A'
        WHEN NEW.quiz_score >= 75 THEN 'B'
        ELSE 'C'
    END;
END //
💡
Use FOLLOWS trigger_name or PRECEDES trigger_name to control the exact order when multiple triggers fire for the same event on the same table.

🛠️ Trigger Use Cases

Six classic real-world patterns where triggers replace application-level logic with database-level automation.

📋
1. Audit Log — Track Every Change
AFTER UPDATE/DELETE trigger writes old and new values to a history table — a permanent record of who changed what and when.
🕐
2. Auto-Update Timestamp
BEFORE UPDATE trigger automatically sets updated_at = NOW() — no application needs to remember to do this manually.
⚖️
3. Enforce Business Rules
BEFORE INSERT/UPDATE trigger validates data (e.g., score 0-100, no negative fees) and blocks invalid changes with SIGNAL.
🔒
4. Prevent Critical Deletes
BEFORE DELETE trigger blocks deletion of protected records (admin accounts, paid invoices) by raising a custom error.
🔗
5. Cascade Custom Logic
AFTER DELETE on students could also clean up related rows in tables that don't have a formal foreign key relationship.
📊
6. Maintain Summary Tables
AFTER INSERT/DELETE trigger keeps a batch_student_count column updated automatically, avoiding expensive COUNT() queries.
🕐 Use Case 2 in Detail — Auto-Update Timestamp
SQL
-- Table has an updated_at column
ALTER TABLE guvi_learners ADD COLUMN updated_at TIMESTAMP;

DELIMITER //

CREATE TRIGGER trg_auto_timestamp
BEFORE UPDATE ON guvi_learners
FOR EACH ROW
BEGIN
    SET NEW.updated_at = NOW();
END //

DELIMITER ;

-- Every UPDATE — from ANY source — now stamps the time automatically
UPDATE guvi_learners SET quiz_score = 90 WHERE student_id = 'S01';
-- updated_at is automatically set to the current timestamp
🔗 Use Case 5 in Detail — Cascade Beyond Foreign Keys
SQL
-- When a batch is deleted, archive its students instead of relying on FK CASCADE
DELIMITER //

CREATE TRIGGER trg_archive_on_batch_delete
BEFORE DELETE ON batches
FOR EACH ROW
BEGIN
    -- Move affected students to an archive table BEFORE the batch disappears
    INSERT INTO archived_students
        SELECT * FROM students WHERE batch_id = OLD.batch_id;

    -- Log this significant event
    INSERT INTO system_log (event, detail, logged_at)
    VALUES ('BATCH_DELETED', CONCAT('Batch ', OLD.batch_name, ' removed'), NOW());
END //

DELIMITER ;

🏆 Build: Audit Trigger on Learner Table

A complete, production-ready audit trail system — log every UPDATE to guvi_learners into a history table automatically. This is one of the most commonly requested real-world features.

🎯
The GoalEvery time someone updates a row in guvi_learners (score correction, city change, batch transfer), automatically record: which row changed, what the old value was, what the new value is, and exactly when it happened — with zero application code.
Step 1 — Create the Audit Log Table
SQL
CREATE TABLE learner_audit_log (
    log_id        INT AUTO_INCREMENT PRIMARY KEY,
    student_id    VARCHAR(10)     NOT NULL,
    field_changed VARCHAR(50)     NOT NULL,
    old_value     VARCHAR(100),
    new_value     VARCHAR(100),
    changed_by    VARCHAR(50)     DEFAULT CURRENT_USER(),
    changed_at    TIMESTAMP      DEFAULT CURRENT_TIMESTAMP
);
Step 2 — Create the AFTER UPDATE Trigger
SQL
DELIMITER //

CREATE TRIGGER trg_audit_learner_update
AFTER UPDATE ON guvi_learners
FOR EACH ROW
BEGIN
    -- Log score changes
    IF OLD.quiz_score <=> NEW.quiz_score IS FALSE THEN
        INSERT INTO learner_audit_log
            (student_id, field_changed, old_value, new_value)
        VALUES
            (NEW.student_id, 'quiz_score',
             CAST(OLD.quiz_score AS CHAR), CAST(NEW.quiz_score AS CHAR));
    END IF;

    -- Log city changes
    IF OLD.city <=> NEW.city IS FALSE THEN
        INSERT INTO learner_audit_log
            (student_id, field_changed, old_value, new_value)
        VALUES
            (NEW.student_id, 'city', OLD.city, NEW.city);
    END IF;

    -- Log batch transfers
    IF OLD.batch_id <=> NEW.batch_id IS FALSE THEN
        INSERT INTO learner_audit_log
            (student_id, field_changed, old_value, new_value)
        VALUES
            (NEW.student_id, 'batch_id', OLD.batch_id, NEW.batch_id);
    END IF;
END //

DELIMITER ;
💡
Why <=> (null-safe equals)?Regular != returns NULL (not TRUE) when comparing against NULL, so changes involving NULL would be silently skipped. The <=> operator handles NULL comparisons correctly — essential for robust audit triggers.
Step 3 — Test It
SQL
-- Make a change
UPDATE guvi_learners
SET    quiz_score = 92, city = 'Bangalore'
WHERE  student_id = 'S01';

-- Check the audit trail — TWO rows logged automatically!
SELECT * FROM learner_audit_log
WHERE  student_id = 'S01'
ORDER BY changed_at DESC;

✅
One UPDATE statement, two field changes, automatically logged as two separate audit entries — with zero application-side logging code. This pattern is used in HDFC's transaction systems, GUVI's grade-change tracking, and IRCTC's booking modification logs.

🗂️ DROP TRIGGER / SHOW TRIGGERS

Managing the lifecycle of your triggers — viewing what exists, and removing triggers you no longer need.

🔍 SHOW TRIGGERS — See What's Active
SQL
-- List all triggers in the current database
SHOW TRIGGERS;

-- Filter to a specific table
SHOW TRIGGERS WHERE `Table` = 'guvi_learners';

-- More detail via information_schema
SELECT trigger_name, event_manipulation, event_object_table, action_timing
FROM   information_schema.triggers
WHERE  trigger_schema = 'guvi_db';
🗑️ DROP TRIGGER
SQL
DROP TRIGGER IF EXISTS trg_audit_learner_update;

-- To MODIFY a trigger, you must DROP and recreate it —
-- there's no ALTER TRIGGER in MySQL
DROP TRIGGER IF EXISTS trg_validate_score;

DELIMITER //
CREATE TRIGGER trg_validate_score
BEFORE INSERT ON guvi_learners
FOR EACH ROW
BEGIN
    -- updated logic here
END //
DELIMITER ;
⚠️
MySQL has no ALTER TRIGGER for changing logic. To update a trigger, you must DROP it and CREATE it again with the new logic.

📅 Scheduled Events with CREATE EVENT

While triggers react to data changes, EVENTS run on a time schedule — like a cron job that lives inside the database itself.

⏰
Triggers vs Events
Trigger → fires in response to a data change (INSERT/UPDATE/DELETE)
Event → fires on a time schedule (every hour, every night at 2 AM, once on a specific date)
📖 CREATE EVENT Syntax
MySQL
-- First, enable the event scheduler (server-level setting)
SET GLOBAL event_scheduler = ON;

DELIMITER //

CREATE EVENT event_name
ON SCHEDULE EVERY 1 DAY          -- recurring schedule
STARTS '2024-06-15 02:00:00'      -- when to start
DO
BEGIN
    -- SQL logic to run on schedule
END //

DELIMITER ;
🏫 GUVI Example — Nightly Cleanup & Summary
SQL
DELIMITER //

CREATE EVENT nightly_learner_summary
ON SCHEDULE EVERY 1 DAY
STARTS '2024-06-15 02:00:00'
DO
BEGIN
    -- Refresh a daily summary table every night at 2 AM
    DELETE FROM daily_batch_summary;

    INSERT INTO daily_batch_summary (batch_id, learner_count, avg_score, summary_date)
    SELECT batch_id, COUNT(*), ROUND(AVG(quiz_score),1), CURDATE()
    FROM   guvi_learners
    GROUP BY batch_id;

    -- Also clean up old session logs (older than 90 days)
    DELETE FROM session_logs
    WHERE  log_date < DATE_SUB(CURDATE(), INTERVAL 90 DAY);
END //

DELIMITER ;
🗂️ Managing Events
SQL
-- List all events
SHOW EVENTS;

-- Disable an event temporarily (without deleting it)
ALTER EVENT nightly_learner_summary DISABLE;

-- Re-enable it
ALTER EVENT nightly_learner_summary ENABLE;

-- Delete an event entirely
DROP EVENT IF EXISTS nightly_learner_summary;

-- One-time event (no EVERY clause — runs once and disappears)
CREATE EVENT one_time_migration
ON SCHEDULE AT '2024-07-01 00:00:00'
DO
  UPDATE guvi_learners SET status = 'archived' WHERE enrolled_on < '2023-01-01';
