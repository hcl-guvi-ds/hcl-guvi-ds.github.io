Topic 1

SELECT & FROM — Retrieving Data

SELECT retrieves data from a table. FROM tells SQL which table to look in. Together, they form the foundation of every SQL query ever written.

SYNTAX
SELECT column1, column2, ... -- or * for all
FROM table_name;
SELECT name, city, score ← Which columns to show
FROM   students;             ← Which table to read from
🎮 Try It Live Exercise 1
Edit the query and press Run
💡 Use * to select all columns, or list specific ones separated by commas
⚠️
Avoid SELECT * in production code
While SELECT * is great for exploration, always specify column names in production queries. Why?
1. Fetching unused columns wastes network bandwidth
2. If someone adds a column, your query's output changes unexpectedly
3. It makes code harder to understand — the reader doesn't know which columns are relevant
🎮 Try It Live Exercise 2
💡 SELECT name, city, course, score → only these 4 columns appear in results

Topic 2

Column Aliases — Renaming Output with AS

An alias gives a column (or table) a temporary name in the output. It doesn't change the actual column name in the database — it only affects how results are displayed. Aliases are essential for making output readable, especially for computed columns.

SYNTAX
SELECT column_name AS alias_name
FROM table_name;
🎮 Aliases in Action Exercise 3
💡 AS is optional — SELECT name student_name works too, but AS makes it explicit and readable
📌
Alias Rules
✅ Use quotes for aliases with spaces: score AS "Total Marks"
✅ Aliases can be used in ORDER BY but NOT in WHERE (SQL processes WHERE before SELECT)
✅ Aliases are commonly used for: computed columns, long column names, aggregation functions
✅ Table aliases shorten JOINs: FROM students s JOIN courses c ON s.id = c.id
🔎
Topic 3

WHERE — Filtering Rows

WHERE filters rows — only rows that satisfy the condition are returned. It's the most important clause after SELECT and FROM. Without WHERE, you get ALL rows; with WHERE, you get only what you need.

SYNTAX
SELECT columns
FROM table
WHERE condition;
🇮🇳 Real-World Use Cases
WHERE in everyday queries
• Swiggy: WHERE city = 'Mumbai' AND order_status = 'delivered'
• IRCTC: WHERE train_date = '2024-12-25' AND from_station = 'MAS'
• GUVI: WHERE course = 'SQL' AND score >= 80
• Bank: WHERE account_type = 'Savings' AND balance < 1000
🎮 WHERE in Action Exercise 4
💡 String values need quotes ('Chennai'), numbers don't (score >= 80)
💡
SQL Execution Order — WHERE runs before SELECT!
SQL doesn't execute top-to-bottom. The order is: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. This is why you cannot use a column alias in WHERE — the alias is defined in SELECT, but WHERE runs first.
⚖️

Topic 5

AND, OR, NOT — Logical Operators

Logical operators combine multiple conditions in a WHERE clause. They work exactly like boolean logic in Python/programming.

AND — Both conditions must be true
Narrows results — fewer rows returned
Like filtering a dataset with multiple conditions
WHERE city='Chennai' AND score>80
Only Chennai students with score above 80
OR — At least one condition must be true
Widens results — more rows returned
Like a union of two filters
WHERE city='Chennai' OR city='Delhi'
Students from Chennai OR Delhi (or both)
⚠️
Operator Precedence — AND evaluates before OR!
WHERE city='Chennai' OR city='Delhi' AND score > 80
This is read as: WHERE city='Chennai' OR (city='Delhi' AND score > 80) — NOT what you might intend!

Always use parentheses to be explicit:
WHERE (city='Chennai' OR city='Delhi') AND score > 80
🎮 AND, OR, NOT Exercise 6
💡 Use parentheses liberally — they make AND/OR logic unambiguous and readable

Topic 6

BETWEEN & IN — Shorthand Filters

These operators are shortcuts that make your WHERE clauses shorter and more readable — and they perform identically to the expanded form.

BETWEEN … AND … — Range filter
Inclusive — includes both boundary values
Works on numbers, dates, and strings
WHERE score BETWEEN 60 AND 80
= WHERE score >= 60 AND score <= 80
IN (list) — Match any value in a list
Shorthand for multiple OR conditions
Can use with numbers, strings, subqueries
WHERE city IN ('Chennai','Delhi','Pune')
= WHERE city='Chennai' OR city='Delhi'…
🎮 BETWEEN & IN Exercise 7
💡 NOT BETWEEN and NOT IN also work — they're negations of the range/list filters
📌
BETWEEN on dates — very common in analytics
WHERE order_date BETWEEN '2024-01-01' AND '2024-03-31'
This gets all orders in Q1 2024. BETWEEN always includes both end dates. For dates, remember: '2024-03-31' means midnight at the start of that day — if you want all of March 31st, use < '2024-04-01' instead.
🔤
Topic 7

LIKE — Pattern Matching with Wildcards

LIKE is used for partial text matching using wildcards. Unlike = which needs an exact match, LIKE lets you match patterns.

%
Percent (%) — Any sequence

Matches zero or more characters — any sequence of any length

'A%' → Starts with A → Ananya, Arjun
'%a' → Ends with a → Priya, Neha
'%ar%' → Contains "ar" → Karthik, Arjun
'%' → Matches everything
_
Underscore (_) — One character

Matches exactly ONE character — any single character

'_ai%' → 2nd+3rd chars "ai" → Karthik, Sai
'R___l' → R + 3 chars + l → Rahul
'__@%' → 3rd char is @ (emails)
'___' → Exactly 3 characters
🎮 LIKE Pattern Matching Exercise 8
💡 LIKE is case-insensitive in MySQL by default. Use BINARY LIKE for case-sensitive matching
💡
LIKE vs = performance
WHERE name = 'Priya' uses an index (fast). WHERE name LIKE 'Pr%' can use a prefix index (reasonably fast). WHERE name LIKE '%riya' or WHERE name LIKE '%riy%' cannot use an index — it scans the entire table (slow on large tables). Prefer prefix patterns when possible.
∅
Topic 8

IS NULL / IS NOT NULL — Handling Missing Data

NULL in SQL means "unknown" or "missing value" — it is NOT the same as zero, empty string, or false. NULL is the absence of a value. This has a very important consequence: you CANNOT use = to check for NULL.

⚠️
The NULL trap — a very common mistake!
WHERE email = NULL → NEVER WORKS — returns 0 rows always!
WHERE email != NULL → NEVER WORKS — returns 0 rows always!

Why? Because NULL = NULL is UNKNOWN in SQL (not TRUE). NULL is not equal to anything, including itself.

✅ Correct: WHERE email IS NULL
✅ Correct: WHERE email IS NOT NULL
🎮 NULL Handling Exercise 9
💡 Always use IS NULL / IS NOT NULL — never use = NULL or != NULL!
🎯
NULL in arithmetic and comparisons
Any arithmetic with NULL gives NULL: score + NULL = NULL
Any comparison with NULL gives UNKNOWN: score > NULL = UNKNOWN
Use COALESCE(score, 0) to replace NULL with a default value for calculations.
Use NULLIF(score, 0) to convert a specific value to NULL (opposite of COALESCE).
↕️
Topic 9

ORDER BY — Sorting Results

ORDER BY sorts the result set. Without it, SQL returns rows in no guaranteed order — the order depends on how the database engine retrieves them internally.

SYNTAX
SELECT columns
FROM table
WHERE condition -- optional
ORDER BY column1 ASC|DESC, column2 ASC|DESC, ...;
ASC (default)
Ascending order — smallest to largest, A to Z, oldest to newest
1, 2, 3 ... | A, B, C ... | Jan, Feb, Mar...
DESC
Descending order — largest to smallest, Z to A, newest to oldest
100, 99, 98 ... | Z, Y, X ... | Dec, Nov, Oct...
Multi-column sort
Sort by first column, then by second column for ties in the first
ORDER BY city ASC, score DESC
NULL in sorting
MySQL: NULLs appear first in ASC, last in DESC. PostgreSQL: opposite (NULLS LAST/FIRST)
ORDER BY email ASC NULLS LAST (PostgreSQL)
🎮 ORDER BY Exercise 10
💡 ORDER BY is always the LAST clause to execute. Use column positions: ORDER BY 3 DESC = sort by 3rd column
✨
Topic 10

DISTINCT — Removing Duplicates

DISTINCT removes duplicate rows from the result set. It's placed immediately after SELECT and applies to the entire row (not just one column when multiple columns are selected).

❌ Without DISTINCT
SELECT city FROM students;
Chennai
Chennai ← duplicate!
Delhi
Bangalore
Delhi ← duplicate!
Chennai ← duplicate!
✅ With DISTINCT
SELECT DISTINCT city FROM students;
Chennai
Delhi
Bangalore
Mumbai
Hyderabad
🎮 DISTINCT Exercise 11
💡 COUNT(DISTINCT column) counts unique values — very useful in analytics!
📌
DISTINCT applies to all selected columns combined
SELECT DISTINCT city, course FROM students removes rows where BOTH city AND course are the same. A student in Chennai doing SQL and another in Chennai doing Python are NOT duplicates — they produce 2 distinct rows.

LIMIT / TOP / FETCH FIRST — Controlling Row Count

These clauses limit how many rows are returned. Essential for "Top N" queries, pagination, and preventing accidental huge result sets in production.

DatabaseSyntaxExample
MySQL
PostgreSQL
SQLite		LIMIT nSELECT * FROM students LIMIT 5;
MySQL
PostgreSQL		LIMIT n OFFSET mSELECT * FROM students LIMIT 5 OFFSET 10; (rows 11–15)
SQL Server
MS Access		TOP nSELECT TOP 5 * FROM students;
SQL Standard
Oracle
PostgreSQL		FETCH FIRST n ROWS ONLYSELECT * FROM students FETCH FIRST 5 ROWS ONLY;
🇮🇳 Common Use Case — Pagination
Implementing page-by-page results (like Flipkart product pages)
Page 1: LIMIT 20 OFFSET 0 → rows 1–20
Page 2: LIMIT 20 OFFSET 20 → rows 21–40
Page 3: LIMIT 20 OFFSET 40 → rows 41–60

Formula: OFFSET = (page_number - 1) × items_per_page
🎮 LIMIT & Pagination Exercise 12
💡 Always pair LIMIT with ORDER BY — otherwise the "top" rows are random!

Topic 12

CASE WHEN — Conditional Logic in SQL

CASE WHEN is SQL's version of an if-else statement. It lets you create new computed columns based on conditions — like categorizing scores as Pass/Fail, labeling tiers, or transforming values on the fly.

SYNTAX — Searched CASE
CASE
  WHEN condition1 THEN result1
  WHEN condition2 THEN result2
  ELSE default_result -- optional
END AS alias

How CASE WHEN evaluates — like a flowchart:

CONDITION
THEN (result)
WHEN score >= 90
→ 'Distinction'
WHEN score >= 75
→ 'First Class'
WHEN score >= 60
→ 'Second Class'
WHEN score >= 40
→ 'Pass'
ELSE (default)
→ 'Fail'
🎮 CASE WHEN Exercise 13
💡 CASE WHEN evaluates top-to-bottom — the FIRST matching condition wins. Order matters!
💡
Simple CASE — when checking one column for equality
There's also a "Simple CASE" that's useful when comparing one column to many values:
CASE course WHEN 'SQL' THEN 'Database' WHEN 'Python' THEN 'Programming' ELSE 'Other' END
This is equivalent to the "Searched CASE" but more concise for equality checks.
