Subqueries — Queries Within Queries

A subquery is a question asked inside another question. They let you use the result of one query as input to another — without ever leaving SQL or touching Python.

Scalar Subqueries Subqueries in WHERE Subqueries in FROM Correlated Subqueries EXISTS / NOT EXISTS
What Is a Subquery?

A subquery (or "inner query" or "nested query") is a SELECT statement embedded inside another SQL statement. The outer query uses the subquery's result as if it were a value, a list, or a temporary table.

Outer Query
SELECT name, salary FROM employees
WHERE salary > (
Inner Query (Subquery)
SELECT AVG(salary) FROM employees
);

The database evaluates the inner query first, gets a single number (the average salary), and then the outer query uses that number to filter rows. This entire operation runs in one round trip to the database — far more efficient than running two separate queries from your application code.

🎓 Analogy — The Recipe Within a Recipe

Imagine a recipe that says "add the sauce" — and the sauce itself is a separate recipe with its own ingredients and steps. You don't cook the sauce separately and bring it to the table; it's part of making the dish. A subquery works the same way: it's a fully formed query, computed as a step inside a bigger query, never seen on its own by the end user.

The Four Types of Subqueries

Subqueries are classified by what they return and where they're placed in the outer query.

Scalar
Returns exactly ONE value (one row, one column). Can be used anywhere a single value is expected.
WHERE salary > (SELECT AVG(salary)...)
Column / Row
Returns a single column with multiple rows (a list), or a single row with multiple columns.
WHERE city IN (SELECT city FROM...)
Table (Derived)
Returns a full result set (multiple rows AND columns) used in place of a table, typically in FROM.
FROM (SELECT ... ) AS summary
Correlated
References a column from the OUTER query — re-evaluated once for every row of the outer query.
WHERE salary > (SELECT AVG(salary) FROM e2 WHERE e2.dept = e.dept)
Where Subqueries Can Live in a Query
ClauseWhat the subquery must returnExample use case
SELECTA single value (scalar)Show each employee's salary alongside the company average
FROMA full table (rows + columns)Pre-aggregate data, then join or filter the summary
WHEREA scalar, a list, or a rowFilter rows based on a computed threshold or a list of valid IDs
HAVINGA scalar (compared to an aggregate)Filter groups whose aggregate exceeds a dynamically computed value
🎓 Teaching Tip — Subqueries Are Just Nested Questions

When a student is stuck writing a subquery, tell them to first answer the INNER question alone, run it, see the result, and only then wrap it inside the OUTER query. Building bottom-up instead of top-down removes 90% of subquery confusion. "What is the average salary?" → run that first. Then: "Now show me everyone above that number."

🔢 Scalar Subqueries

Scalar Subqueries

A subquery that returns exactly one value. The simplest, most common, and safest type to start with.

Example 1 — Scalar Subquery in WHERE: Above-Average Filter
The classic "above average" patternEx 1
-- Step 1: run the inner query alone first — see what it returns
SELECT AVG(salary) FROM employees;
-- Returns: 70750.00 (a single value)

-- Step 2: wrap it in the outer query's WHERE clause
SELECT name, salary
FROM   employees
WHERE  salary > (SELECT AVG(salary) FROM employees)
ORDER BY salary DESC;

-- Same idea: find employees earning more than the HIGHEST salary in HR
SELECT name, salary
FROM   employees
WHERE  salary > (
    SELECT MAX(salary) FROM employees e
    JOIN   departments d ON e.dept_id = d.dept_id
    WHERE  d.dept_name = 'HR'
);
Output — employees above the company average (70,750)
name | salary -----------+---------- Vikram I | 95000.00 Priya N | 92000.00 Ananya K | 85000.00 Divya R | 78000.00
Key rule for scalar subqueries: if the inner query accidentally returns more than one row, the outer query will throw an error: "Subquery returns more than 1 row." Always make sure your scalar subquery uses an aggregate function (AVG, MAX, MIN, SUM, COUNT) or a WHERE clause guaranteed to match exactly one row.
Example 2 — Scalar Subquery in SELECT: Show Value + Benchmark Together
Putting a subquery directly in the SELECT listEx 2
-- Show each employee's salary alongside the company-wide average
-- AND the difference between their salary and the average
SELECT
    name,
    salary,
    (SELECT ROUND(AVG(salary), 0) FROM employees) AS company_avg,
    salary - (SELECT AVG(salary) FROM employees)     AS diff_from_avg
FROM   employees
ORDER BY diff_from_avg DESC;

-- Scalar subquery referencing the department budget
SELECT
    e.name,
    e.salary,
    d.dept_name,
    (SELECT budget FROM departments dd WHERE dd.dept_id = e.dept_id) AS dept_budget
FROM   employees e
JOIN   departments d ON e.dept_id = d.dept_id;
-- Note: this could also be a plain JOIN — shown here to demonstrate
-- that subqueries and joins can often achieve the same result
💡 When to Choose a Subquery vs a JOIN

If you only need ONE value from another table tied to each row, a scalar subquery in SELECT can be simpler to read than a JOIN. But if it's evaluated once per outer row (as in the second example, which is technically correlated), it can be slower on large tables than the equivalent JOIN. We'll cover this performance trade-off explicitly in the Real World section.

🔍 WHERE Subqueries

Subqueries in WHERE

The most common placement. Use IN, NOT IN, comparison operators, or ANY/ALL to filter rows based on another query's result.

Example 3 — IN and NOT IN with Subqueries
Filtering against a list returned by a subqueryEx 3
-- Find employees who work in departments with budget > 150000
SELECT name, dept_id, salary
FROM   employees
WHERE  dept_id IN (
    SELECT dept_id FROM departments WHERE budget > 150000
);

-- Find employees who have NEVER placed an order (NOT IN)
SELECT name, city
FROM   employees
WHERE  emp_id NOT IN (
    SELECT emp_id FROM orders WHERE emp_id IS NOT NULL
);

-- Find products that appear in orders above ₹20,000
SELECT DISTINCT product
FROM   orders
WHERE  product IN (
    SELECT product FROM orders WHERE amount > 20000
);
Output — employees in well-funded departments
name | dept_id | salary ----------+---------+---------- Ananya K | 1 | 85000.00 Priya N | 1 | 92000.00 Divya R | 1 | 78000.00 Vikram I | 1 | 95000.00 Rajan M | 2 | 52000.00 Arjun D | 2 | 61000.00
⚠️ The NOT IN + NULL Trap

If the subquery's column contains even ONE NULL value, NOT IN will return ZERO rows — silently! This is because x NOT IN (1, 2, NULL) evaluates to UNKNOWN, not TRUE, for every value of x. Always filter out NULLs in the subquery (as shown above with WHERE emp_id IS NOT NULL) or use NOT EXISTS instead (covered in Example 8) — it doesn't have this problem.

Example 4 — ANY, ALL, and SOME with Subqueries
Comparison operators against a set of valuesEx 4
-- ANY: TRUE if the comparison holds for AT LEAST ONE value
-- "Find employees who earn more than ANY (at least one) HR employee"
SELECT name, salary
FROM   employees
WHERE  salary > ANY (
    SELECT salary FROM employees e
    JOIN   departments d ON e.dept_id = d.dept_id
    WHERE  d.dept_name = 'HR'
);
-- Equivalent to: salary > MIN(...) — true if greater than the smallest value in the set

-- ALL: TRUE only if the comparison holds for EVERY value
-- "Find employees who earn more than ALL Marketing employees"
SELECT name, salary
FROM   employees
WHERE  salary > ALL (
    SELECT salary FROM employees e
    JOIN   departments d ON e.dept_id = d.dept_id
    WHERE  d.dept_name = 'Marketing'
);
-- Equivalent to: salary > MAX(...) — true if greater than the LARGEST value in the set

-- SOME is just a synonym for ANY (identical behaviour)
SELECT name FROM employees
WHERE  salary > SOME (SELECT salary FROM employees WHERE dept_id = 4);
ANY and ALL are rarely used in practice because they're easy to confuse and most people just write the equivalent MAX/MIN scalar subquery directly — it's clearer. salary > ALL(...) is identical to salary > (SELECT MAX(...)), and the MAX version is what most working SQL developers actually write. Still, recognise ANY/ALL when you see them in legacy code or interview questions.
🗂️ FROM Subqueries

Subqueries in FROM (Derived Tables)

Treat a subquery's result as if it were a real table — query the output of a query.

Example 5 — Derived Table: Query a Pre-Aggregated Result
A subquery in FROM must always have an aliasEx 5
-- Step 1: build the inner summary query — department-level stats
SELECT
    dept_id,
    COUNT(*)    AS headcount,
    AVG(salary) AS avg_salary
FROM   employees
GROUP BY dept_id;

-- Step 2: wrap that as a derived table and filter/join on it
-- Find departments where avg_salary exceeds 65000, joined to dept names
SELECT
    d.dept_name,
    dept_stats.headcount,
    dept_stats.avg_salary
FROM (
    SELECT
        dept_id,
        COUNT(*)    AS headcount,
        AVG(salary) AS avg_salary
    FROM   employees
    GROUP BY dept_id
) AS dept_stats                          -- ← MUST alias the derived table
JOIN   departments d ON d.dept_id = dept_stats.dept_id
WHERE  dept_stats.avg_salary > 65000
ORDER BY dept_stats.avg_salary DESC;

-- Another use: ranking within a derived table (pre-window-function era trick)
SELECT *
FROM (
    SELECT name, salary,
           (SELECT COUNT(*) FROM employees e2
            WHERE e2.salary > e1.salary) + 1 AS salary_rank
    FROM employees e1
) AS ranked
WHERE salary_rank <= 3
ORDER BY salary_rank;
-- This finds the top 3 highest earners — though Chapter 12's
-- window functions (RANK, DENSE_RANK) do this far more cleanly!
Derived tables MUST have an alias (AS dept_stats above) — MySQL will throw a syntax error otherwise. Think of the derived table exactly like a regular table: you can JOIN to it, filter it with WHERE, and reference its columns with dot notation. This pattern is extremely common for "pre-aggregate then filter/join" logic that would otherwise need a temporary table.
🎓 Foreshadowing CTEs

Show students that the ranking subquery above is clunky and hard to read. Tell them: "There's a cleaner way to write nested logic like this — it's called a CTE (Common Table Expression), and we'll learn it in the very next chapter, along with window functions that make the ranking trick above completely unnecessary." This builds anticipation and shows subqueries' real-world limitations honestly.

🔁 Correlated Subqueries

Correlated Subqueries

The inner query references the outer query's current row — re-running once per row, like a loop inside SQL.

Example 6 — Correlated Subquery: Compare Each Row to Its Own Group
The subquery uses the outer row's value — that's what makes it "correlated"Ex 6
-- Find employees earning more than THEIR OWN department's average
-- (not the company-wide average — each employee compared to their peers)
SELECT
    e1.name,
    e1.dept_id,
    e1.salary,
    (SELECT ROUND(AVG(e2.salary), 0)
     FROM   employees e2
     WHERE  e2.dept_id = e1.dept_id)   AS dept_avg   -- ← references e1 (outer)!
FROM   employees e1
WHERE  e1.salary > (
    SELECT AVG(e2.salary)
    FROM   employees e2
    WHERE  e2.dept_id = e1.dept_id          -- ← correlation: links inner to outer
);
Output — only employees beating their OWN department's average
name | dept_id | salary | dept_avg ----------+---------+----------+---------- Vikram I | 1 | 95000.00 | 87500 Priya N | 1 | 92000.00 | 87500 Arjun D | 2 | 61000.00 | 56500 ← Ananya/Divya (dept 1) are below 87500 avg, so excluded. Meera has no peers in dept 3, trivially "average".
How it executes: for EVERY row in the outer query (e1), the inner query (e2) re-runs, filtered to just that row's department. With 8 employees, the inner query runs up to 8 times. This is fundamentally different from a regular (non-correlated) subquery, which runs exactly ONCE regardless of how many outer rows there are.
Example 7 — Correlated Subquery: Most Recent Order Per Employee
A very common real-world correlated subquery patternEx 7
-- For each employee, find their most recent order date
SELECT
    e.name,
    (SELECT MAX(o.order_date)
     FROM   orders o
     WHERE  o.emp_id = e.emp_id)   AS last_order_date
FROM   employees e;

-- For each employee, find the AMOUNT of their highest single order
SELECT
    e.name,
    (SELECT MAX(o.amount)
     FROM   orders o
     WHERE  o.emp_id = e.emp_id)   AS biggest_order
FROM   employees e
WHERE  EXISTS (SELECT 1 FROM orders o WHERE o.emp_id = e.emp_id);
-- The WHERE EXISTS filters out employees with NO orders at all
-- (otherwise they'd show NULL for biggest_order, which is fine too —
-- depends on your reporting requirement)

-- Performance comparison: the JOIN+GROUP BY equivalent (usually faster!)
SELECT
    e.name,
    MAX(o.amount) AS biggest_order
FROM      employees e
INNER JOIN orders o ON o.emp_id = e.emp_id
GROUP BY e.emp_id, e.name;
-- Same logical result as the correlated subquery above —
-- but the database can usually optimise a JOIN better than
-- a correlated subquery, especially on large tables
⚠️ Performance Warning — Correlated Subqueries Can Be Slow

Because a correlated subquery re-executes once per outer row, a table with 1 million rows could trigger 1 million inner query executions — extremely slow without proper indexing. Whenever a correlated subquery can be rewritten as a JOIN + GROUP BY (as shown above), prefer the JOIN version for production code on large tables. Use correlated subqueries for clarity on small datasets or when no JOIN-based rewrite is possible (some EXISTS patterns).

✅ EXISTS

EXISTS & NOT EXISTS

EXISTS only checks whether ANY row matches — it doesn't care about values, just presence. Often faster and safer than IN.

Example 8 — EXISTS and NOT EXISTS Basics
EXISTS returns TRUE/FALSE — it never looks at actual column valuesEx 8
-- Find employees who HAVE placed at least one order
SELECT e.name
FROM   employees e
WHERE  EXISTS (
    SELECT 1 FROM orders o WHERE o.emp_id = e.emp_id
);
-- Note: "SELECT 1" is a convention — EXISTS doesn't care what
-- columns you select inside it, only WHETHER any row is returned

-- Find employees who have NEVER placed an order (the SAFE version of NOT IN)
SELECT e.name
FROM   employees e
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.emp_id = e.emp_id
);
-- Unlike NOT IN, this is completely unaffected by NULLs in orders.emp_id!

-- Find departments that have at least one employee earning over 90000
SELECT d.dept_name
FROM   departments d
WHERE  EXISTS (
    SELECT 1 FROM employees e
    WHERE  e.dept_id = d.dept_id AND e.salary > 90000
);
Example 9 — IN vs EXISTS: Side-by-Side
📋 IN — compares against a LIST of values
• Subquery returns a column of values
• Outer value is checked against that list
• ⚠️ Breaks with NULLs in NOT IN
• Best for: small, static-ish lists

WHERE dept_id IN (SELECT dept_id FROM departments WHERE budget > 100000)
✅ EXISTS — checks for ROW PRESENCE
• Subquery just needs to find ANY matching row
• Stops searching as soon as one match is found ("short-circuit")
• ✅ Immune to the NULL trap
• Best for: correlated existence checks, large tables

WHERE EXISTS (SELECT 1 FROM departments d WHERE d.dept_id = e.dept_id AND d.budget > 100000)
Both queries below return the IDENTICAL resultEx 9
-- Version A: using IN
SELECT name FROM employees
WHERE  dept_id IN (SELECT dept_id FROM departments WHERE budget > 150000);

-- Version B: using EXISTS (correlated)
SELECT name FROM employees e
WHERE EXISTS (
    SELECT 1 FROM departments d
    WHERE  d.dept_id = e.dept_id AND d.budget > 150000
);
-- For small lookup tables, performance is nearly identical.
-- For large subquery result sets, EXISTS is generally faster because
-- it stops at the FIRST match instead of building the entire list first.
✅ Practical Rule of Thumb

Use EXISTS when checking for presence/absence (does at least one matching row exist?) — especially with correlated logic and especially when NULLs might be present. Use IN when you have a clean, small, known list of values, particularly a hardcoded list like WHERE status IN ('Pending','Shipped'). When in doubt on large tables, EXISTS is the safer default.

Example 10 — Capstone: Combining Everything from This Chapter
Scalar + WHERE + FROM + correlated + EXISTS, all in one queryCapstone
-- Business question: "Show departments that have above-average headcount,
-- along with their best-paid employee, but only for departments
-- that have placed at least one order over ₹15,000"

SELECT
    d.dept_name,
    dept_summary.headcount,

    -- Scalar subquery: best-paid employee's NAME in this department
    (SELECT e2.name
     FROM   employees e2
     WHERE  e2.dept_id = d.dept_id
     ORDER BY e2.salary DESC
     LIMIT 1)                       AS top_earner,

    dept_summary.avg_salary

FROM departments d

-- FROM subquery (derived table): pre-aggregated headcount per dept
JOIN (
    SELECT dept_id, COUNT(*) AS headcount, AVG(salary) AS avg_salary
    FROM   employees
    GROUP BY dept_id
) AS dept_summary ON dept_summary.dept_id = d.dept_id

WHERE
    -- WHERE subquery: headcount above the average headcount of all departments
    dept_summary.headcount > (
        SELECT AVG(dept_count)
        FROM (
            SELECT COUNT(*) AS dept_count
            FROM employees
            GROUP BY dept_id
        ) AS all_dept_counts
    )

    AND
    -- EXISTS: this department has at least one order over 15000
    EXISTS (
        SELECT 1
        FROM   orders o
        JOIN   employees e3 ON o.emp_id = e3.emp_id
        WHERE  e3.dept_id = d.dept_id AND o.amount > 15000
    )

ORDER BY dept_summary.headcount DESC;
This query uses all four subquery types in one place: a derived table in FROM (dept_summary), a scalar correlated subquery in SELECT (top_earner), a nested scalar subquery in WHERE (average headcount), and an EXISTS check. It's intentionally complex to show students how these building blocks combine in real analytical work — and to set up the motivation for CTEs in Chapter 8, which would make this dramatically more readable.
🌍 Real World

Subqueries in Production

🛒 E-Commerce — "Customers Who Bought This Also Bought"

Recommendation engines run: find products purchased by customers who ALSO bought product X — a classic EXISTS / IN combination. "Show products never returned by any customer in the last 90 days" uses NOT EXISTS against the returns table. Inventory alerts ("show products where stock is below the reorder threshold defined per category") use scalar subqueries referencing a category_settings table.

🏦 Banking — Fraud & Compliance Queries

"Flag accounts where the latest transaction amount exceeds 3× their own historical average" is a textbook correlated scalar subquery — each account compared against its OWN average, not a global one. Compliance reports use NOT EXISTS to find accounts with no KYC document on file.

🎓 EdTech — GUVI Learner Risk Detection

"Find learners whose quiz scores are below their own batch's average" — correlated subquery, batch-by-batch. "Find batches that have NO mentor assigned yet" — NOT EXISTS against the mentor_assignments table. "Show courses where completion rate is below the average completion rate across all courses" — scalar subquery in HAVING.

📊 Data Science — Outlier Detection Without Python

Before even opening a Jupyter notebook, data scientists often run a quick SQL check: WHERE value > (SELECT AVG(value) + 3 * STDDEV(value) FROM table) to flag statistical outliers directly in the database — using a scalar subquery for the threshold. It's often faster to filter outliers in SQL before pulling data into pandas.

🎓 Teaching Strategy — The "Sentence to Subquery" Drill

Give students 5 English sentences and have them identify (a) what's the outer question, (b) what's the inner question, (c) which type of subquery fits. Example: "Show me students who scored higher than their class average." Outer: select students + scores. Inner: average score, correlated by class. This decomposition skill matters more than syntax memorisation.

❓ FAQ

Questions Students Always Ask

Q: I got the error "Subquery returns more than 1 row." What does that mean and how do I fix it?▾
You used a scalar context (like = or >) but your subquery returned multiple rows. SQL doesn't know which one to compare against.

-- ❌ ERROR: this subquery can return many salaries, not one SELECT name FROM employees WHERE salary = (SELECT salary FROM employees WHERE dept_id = 1); -- ✅ FIX 1: aggregate it down to one value SELECT name FROM employees WHERE salary = (SELECT MAX(salary) FROM employees WHERE dept_id = 1); -- ✅ FIX 2: switch to IN if you actually want to match ANY of the values SELECT name FROM employees WHERE salary IN (SELECT salary FROM employees WHERE dept_id = 1);
Q: How do I know if a subquery is correlated or not?▾
Simple test: does the inner query reference any column from a table defined in the OUTER query? If yes, it's correlated. If the inner query could run completely on its own with no reference to anything outside it, it's NOT correlated (sometimes called a "simple" or "non-correlated" subquery).

-- NOT correlated — runs fine standalone SELECT name FROM employees WHERE salary > (SELECT AVG(salary) FROM employees); -- CORRELATED — references e.dept_id from the OUTER query SELECT name FROM employees e WHERE salary > (SELECT AVG(salary) FROM employees WHERE dept_id = e.dept_id);
Tip: try copy-pasting just the inner SELECT and running it alone. If it errors out with "unknown column e.dept_id," that confirms it's correlated.
Q: Why does my derived table (FROM subquery) give a syntax error?▾
The most common cause: forgetting the alias. MySQL requires every derived table to have a name.
-- ❌ ERROR: Every derived table must have its own alias SELECT * FROM (SELECT dept_id, COUNT(*) FROM employees GROUP BY dept_id); -- ✅ FIX: add AS alias_name after the closing parenthesis SELECT * FROM (SELECT dept_id, COUNT(*) FROM employees GROUP BY dept_id) AS dept_counts;
Also double-check every opening parenthesis has a matching closing one — nested subqueries are a common source of mismatched parens.
Q: Are subqueries always slower than JOINs?▾
Not always — it depends entirely on the type and the database's query optimiser.

Non-correlated subqueries (run once) are usually just as fast as an equivalent JOIN — modern optimisers often rewrite them internally to the same execution plan.

Correlated subqueries (run once per outer row) CAN be significantly slower on large tables because the inner query re-executes repeatedly — though good indexing helps a lot, and modern MySQL/PostgreSQL optimisers can sometimes "de-correlate" them automatically.

EXISTS / NOT EXISTS are usually well-optimised even when correlated, because the database can stop as soon as one match is found rather than computing every result.

Best practice: write the version that's clearest to read first, then check the EXPLAIN plan if performance becomes a real issue on production-sized data.
Q: Can I use ORDER BY inside a subquery?▾
Generally no for subqueries used in WHERE/comparisons — ORDER BY doesn't make sense without a LIMIT, since the comparison just cares about the set of values, not their order.

However, it IS allowed (and useful) combined with LIMIT to get "the single highest/lowest value" as a scalar:
SELECT name FROM employees WHERE salary = ( SELECT salary FROM employees ORDER BY salary DESC LIMIT 1 -- guarantees exactly 1 row returned );
This is a common pattern for "find the row with the single max/min value" when you specifically need the full row, not just the aggregate.
