DML — Data Manipulation Language

DML is where data lives, moves, and changes. INSERT puts data in, UPDATE changes it, DELETE removes it — but each command carries serious responsibility. One missing WHERE clause can wipe an entire table. This chapter teaches you not just the syntax, but the safety patterns that professional developers use to never lose data.

Topic 1

DML Overview — What Changes Data

DML (Data Manipulation Language) is the set of SQL commands that read, add, change, or remove data inside tables. Unlike DDL (which changes structure), DML works with the rows inside existing tables.

➕
INSERT
Add new rows to a table
✏️
UPDATE
Modify existing row values
🗑️
DELETE
Remove rows from a table
🔁
SELECT
Read data (DQL / part of DML)
📌
DML vs DDL — The Critical Difference
DDL (CREATE/ALTER/DROP) — changes structure. Auto-committed, usually can't rollback.
DML (INSERT/UPDATE/DELETE) — changes data. Participates in transactions — you CAN rollback before committing!

This is why DML safety matters so much: you have a window to undo mistakes using ROLLBACK — but only if you use transactions.

Topic 2

INSERT INTO — Single Row

INSERT INTO adds a new row to a table. You can specify columns explicitly (recommended) or insert in the table's column order.

SYNTAX
INSERT INTO table_name (col1, col2, ...) VALUES (val1, val2, ...);
🎮 INSERT Single Row Exercise 1
Modifies the live database above
💡 Always specify column names — if a new column is added to the table, positional INSERT breaks!
💡
After INSERT — Get the new row's ID
In MySQL: SELECT LAST_INSERT_ID(); → returns the AUTO_INCREMENT value just assigned
In PostgreSQL: INSERT INTO ... RETURNING id; → returns the new row's values
In Python (mysql-connector): cursor.lastrowid
This is essential in applications — you need the new ID to create related records.

Topic 3

INSERT INTO — Multi-Row Batch

Inserting many rows one-by-one is extremely slow. A multi-row INSERT sends all rows in a single statement — up to 100× faster for large datasets. This is the standard approach for data loading.

Performance ComparisonMySQL
-- ❌ SLOW: 1000 separate INSERT statements (1000 round trips to DB) INSERT INTO students (name, city, course, score, fee_paid) VALUES ('Priya', 'Chennai', 'SQL', 88, 12000); INSERT INTO students (name, city, course, score, fee_paid) VALUES ('Rahul', 'Delhi', 'ML', 74, 15000); -- ... 998 more statements → ~2 seconds -- ✅ FAST: One multi-row INSERT (1 round trip) INSERT INTO students (name, city, course, score, fee_paid) VALUES ('Priya', 'Chennai', 'SQL', 88.0, 12000), ('Rahul', 'Delhi', 'ML', 74.0, 15000), ('Ananya', 'Bangalore', 'Python', 91.5, 10000), -- ... 997 more rows ('Karthik', 'Mumbai', 'SQL', 65.0, 12000); -- → ~0.02 seconds (100× faster!)
🎮 Multi-Row Batch INSERT Exercise 2
💡 For very large datasets (100k+ rows), consider LOAD DATA INFILE (MySQL) or COPY (PostgreSQL) — even faster!

Topic 4

INSERT INTO … SELECT — Copy Data

INSERT INTO … SELECT copies rows from one table into another in a single SQL statement. Extremely useful for ETL pipelines, archiving, creating summaries, and data migration.

INSERT INTO … SELECT — All PatternsMySQL
-- Pattern 1: Copy ALL rows from one table to another INSERT INTO students_backup SELECT * FROM students; -- Pattern 2: Copy selected columns + filter INSERT INTO top_students (name, city, course, score) SELECT name, city, course, score FROM students WHERE score >= 80; -- Pattern 3: Copy with transformation INSERT INTO student_report (student_name, grade, city) SELECT name, CASE WHEN score >= 90 THEN 'A' WHEN score >= 75 THEN 'B' ELSE 'C' END, city FROM students; -- Pattern 4: Create table AND insert in one shot CREATE TABLE sql_students AS SELECT * FROM students WHERE course = 'SQL'; -- ⚠️ This creates the table BUT does NOT copy constraints/indexes! -- Pattern 5: Archive old data (ETL pattern) INSERT INTO students_archive_2023 SELECT *, NOW() AS archived_at FROM students WHERE batch = 'B2023';
🎮 INSERT INTO SELECT Exercise 3
💡 INSERT INTO SELECT is atomic — either all rows are inserted or none. Perfect for batch ETL operations.

Topic 5

UPDATE SET WHERE — Modifying Data Safely

UPDATE modifies existing rows. The WHERE clause controls WHICH rows get updated. Without WHERE, every single row in the table is modified — one of the most common disasters in SQL.

SYNTAX
UPDATE table_name SET col1 = new_value1, col2 = new_value2 WHERE condition; ← NEVER OMIT THIS!
🎮 UPDATE — Safe Editing Exercise 4
💡 Always run SELECT first to see which rows will be affected. Then replace SELECT with UPDATE.
💀 Common Disaster — UPDATE Without WHERE
😱 True story scenario: Developer wants to update Priya's city from Chennai to Bangalore.

They type: UPDATE students SET city = 'Bangalore';

They forgot the WHERE clause. Result? ALL 50,000 students now live in Bangalore.
If not in a transaction, this is permanent. Hours of panic follow.
✅ The correct, safe query:
UPDATE students SET city = 'Bangalore' WHERE student_id = 1;

✅ Even safer — run this SELECT first to confirm:
SELECT student_id, name, city FROM students WHERE student_id = 1;
— verify you see exactly the row you want to update, THEN run the UPDATE.
🗑️
Topic 6

DELETE FROM WHERE — Removing Rows

DELETE FROM removes rows that satisfy the WHERE condition. Like UPDATE, a missing WHERE clause deletes ALL rows.

SYNTAX
DELETE FROM table_name WHERE condition; ← CRITICAL — always include!
🎮 DELETE — Removing Rows Exercise 5
💡 Use LIMIT 1 in DELETE when targeting a single row by non-unique criteria — prevents accidental bulk deletion!
⚠️
DELETE vs TRUNCATE — Choose carefully
DELETE FROM students; — removes all rows, but: logs every row, fires triggers, can ROLLBACK. Slow on big tables.
TRUNCATE TABLE students; — removes all rows, but: NOT logged row-by-row, NO rollback in MySQL, NO triggers. Lightning fast.

For targeted removal: DELETE with WHERE. For complete table wipe: TRUNCATE. For reverting: always use DELETE inside a transaction.

Topic 7

REPLACE INTO & UPSERT

Sometimes you want to: "Insert this row, but if it already exists, update it instead." This is called an UPSERT (Update + Insert). Different databases handle this differently.

UPSERT Patterns — All DatabasesMySQL / PostgreSQL
── MySQL: REPLACE INTO ──────────────────────────────────────── -- REPLACE = DELETE existing row if PK/UNIQUE conflict, then INSERT fresh -- ⚠️ Deletes and re-inserts — AUTO_INCREMENT ID changes! REPLACE INTO students (student_id, name, city, course, score) VALUES (1, 'Priya Sharma', 'Bangalore', 'SQL', 92.0); ── MySQL: INSERT … ON DUPLICATE KEY UPDATE (preferred!) ────── -- Safer: Only UPDATEs the conflicting columns, keeps the row ID INSERT INTO students (student_id, name, city, score) VALUES (1, 'Priya Sharma', 'Bangalore', 92.0) ON DUPLICATE KEY UPDATE city = VALUES(city), score = VALUES(score); -- If student_id=1 exists: updates city & score only -- If student_id=1 doesn't exist: inserts the full row -- Increment counter on conflict (common in analytics): INSERT INTO page_views (page_id, views) VALUES (101, 1) ON DUPLICATE KEY UPDATE views = views + 1; ── PostgreSQL: INSERT … ON CONFLICT (modern SQL standard) ──── INSERT INTO students (student_id, name, city, score) VALUES (1, 'Priya Sharma', 'Bangalore', 92.0) ON CONFLICT (student_id) DO UPDATE SET city = EXCLUDED.city, score = EXCLUDED.score; -- PostgreSQL: Do nothing on conflict (skip duplicates) INSERT INTO students (student_id, name) VALUES (1, 'Priya') ON CONFLICT (student_id) DO NOTHING;
💡
REPLACE INTO vs ON DUPLICATE KEY UPDATE
REPLACE INTO: Deletes the existing row, then inserts fresh. AUTO_INCREMENT ID changes. Any FK references to the old row break. Use rarely.

ON DUPLICATE KEY UPDATE: Updates only the specified columns. Row ID stays the same. FK references intact. This is almost always what you want.

In production code, use ON DUPLICATE KEY UPDATE — it's safer and more predictable.
⚠️
Topic 8

WHERE Clause Discipline — The #1 Safety Rule

The single most important habit in DML: never run UPDATE or DELETE without a WHERE clause — unless you explicitly intend to affect every row. Here are the rules professional developers follow.

💀 Real Disasters — The Hall of Shame
🚨 GitLab (2017): A DBA meant to delete data from a staging database. He deleted the wrong database — on production. 300 GB of data lost in seconds. They had to restore from a 6-hour-old backup. 5000+ GitLab.com users lost data.
The culprit: Running a command on the wrong server, no WHERE clause.
Source: GitLab's public post-mortem
🚨 Every beginner at some point: UPDATE students SET city = 'Mumbai'; (missing WHERE student_id = 5)
Result: All 20,000 students now live in Mumbai.
✅ Prevention rules:
1. Always write WHERE clause BEFORE the SET clause in your mind
2. Never copy-paste an UPDATE/DELETE from one context to another
3. Always use transactions for any DML in production
4. Add MySQL's safe mode: SET SQL_SAFE_UPDATES = 1; (blocks UPDATE/DELETE without WHERE + key column)
Safe Update Mode & Best PracticesMySQL
-- Enable safe update mode (MySQL) — prevents dangerous queries SET SQL_SAFE_UPDATES = 1; -- This will now FAIL (no WHERE with key column): UPDATE students SET city = 'Mumbai'; -- Error 1175: You are using safe update mode... -- This will SUCCEED (WHERE with key column): UPDATE students SET city = 'Mumbai' WHERE student_id = 5; -- key column in WHERE = safe -- Best practices checklist: -- ✅ WHERE clause always present -- ✅ Use PK in WHERE when updating specific row -- ✅ Run SELECT first with same WHERE to verify -- ✅ Wrap in BEGIN/COMMIT transaction -- ✅ Use LIMIT on UPDATE/DELETE for safety net -- ✅ Backup or test on staging before production

Topic 9

The Dry-Run Pattern — SELECT Before You Wreck

The dry-run pattern is a professional habit: before running any UPDATE or DELETE, first run a SELECT with the same WHERE clause to preview exactly which rows will be affected. Only then execute the modification.

1️⃣
Write the SELECT (dry run) PREVIEW
First, write a SELECT with the exact same WHERE condition you plan to use in your UPDATE or DELETE. This shows you PRECISELY which rows will be affected.
-- STEP 1: See which rows match your condition SELECT student_id, name, score, city FROM students WHERE course = 'Python' AND score < 50;
2️⃣
Verify the results CHECK
Look at the SELECT results carefully. Ask: "Is this exactly what I want to update/delete?" Count the rows. Check the values. If anything looks wrong — fix the WHERE clause and re-run the SELECT.
Checklist: ✅ Right number of rows? ✅ Right students? ✅ Values look correct? ✅ No unexpected rows included?
3️⃣
Replace SELECT with UPDATE/DELETE EXECUTE
Only after verifying, change SELECT to UPDATE/DELETE — keeping the exact same WHERE clause.
-- STEP 3: Same WHERE, now with UPDATE UPDATE students SET is_active = FALSE WHERE course = 'Python' AND score < 50;
4️⃣
Verify after the operation CONFIRM
Run a SELECT again after the UPDATE/DELETE to confirm the changes took effect as expected. Check rows affected count. Verify before committing in production.
-- STEP 4: Verify the change SELECT student_id, name, is_active FROM students WHERE course = 'Python' AND score < 50;

💡 This pattern saves databases. Senior developers do this every time, without exception.

Topic 10

Soft Delete vs Hard Delete

In real applications, you almost never truly delete data. Instead, you use soft delete — marking a record as "deleted" with a flag, while keeping the data. This is standard practice in production systems.

AspectHard Delete (DELETE FROM)Soft Delete (UPDATE is_deleted)
Data afterGone permanentlyStill in database, just flagged
Recovery❌ Only from backup✅ Just flip the flag back
Audit trail❌ No history✅ Who deleted, when, why
FK references❌ Can break child tables✅ FK still valid
Compliance❌ May violate data retention laws✅ Data preserved for auditors
Query complexitySimple — just SELECTMust add WHERE is_deleted = FALSE
StorageLess data storedMore data stored
Used inTemp data, logs, test dataUsers, orders, financial records
Implementing Soft Delete — Complete PatternMySQL
-- Step 1: Add soft delete columns to table ALTER TABLE students ADD COLUMN is_deleted BOOLEAN DEFAULT FALSE, ADD COLUMN deleted_at TIMESTAMP NULL, ADD COLUMN deleted_reason VARCHAR(200); -- Step 2: "Delete" = UPDATE the flag (not DELETE) UPDATE students SET is_deleted = TRUE, deleted_at = NOW(), deleted_reason = 'Student requested account removal' WHERE student_id = 5; -- Step 3: All regular queries must filter out deleted records SELECT * FROM students WHERE is_deleted = FALSE; -- add this to EVERY query -- Step 4: "Restore" a deleted record easily UPDATE students SET is_deleted = FALSE, deleted_at = NULL, deleted_reason = NULL WHERE student_id = 5; -- Step 5: Admin view — see all deleted records SELECT student_id, name, deleted_at, deleted_reason FROM students WHERE is_deleted = TRUE ORDER BY deleted_at DESC;

💡 Notice: the student's data still exists — we can restore it anytime by setting is_active = TRUE again!

Topic 11

Cascading Deletes with FK Constraints

When tables are linked by Foreign Keys with ON DELETE CASCADE, deleting a parent row automatically deletes all child rows across multiple tables — like a chain reaction. Understanding this is critical for both intentional design and accidental disasters.

🇮🇳 Real-World Cascade Scenario
GUVI LMS — What happens when a student account is deleted?
Table: students (parent)
↓ ON DELETE CASCADE →
Table: enrollments (child) — student's course enrollments deleted
↓ ON DELETE CASCADE →
Table: quiz_attempts (grandchild) — their quiz results deleted
↓ ON DELETE CASCADE →
Table: certificates (great-grandchild) — issued certificates deleted

One DELETE FROM students WHERE student_id = 5 can cascade through 4 tables and delete hundreds of rows. All automatic. This is why most production systems use soft delete instead!
Cascade Delete — Schema & BehaviorMySQL
-- Schema with cascading FK relationships CREATE TABLE students ( student_id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(100) ); CREATE TABLE enrollments ( enroll_id INT PRIMARY KEY AUTO_INCREMENT, student_id INT NOT NULL, course VARCHAR(50), FOREIGN KEY (student_id) REFERENCES students(student_id) ON DELETE CASCADE -- enrollments deleted when student deleted ); CREATE TABLE quiz_attempts ( attempt_id INT PRIMARY KEY AUTO_INCREMENT, enroll_id INT NOT NULL, score DECIMAL(5,2), FOREIGN KEY (enroll_id) REFERENCES enrollments(enroll_id) ON DELETE CASCADE -- quiz attempts deleted when enrollment deleted ); -- ONE delete cascades through 3 tables: DELETE FROM students WHERE student_id = 5; -- AUTO-DELETES: all enrollments WHERE student_id=5 -- all quiz_attempts WHERE enroll_id IN (deleted enrollments) -- To see cascade impact BEFORE deleting (dry run): SELECT s.name, COUNT(DISTINCT e.enroll_id) AS enrollments_to_delete, COUNT(DISTINCT q.attempt_id) AS quiz_attempts_to_delete FROM students s LEFT JOIN enrollments e ON s.student_id = e.student_id LEFT JOIN quiz_attempts q ON e.enroll_id = q.enroll_id WHERE s.student_id = 5 GROUP BY s.name;
💀
Cascade Delete is powerful — and dangerous
Always run a dry-run SELECT JOIN query to understand cascade impact before deleting parent rows. A single DELETE can remove thousands of rows across multiple tables in milliseconds — and if not in a transaction, it's gone.

Production best practice: Use ON DELETE RESTRICT (blocks delete, forces manual cleanup), or use soft delete on parent records.

Topic 12

Transactions for Bulk DML — Your Safety Net

A transaction is a group of DML statements that execute as a single unit — all succeed together, or none of them apply. Transactions are your ultimate safety net: if anything goes wrong, you can ROLLBACK and restore the exact previous state.

🔄 Transaction Lifecycle
🚀
BEGIN / START TRANSACTION
Start a transaction — all subsequent DML is temporary, not yet committed
✏️
INSERT / UPDATE / DELETE
Run DML operations — changes visible to THIS session but not yet to others
🔍
SELECT to verify
Within the transaction, run SELECT to verify changes look correct before committing
✅
COMMIT → changes permanent
All changes are written to disk and visible to all other connections
↩️
OR: ROLLBACK → undo everything
If anything went wrong, ALL changes in this transaction are reversed instantly
🎮 Transactions — Bulk DML with Safety Net Exercise 8
💡 In production: ALWAYS wrap bulk DML in transactions. One mistake shouldn't corrupt your entire database.
✅ Safe Bulk UPDATE Pattern
Wraps all changes in a transaction with verification at each step.
START TRANSACTION; -- Dry run first SELECT COUNT(*) FROM students WHERE score < 40; -- Execute UPDATE students SET is_active = FALSE WHERE score < 40; -- Verify SELECT student_id, name, is_active FROM students WHERE score < 40; -- Only if all looks good: COMMIT; -- Otherwise: ROLLBACK;
🏦 Bank Transfer Pattern
Classic ACID example — atomic debit + credit pair.
START TRANSACTION; -- Debit sender UPDATE accounts SET balance = balance - 5000 WHERE account_id = 101 AND balance >= 5000; -- check funds! -- Verify debit happened IF ROW_COUNT() = 0 THEN ROLLBACK; -- insufficient funds END IF; -- Credit receiver UPDATE accounts SET balance = balance + 5000 WHERE account_id = 202; COMMIT; -- both happen or neither
📦 ETL Load Pattern
For loading batch data — all rows or none.
START TRANSACTION; -- Clear staging TRUNCATE TABLE staging_students; -- Load new data INSERT INTO staging_students SELECT * FROM csv_import; -- Validate counts SELECT COUNT(*) FROM staging_students; -- If count matches expected: proceed -- Move to production INSERT INTO students SELECT * FROM staging_students; COMMIT; -- or ROLLBACK if counts wrong

DML Cheatsheet & Quiz

📋 DML Quick Reference
INSERT
-- Single row INSERT INTO t (col1,col2) VALUES (v1,v2); -- Multi-row batch INSERT INTO t (col1) VALUES (a),(b),(c); -- From SELECT INSERT INTO t2 SELECT * FROM t1 WHERE ...;
UPDATE (safe)
-- DRY RUN FIRST: SELECT * FROM t WHERE condition; -- THEN update: UPDATE t SET col = new_val WHERE condition; ← NEVER OMIT
DELETE (safe)
-- DRY RUN FIRST: SELECT * FROM t WHERE condition; -- THEN delete: DELETE FROM t WHERE condition LIMIT n; ← safety net
UPSERT (MySQL)
INSERT INTO t (id, col) VALUES (1, 'new') ON DUPLICATE KEY UPDATE col = VALUES(col);
Transactions
START TRANSACTION; -- DML statements... SELECT * ... -- verify COMMIT; -- save -- OR: ROLLBACK; -- undo all
Soft Delete Pattern
-- Mark deleted (don't actually delete) UPDATE t SET is_deleted=TRUE, deleted_at=NOW() WHERE id=5; -- Query active only: SELECT * FROM t WHERE is_deleted=FALSE;
