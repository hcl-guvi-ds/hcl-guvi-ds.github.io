What's an Analytics Pattern?

A pattern is a repeatable SQL structure that solves a whole class of business questions. Once you recognise the pattern, you can apply it to any dataset.

🧲 Cohort Analysis Using Window Functions

A cohort is a group of users who share the same start event — usually their first activity month. Cohort analysis answers: "of the users who started in January, what % were still active in February? March?"

🎯 Why Cohort Analysis Matters

If you just look at total active users each month, a growing user base can hide terrible retention. Cohort analysis separates "new users keeping the number up" from "old users still coming back."

🏫
GUVI example:Cohort = learners who enrolled in the same month. Retention = did they log in and complete at least one lesson in each subsequent month?
🔨 Step 1 — Find Each User's Cohort Month
SQL
-- Step 1: assign each learner their cohort (first-ever enrollment month)
WITH cohorts AS (
  SELECT
    student_id,
    DATE_FORMAT(MIN(enrolled_on), '%Y-%m') AS cohort_month
  FROM   guvi_learners
  GROUP BY student_id
),

-- Step 2: join every activity to the cohort month, compute "months since start"
activity AS (
  SELECT
    a.student_id,
    c.cohort_month,
    DATE_FORMAT(a.activity_date, '%Y-%m') AS activity_month,
    TIMESTAMPDIFF(MONTH,
      STR_TO_DATE(CONCAT(c.cohort_month,'-01'),'%Y-%m-%d'),
      STR_TO_DATE(CONCAT(DATE_FORMAT(a.activity_date,'%Y-%m'),'-01'),'%Y-%m-%d')
    ) AS month_number
  FROM   user_activity a
  JOIN   cohorts c ON a.student_id = c.student_id
),

-- Step 3: count distinct users retained per cohort × month_number
retained AS (
  SELECT
    cohort_month, month_number,
    COUNT(DISTINCT student_id) AS retained_users
  FROM   activity
  GROUP BY cohort_month, month_number
),

cohort_sizes AS (
  SELECT cohort_month, retained_users AS cohort_size
  FROM   retained
  WHERE  month_number = 0
)

SELECT
  r.cohort_month,
  r.month_number,
  r.retained_users,
  cs.cohort_size,
  ROUND(100.0 * r.retained_users / cs.cohort_size, 1) AS retention_pct
FROM   retained r
JOIN   cohort_sizes cs ON r.cohort_month = cs.cohort_month
ORDER BY r.cohort_month, r.month_number;

🔻 Funnel Analysis — Stage Drop-off Rates

A funnel tracks how many users complete each step of a process — and crucially, where they fall off. Every product has a funnel: registration → course start → first quiz → certificate.

🏫 GUVI Learner Activation Funnel
SQL
SELECT
  COUNT(DISTINCT student_id)                                              AS registered,
  COUNT(DISTINCT CASE WHEN first_login  IS NOT NULL THEN student_id END)   AS activated,
  COUNT(DISTINCT CASE WHEN first_lesson IS NOT NULL THEN student_id END)   AS started_learning,
  COUNT(DISTINCT CASE WHEN quiz_score   IS NOT NULL THEN student_id END)   AS attempted_quiz,
  COUNT(DISTINCT CASE WHEN certificate  = 'Y'         THEN student_id END)   AS certified
FROM   guvi_learners;
💡
The biggest drop-off in this funnel is between Registered → Activated (only 73% actually log in). This is the #1 improvement opportunity — investigate the onboarding email sequence.
📅 Funnel with Time Constraint — "Within 7 Days"
SQL
-- Ordered funnel: each step must happen AFTER the previous, within 7 days
WITH funnel_steps AS (
  SELECT
    student_id,
    enrolled_on,
    MIN(CASE WHEN event_type = 'login'       THEN event_time END) AS t_login,
    MIN(CASE WHEN event_type = 'lesson_open' THEN event_time END) AS t_lesson,
    MIN(CASE WHEN event_type = 'quiz_submit' THEN event_time END) AS t_quiz
  FROM   guvi_events
  GROUP BY student_id, enrolled_on
)
SELECT
  COUNT(*)                                                         AS step1_enrolled,
  COUNT(CASE WHEN t_login  <= enrolled_on + INTERVAL 7 DAY THEN 1 END) AS step2_login,
  COUNT(CASE WHEN t_lesson <= enrolled_on + INTERVAL 7 DAY THEN 1 END) AS step3_lesson,
  COUNT(CASE WHEN t_quiz   <= enrolled_on + INTERVAL 7 DAY THEN 1 END) AS step4_quiz
FROM   funnel_steps;

🔗 Sessionisation — Grouping User Events

Sessionisation groups raw event logs into "sessions" — bursts of user activity separated by idle gaps. A session typically ends when no activity occurs for 30 minutes.

🔨 Sessionisation with Window Functions
SQL
WITH events_with_prev AS (
  SELECT
    student_id, event_time, event_type,
    LAG(event_time) OVER (PARTITION BY student_id ORDER BY event_time)
      AS prev_event_time
  FROM   guvi_events
),
session_flags AS (
  SELECT *,
    -- flag = 1 when gap from previous event > 30 minutes → NEW session
    CASE
      WHEN prev_event_time IS NULL
        OR TIMESTAMPDIFF(MINUTE, prev_event_time, event_time) > 30
      THEN 1 ELSE 0
    END AS new_session_flag
  FROM   events_with_prev
)
SELECT
  student_id, event_time, event_type,
  -- running sum of new_session_flag = unique session number per student
  SUM(new_session_flag) OVER (
    PARTITION BY student_id
    ORDER BY event_time
  ) AS session_id
FROM   session_flags;
💡
The trick:When we flag the start of each session as 1 and all other events as 0, a running SUM of that flag produces a unique, incrementing session number per user. This is the standard sessionisation pattern used by Zomato, Swiggy, and every product analytics team.

📆 Year-over-Year & Quarter-over-Quarter

YoY and QoQ comparisons tell you whether a metric is growing or declining relative to the same period last year/quarter — controlling for seasonality.

📆 Year-over-Year Revenue — LAG with 12-Month Offset
SQL
WITH monthly AS (
  SELECT
    YEAR(payment_date)  AS yr,
    MONTH(payment_date) AS mo,
    SUM(amount)         AS revenue
  FROM   payments
  GROUP BY yr, mo
)
SELECT
  yr, mo, revenue,
  LAG(revenue, 12) OVER (ORDER BY yr, mo) AS same_month_last_year,
  ROUND(
    100.0 * (revenue - LAG(revenue,12) OVER(ORDER BY yr, mo))
    / NULLIF(LAG(revenue,12) OVER(ORDER BY yr, mo), 0),
  1) AS yoy_pct
FROM   monthly
ORDER BY yr, mo;
📊 Quarter-over-Quarter Enrollments
SQL
SELECT
  YEAR(enrolled_on)    AS yr,
  QUARTER(enrolled_on) AS qtr,
  COUNT(*)            AS new_learners,
  LAG(COUNT(*),1) OVER(ORDER BY YEAR(enrolled_on), QUARTER(enrolled_on)) AS prev_quarter,
  ROUND(100.0 * (COUNT(*) - LAG(COUNT(*),1) OVER(ORDER BY YEAR(enrolled_on), QUARTER(enrolled_on)))
        / NULLIF(LAG(COUNT(*),1) OVER(ORDER BY YEAR(enrolled_on), QUARTER(enrolled_on)),0),1) AS qoq_pct
FROM   guvi_learners
GROUP BY yr, qtr
ORDER BY yr, qtr;

SQL — Pivot with CASE WHEN
SELECT
  DATE_FORMAT(enrolled_on, '%b') AS month,
  SUM(CASE WHEN city = 'Chennai'   THEN 1 ELSE 0 END) AS Chennai,
  SUM(CASE WHEN city = 'Mumbai'    THEN 1 ELSE 0 END) AS Mumbai,
  SUM(CASE WHEN city = 'Bangalore' THEN 1 ELSE 0 END) AS Bangalore,
  SUM(CASE WHEN city = 'Hyderabad' THEN 1 ELSE 0 END) AS Hyderabad
FROM   guvi_learners
GROUP BY DATE_FORMAT(enrolled_on,'%b'), MONTH(enrolled_on)
ORDER BY MONTH(enrolled_on);
↩️ UNPIVOT — Reverse: Wide → Long

SQL has no native UNPIVOT in MySQL. Use UNION ALL — one SELECT per column being "unpivoted":

SQL — UNPIVOT using UNION ALL
SELECT month, 'Chennai'   AS city, Chennai   AS enrollments FROM monthly_pivot
UNION ALL
SELECT month, 'Mumbai'    AS city, Mumbai    AS enrollments FROM monthly_pivot
UNION ALL
SELECT month, 'Bangalore' AS city, Bangalore AS enrollments FROM monthly_pivot
ORDER BY month, city;

🎯 First/Last Touch Attribution

Attribution answers: "which marketing channel gets credit for a conversion?" First-touch credits the first touchpoint; last-touch credits the final one before conversion.

🏫 GUVI — Which Channel Drove Enrollment?
SQL
WITH touches AS (
  SELECT
    student_id, channel, touched_at,
    ROW_NUMBER() OVER (PARTITION BY student_id ORDER BY touched_at ASC)  AS first_touch_rank,
    ROW_NUMBER() OVER (PARTITION BY student_id ORDER BY touched_at DESC) AS last_touch_rank
  FROM   marketing_touches
  WHERE  student_id IN (SELECT student_id FROM guvi_learners)  -- converted users only
)
SELECT
  'First Touch' AS model, channel, COUNT(*) AS conversions
FROM   touches WHERE first_touch_rank = 1 GROUP BY channel
UNION ALL
SELECT
  'Last Touch' AS model, channel, COUNT(*) AS conversions
FROM   touches WHERE last_touch_rank = 1 GROUP BY channel
ORDER BY model, conversions DESC;

Instagram discovers users (best first touch), but Email is what finally converts them (best last touch). This tells marketing: keep Instagram for awareness spend, invest in email nurturing to close.

⚙️ Feature Engineering in SQL

Feature engineering is computing model-input columns from raw data. SQL is excellent at this — often faster than pandas for large datasets.

📐 RFM Features — Recency, Frequency, Monetary
SQL → ML Feature Table
SELECT
  student_id,
  -- RECENCY: days since last activity
  DATEDIFF(CURDATE(), MAX(activity_date))           AS recency_days,
  -- FREQUENCY: total sessions in last 30 days
  COUNT(DISTINCT session_id)                         AS frequency,
  -- MONETARY (for EdTech: total course value completed)
  SUM(course_value)                                  AS monetary,
  -- DERIVED FEATURES
  COUNT(DISTINCT lesson_id)                          AS lessons_completed,
  ROUND(AVG(quiz_score),1)                          AS avg_score,
  DATEDIFF(MAX(activity_date), MIN(activity_date))  AS tenure_days,
  CASE WHEN MAX(quiz_score) >= 75 THEN 1 ELSE 0 END AS ever_passed  -- binary flag
FROM   guvi_events
WHERE  activity_date >= DATE_SUB(CURDATE(), INTERVAL 30 DAY)
GROUP BY student_id;
📊 NTILE Percentile Bucketing for Classification Labels
SQL
-- Create a "at-risk" label for churn prediction model
WITH features AS (
  SELECT student_id, recency_days, frequency,
         NTILE(5) OVER(ORDER BY recency_days DESC) AS recency_tier,
         NTILE(5) OVER(ORDER BY frequency DESC)    AS freq_tier
  FROM   user_features
)
SELECT *,
  -- combined score: high recency + low frequency → churn risk
  CASE WHEN recency_tier >= 4 AND freq_tier >= 4 THEN 1 ELSE 0 END AS churn_label
FROM   features;
📡 Outlier Detection with STDDEV
SQL
-- Flag scores more than 2 standard deviations from the batch mean
WITH stats AS (
  SELECT batch_id,
         AVG(quiz_score)    AS mean_score,
         STDDEV(quiz_score) AS std_score
  FROM   guvi_learners
  GROUP BY batch_id
)
SELECT
  l.student_id, l.name, l.batch_id, l.quiz_score,
  s.mean_score, s.std_score,
  ABS(l.quiz_score - s.mean_score) / NULLIF(s.std_score,0) AS z_score,
  CASE
    WHEN ABS(l.quiz_score - s.mean_score) > 2 * s.std_score
    THEN '⚠️ Outlier'
    ELSE '✅ Normal'
  END AS outlier_flag
FROM   guvi_learners l
JOIN   stats s ON l.batch_id = s.batch_id
ORDER BY z_score DESC;
🎲 Sampling Strategies — TABLESAMPLE & RAND()
SQL
-- RAND() sampling — works in MySQL (no TABLESAMPLE support)
-- 10% random sample for quick model prototyping
SELECT * FROM guvi_learners
WHERE  RAND() < 0.10;

-- Reproducible sample with a SEED (same rows every run)
SELECT * FROM guvi_learners
WHERE  (RAND(42) * 1000) < 100        -- ~10% with seed 42
LIMIT  1000;

-- PostgreSQL TABLESAMPLE (native)
SELECT * FROM guvi_learners
TABLESAMPLE BERNOULLI(10);            -- 10% of rows, PostgreSQL only

📈 Time-Series Gap Filling

Reports that GROUP BY date skip days with no activity. Gap filling ensures every date appears in the result — crucial for charts that should show flat lines, not missing data points.

🔨 Generate a Date Spine + LEFT JOIN
SQL
-- Step 1: generate every date in range (recursive CTE from Ch 16)
WITH RECURSIVE date_spine AS (
  SELECT CAST('2024-01-01' AS DATE) AS d
  UNION ALL
  SELECT DATE_ADD(d, INTERVAL 1 DAY) FROM date_spine WHERE d < '2024-03-31'
),

-- Step 2: aggregate actual data by date
daily_enrollments AS (
  SELECT DATE(enrolled_on) AS enrollment_date, COUNT(*) AS new_learners
  FROM   guvi_learners
  GROUP BY enrollment_date
)

-- Step 3: LEFT JOIN spine to data — gaps show as 0, not missing
SELECT
  ds.d                                            AS date,
  COALESCE(de.new_learners, 0)                    AS new_learners,
  SUM(COALESCE(de.new_learners,0)) OVER(ORDER BY ds.d) AS running_total
FROM   date_spine ds
LEFT JOIN daily_enrollments de ON ds.d = de.enrollment_date
ORDER BY ds.d;
✅
COALESCE(de.new_learners, 0) replaces NULL (no enrolments that day) with 0 — so your matplotlib chart draws a flat zero line rather than a confusing gap or broken line.

import mysql.connector
import pandas as pd
import matplotlib.pyplot as plt
import matplotlib.gridspec as gridspec

# ── 1. CONNECT ──────────────────────────────────────────
conn = mysql.connector.connect(
    host='localhost', database='guvi_db',
    user='analyst', password='****'
)

# ── 2. EXTRACT: RFM feature table ───────────────────────
rfm_sql = """
    SELECT student_id,
           DATEDIFF(CURDATE(), MAX(activity_date)) AS recency,
           COUNT(DISTINCT session_id)              AS frequency,
           SUM(course_value)                       AS monetary,
           AVG(quiz_score)                         AS avg_score,
           CASE WHEN MAX(quiz_score)>=75 THEN 1 ELSE 0 END AS passed
    FROM   guvi_events
    GROUP BY student_id
"""
df = pd.read_sql(rfm_sql, conn)

# ── 3. TRANSFORM in pandas ──────────────────────────────
df['recency_tier'] = pd.qcut(df['recency'], q=5, labels=[5,4,3,2,1])
df['freq_tier']    = pd.qcut(df['frequency'], q=5, labels=[1,2,3,4,5])
df['rfm_score']   = (df['recency_tier'].astype(int)
                    + df['freq_tier'].astype(int))

# ── 4. ML: simple churn prediction ──────────────────────
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import train_test_split

X = df[['recency','frequency','monetary','avg_score']]
y = (df['recency'] > 30).astype(int)   # 1 = inactive > 30 days

X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.2)
model = RandomForestClassifier(n_estimators=100, random_state=42)
model.fit(X_train, y_train)
df['churn_prob'] = model.predict_proba(X)[:,1]

# ── 5. DASHBOARD ────────────────────────────────────────
fig = plt.figure(figsize=(15, 10))
gs  = gridspec.GridSpec(2, 3, figure=fig)

ax1 = fig.add_subplot(gs[0, 0])    # RFM score distribution
ax2 = fig.add_subplot(gs[0, 1])    # Churn probability histogram
ax3 = fig.add_subplot(gs[0, 2])    # Recency vs frequency scatter
ax4 = fig.add_subplot(gs[1, :])    # Feature importances (full width)

df['rfm_score'].hist(ax=ax1, color='#e11d48', bins=10)
ax1.set_title('RFM Score Distribution')

df['churn_prob'].hist(ax=ax2, color='#7c3aed', bins=20)
ax2.set_title('Predicted Churn Probability')

ax3.scatter(df['recency'], df['frequency'], c=df['churn_prob'],
            cmap='RdYlGn_r', alpha=0.6)
ax3.set_title('Recency vs Frequency (colour = churn risk)')

importances = pd.Series(model.feature_importances_, index=X.columns)
importances.sort_values().plot(kind='barh', ax=ax4, color='#0284c7')
ax4.set_title('Feature Importances')

plt.tight_layout()
plt.savefig('guvi_analytics_dashboard.png', dpi=150)
plt.show()
conn.close()
