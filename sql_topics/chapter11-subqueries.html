Subqueries & Correlated Queries

Chapter 11 · Subqueries & Correlated Queries

A Query Inside a Query.
The Power of Nested SQL.

Subqueries let you use the result of one SELECT as the input to another. They're the building blocks of complex business logic — from "find learners above the class average" to "check if a record exists before inserting".

What is a Subquery?

A subquery (also called an inner query or nested query) is a SELECT statement placed inside another SQL statement — inside WHERE, FROM, SELECT, or HAVING.

🔍 Your First Subquery — Intuition

Imagine you want to find GUVI learners who scored above the class average. You can't write WHERE score > AVG(score) directly — that's invalid. But you can compute the average first as a subquery:

SQL
-- ❌ This is INVALID — can't use aggregate in WHERE
SELECT name FROM students WHERE score > AVG(score);

-- ✅ This works — compute AVG first in a subquery
SELECT name FROM students
WHERE  score > (SELECT AVG(score) FROM students);
📦 Scalar

Returns one value. Used in SELECT, WHERE, HAVING.

📋 Table

Returns a virtual table. Used in FROM clause.

🔗 Correlated

References the outer query. Runs once per outer row.

🎓
Terminology
Outer query — the main SELECT that contains the subquery.
Inner query — the nested SELECT inside parentheses.
SQL always evaluates the inner query first, then passes its result to the outer query (except correlated subqueries — those work differently!).

📦 Subquery Types

There are four types of subqueries, classified by what they return and how they're used. Understanding this distinction is key to choosing the right one.

1️⃣ Scalar Subquery — Returns ONE Value

Returns exactly one row, one column. Can be used anywhere a single value is expected — in SELECT, WHERE, HAVING, or even SET.

SQL — Scalar Subquery
-- In WHERE: find learners scoring above the overall average
SELECT name, score
FROM   guvi_learners
WHERE  score > (SELECT AVG(score) FROM guvi_learners);

-- In SELECT: show each learner's score vs the average
SELECT
  name,
  score,
  (SELECT ROUND(AVG(score),1) FROM guvi_learners) AS class_avg,
  score - (SELECT AVG(score) FROM guvi_learners) AS diff_from_avg
FROM   guvi_learners;
⚠️
If a scalar subquery returns MORE than one row, SQL throws an error. Always make sure it returns exactly 1 value, or use LIMIT 1.
2️⃣ Row Subquery — Returns ONE Row

Returns one complete row (multiple columns). Useful for comparing a row against another row's values.

SQL — Row Subquery
-- Find the student with the exact same score+city as student S01
SELECT * FROM guvi_learners
WHERE (score, city) =
  (SELECT score, city FROM guvi_learners WHERE student_id = 'S01')
AND   student_id != 'S01';
3️⃣ Table Subquery (Derived Table) — Returns a Virtual Table

Returns multiple rows and columns. Used in the FROM clause — the result becomes a temporary table you can query from.

SQL — Derived Table in FROM
-- Find batches where the average score is above 75
SELECT batch_id, avg_score
FROM  (
        SELECT batch_id, AVG(score) AS avg_score
        FROM   guvi_learners
        GROUP BY batch_id
      ) AS batch_summary        -- alias is REQUIRED for derived tables
WHERE avg_score > 75;
💡
The derived table must have an alias (AS batch_summary). Without it, MySQL/PostgreSQL will throw an error. This is a common beginner mistake.
4️⃣ Correlated Subquery — References the Outer Query

The inner query references a column from the outer query — so it runs once for every row of the outer query. More powerful but slower on large datasets.

SQL — Correlated Subquery
-- Find learners scoring above their OWN BATCH's average
SELECT outer_q.name, outer_q.batch_id, outer_q.score
FROM   guvi_learners outer_q
WHERE  outer_q.score > (
         SELECT AVG(inner_q.score)
         FROM   guvi_learners inner_q
         WHERE  inner_q.batch_id = outer_q.batch_id  ← references outer!
       );
🔗
The line WHERE inner_q.batch_id = outer_q.batch_id is what makes it correlated. For each row in outer_q, the subquery re-runs using that row's batch_id. Without this reference, it would be a plain non-correlated subquery.

📍 Subquery Placement

A subquery can appear in four different positions inside a SQL statement. Each placement has a specific use case and different rules.

WHERE clause
SELECT clause
FROM clause
HAVING clause
📍
WHERE clause subquery — Most common placement. Filter rows based on a value or set returned by the inner query.
🎯 Single value (scalar) in WHERE
-- Learners who scored above the overall class average
SELECT name, score
FROM   guvi_learners
WHERE  score > (SELECT AVG(score) FROM guvi_learners);
🎯 List of values (IN) in WHERE
-- Students enrolled in Python or ML courses
SELECT name, batch_id
FROM   students
WHERE  batch_id IN (
         SELECT batch_id FROM batches
         WHERE  course_id IN ('C01', 'C02')
       );
🎯 Most recent record in WHERE
-- Find the student with the highest score (scalar + MAX)
SELECT * FROM guvi_learners
WHERE  score = (SELECT MAX(score) FROM guvi_learners);
📍
SELECT clause subquery — Adds a computed column to each row. Must return exactly one value (scalar). Often used to show comparisons inline.
🎯 Show score vs class average per learner
-- Compare each learner's score against the class average
SELECT
  name,
  score,
  (SELECT ROUND(AVG(score),1) FROM guvi_learners) AS class_avg,
  score - (SELECT AVG(score) FROM guvi_learners)    AS gap
FROM   guvi_learners
ORDER BY gap DESC;
⚠️
Scalar subqueries in SELECT run once per row of the outer query — fine for small datasets, but can be slow on large tables. Use a JOIN or CTE instead for better performance.
🎯 Correlated scalar in SELECT — batch-level average
-- Show each student's score alongside their batch's average
SELECT
  name, batch_id, score,
  (SELECT ROUND(AVG(score),1)
   FROM   guvi_learners inner_q
   WHERE  inner_q.batch_id = outer_q.batch_id) AS batch_avg
FROM   guvi_learners outer_q;
📍
FROM clause subquery (Derived Table) — The subquery returns a temporary table that you query as if it were a real table. Must always have an alias.
🎯 Aggregate then filter with a derived table
-- Two-step aggregation: first calculate per-batch stats,
-- then filter to batches performing above threshold
SELECT
  bs.batch_id,
  bs.avg_score,
  bs.top_score,
  bs.learner_count
FROM (
  SELECT
    batch_id,
    ROUND(AVG(score),2) AS avg_score,
    MAX(score)           AS top_score,
    COUNT(*)            AS learner_count
  FROM   guvi_learners
  GROUP BY batch_id
) AS bs                  -- alias is mandatory!
WHERE  bs.avg_score > 75
ORDER BY bs.avg_score DESC;
🎯 Rank and paginate with a derived table
-- Get top 3 per city using a derived table with ranking
SELECT city, name, score, city_rank
FROM (
  SELECT
    city, name, score,
    RANK() OVER (PARTITION BY city ORDER BY score DESC) AS city_rank
  FROM guvi_learners
) AS ranked
WHERE city_rank <= 3;
📍
HAVING clause subquery — Filter aggregated groups based on a subquery result. Useful when the filter value itself requires a separate aggregation.
🎯 Batches performing above the overall average
-- Find batches whose average score exceeds the overall average
-- (nested aggregation — can't do this with just HAVING)
SELECT
  batch_id,
  ROUND(AVG(score), 2) AS batch_avg
FROM   guvi_learners
GROUP BY batch_id
HAVING AVG(score) > (
  SELECT AVG(score)      -- overall class average
  FROM   guvi_learners
)
ORDER BY batch_avg DESC;
🎯 Cities with more students than the average city size
-- Find cities where learner count is above average city count
SELECT city, COUNT(*) AS learners
FROM   guvi_learners
GROUP BY city
HAVING COUNT(*) > (
  SELECT AVG(city_count)
  FROM (
    SELECT city, COUNT(*) AS city_count
    FROM   guvi_learners
    GROUP BY city
  ) AS city_sizes
);
💡
Notice this is a nested subquery — a subquery inside a subquery! Readable today, but consider converting to a CTE when queries get this deep.

🔗 Correlated Subqueries — The Inner Query Knows the Outer Row

A correlated subquery is one where the inner query references a column from the outer query. This means it cannot run independently — it re-executes for every row the outer query processes.

🧠
How it works
Non-correlated: Inner query runs ONCE → result used by outer query.
Correlated: For each row outer processes → inner query runs AGAIN with that row's data.
🔬 Anatomy of a Correlated Subquery
SQL — Annotated
-- Outer query iterates over each learner one by one
SELECT o.name, o.batch_id, o.score   ← outer columns
FROM   guvi_learners o               ← outer table alias: o
WHERE  o.score > (
         SELECT AVG(i.score)            ← inner aggregate
         FROM   guvi_learners i          ← inner table alias: i
         WHERE  i.batch_id = o.batch_id  ← 🔑 THE LINK
       );                                  ↑ inner references OUTER
🔑
The key line is i.batch_id = o.batch_id. The outer alias o inside the inner query is what makes it correlated. The inner query runs once per row of o, each time using that row's batch_id to compute a batch-specific average.

⚡ Performance Note
⚠️
Correlated subqueries can be slow!If the outer table has 10,000 rows, the inner query runs 10,000 times. For large tables, consider rewriting with a JOIN + derived table, or a CTE — they compute the batch average once and join it in.
Equivalent JOIN (faster)
-- Same result, but batch_avg computed ONCE via GROUP BY
SELECT l.name, l.batch_id, l.score
FROM   guvi_learners l
INNER JOIN (
  SELECT batch_id, AVG(score) AS avg_score
  FROM   guvi_learners
  GROUP BY batch_id
) AS ba ON l.batch_id = ba.batch_id
WHERE  l.score > ba.avg_score;

✅ EXISTS & NOT EXISTS

EXISTS tests whether the inner query returns any rows at all — it doesn't care about the actual values, just whether a match exists. This makes it very efficient.

✅ EXISTS

Returns TRUE if the subquery returns at least one row. Use when asking "does this relationship exist?"

🚫 NOT EXISTS

Returns TRUE if the subquery returns NO rows. Use when asking "is anything missing?"

📖 EXISTS Syntax
SQL
SELECT columns
FROM   outer_table o
WHERE  EXISTS (
         SELECT 1           -- SELECT 1 is conventional — value doesn't matter
         FROM   inner_table i
         WHERE  i.key = o.key  -- correlation condition
       );
💡
Why SELECT 1?EXISTS only checks if any row is returned — it doesn't care what columns you select. Writing SELECT 1 instead of SELECT * is a convention that makes the intent explicit and slightly more efficient.
🏫 GUVI Example — Find Students Who Have Made a Payment
SQL — EXISTS
-- Students who HAVE made at least one payment
SELECT s.name, s.email, s.city
FROM   students s
WHERE  EXISTS (
         SELECT 1
         FROM   payments p
         WHERE  p.student_id = s.student_id
       );
🚫 NOT EXISTS — Find Students with NO Payment
SQL — NOT EXISTS
-- Students who have NOT made any payment yet — fee defaulters!
SELECT s.name, s.email, s.city
FROM   students s
WHERE NOT EXISTS (
         SELECT 1
         FROM   payments p
         WHERE  p.student_id = s.student_id
       );
✅
This is the safest way to find "missing" records. Unlike NOT IN, NOT EXISTS handles NULLs correctly and doesn't return wrong results when the subquery contains NULL values.
🛒 Zomato Example — Restaurants with No Orders This Month
SQL
-- Restaurants that received zero orders in June 2024
SELECT r.restaurant_name, r.city
FROM   restaurants r
WHERE NOT EXISTS (
  SELECT 1
  FROM   orders o
  WHERE  o.restaurant_id = r.restaurant_id
  AND    o.order_date BETWEEN '2024-06-01' AND '2024-06-30'
);

⚡ IN vs EXISTS — Performance & Behaviour

Both IN and EXISTS can solve the same problems, but they behave differently under the hood. Knowing when to use which is an important interview and production skill.

🔄 Same Problem — Two Solutions
SQL — Using IN
-- Find students enrolled in AIML courses — using IN
SELECT name FROM students
WHERE  batch_id IN (
  SELECT batch_id FROM batches WHERE course_id = 'C01'
);
SQL — Using EXISTS
-- Same result — using EXISTS
SELECT s.name FROM students s
WHERE EXISTS (
  SELECT 1 FROM batches b
  WHERE  b.batch_id   = s.batch_id
  AND    b.course_id  = 'C01'
);

☠️ The NULL Trap with NOT IN
SQL — The Dangerous Case
-- Suppose payments table has student_id: S01, S02, NULL
-- NOT IN with a NULL in the list ALWAYS returns FALSE!

-- This returns 0 rows even if many students have no payment:
SELECT name FROM students
WHERE  student_id NOT IN (
  SELECT student_id FROM payments  -- if this returns ANY NULL...
);                                  -- ...the whole query returns nothing!

-- Safe alternative: filter NULLs explicitly
WHERE  student_id NOT IN (
  SELECT student_id FROM payments WHERE student_id IS NOT NULL
);

-- Or better — use NOT EXISTS
🚨
Rule: Never use NOT IN when the subquery might return NULLs.Always use NOT EXISTS, or add WHERE col IS NOT NULL to the subquery. This is one of the most common silent bugs in SQL production code.

🏆 Classic: Learners Scoring Above Their Batch Average

This is the most famous correlated subquery example — a favourite in interviews, assessments, and real analytics. Let's build it three different ways and understand the trade-offs.

🎯
The ProblemFind all GUVI learners who scored above the average score of their own batch — not the overall class average, but their specific batch's average. Each batch's average is different.
Method 1 — Correlated Subquery (Classic)
SQL — Correlated
-- For each learner, compute their batch's average on the fly
SELECT
  o.name,
  o.batch_id,
  o.score,
  ROUND((SELECT AVG(i.score)
          FROM   guvi_learners i
          WHERE  i.batch_id = o.batch_id), 1) AS batch_avg
FROM   guvi_learners o
WHERE  o.score > (
         SELECT AVG(i.score)
         FROM   guvi_learners i
         WHERE  i.batch_id = o.batch_id
       )
ORDER BY o.batch_id, o.score DESC;
⚠️
Elegant but slow on large tables. Inner query runs once per learner row.
Method 2 — JOIN with Derived Table (Faster ⚡)
SQL — JOIN
-- Compute batch averages ONCE, then join — more efficient
SELECT
  l.name,
  l.batch_id,
  l.score,
  ba.avg_score AS batch_avg
FROM   guvi_learners l
INNER JOIN (
  SELECT batch_id, ROUND(AVG(score),1) AS avg_score
  FROM   guvi_learners
  GROUP BY batch_id
) AS ba ON l.batch_id = ba.batch_id
WHERE  l.score > ba.avg_score
ORDER BY l.batch_id, l.score DESC;
✅
Batch averages computed only once. Much faster for large datasets.
Method 3 — CTE (Most Readable 📖)
SQL — CTE
-- Same logic, but CTE makes it self-documenting
WITH batch_avgs AS (
  SELECT batch_id, ROUND(AVG(score),1) AS avg_score
  FROM   guvi_learners
  GROUP BY batch_id
)
SELECT
  l.name, l.batch_id, l.score,
  ba.avg_score AS batch_avg,
  l.score - ba.avg_score AS above_by
FROM   guvi_learners l
JOIN   batch_avgs    ba ON l.batch_id = ba.batch_id
WHERE  l.score > ba.avg_score
ORDER BY above_by DESC;
💡
CTE = clearest intention, easiest to debug, same performance as derived table. This is the production-preferred pattern.

🧭 Subquery vs JOIN vs CTE — When to Use Which?

These three tools often solve the same problem. The right choice depends on readability, performance, and reuse needs.

🗺️ Decision Guide
Do you need the intermediate result more than once?
↙ YES                          NO ↘
→ Use CTE
Define once, use many times. Best for readable complex logic.
Is it a simple existence check?
↙ YES      NO ↘
→ Use EXISTS
Fastest for "does a match exist?"
→ Use JOIN
Best for combining columns from multiple tables.

📝 The Readability Trap — Nested Subqueries
🚨
Warning: Nested subqueries become unreadable fast!When you see more than 2 levels of nesting, it's a strong signal to refactor into a CTE.
Hard to read — 3 levels of nesting
SELECT * FROM students
WHERE batch_id IN (
  SELECT batch_id FROM batches
  WHERE course_id IN (
    SELECT course_id FROM courses
    WHERE fee > (SELECT AVG(fee) FROM courses)
  )
);
Same logic — clean with CTEs
WITH above_avg_courses AS (
  SELECT course_id FROM courses
  WHERE  fee > (SELECT AVG(fee) FROM courses)
),
premium_batches AS (
  SELECT batch_id FROM batches
  WHERE  course_id IN (SELECT course_id FROM above_avg_courses)
)
SELECT * FROM students
WHERE  batch_id IN (SELECT batch_id FROM premium_batches);
