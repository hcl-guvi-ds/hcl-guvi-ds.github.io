SQL Joins — All Types

Chapter 10 · SQL Joins

Data Lives in Multiple Tables.
Joins Bring It Together.

In the real world, no single table holds everything. Student data is separate from course data, instructor data, payment data. Joins are how you combine them into one meaningful result.

Why Joins Exist

Databases are designed with normalisation — each piece of information is stored in one place to avoid redundancy. Joins are the bridge that brings these separate tables together for a query.

🗂️ All 6 Join Types at a Glance
🔵
INNER JOIN
Only matching rows from both tables
🟢
LEFT JOIN
All rows from left + matches from right
🟣
RIGHT JOIN
All rows from right + matches from left
🔷
FULL OUTER
Everything from both tables
✖️
CROSS JOIN
Every combination (cartesian product)
🔄
SELF JOIN
Table joined with itself
💡
Venn Diagram Mental ModelThink of each table as a circle. The JOIN type determines which parts of the overlapping circles you keep — just the intersection, or also the parts that don't overlap.

🔵 INNER JOIN — Only Matching Rows

INNER JOIN returns only rows where the join condition is TRUE in BOTH tables. If a row in the left table has no match in the right table (or vice versa), it is excluded.

🎯
Think of it as an intersection.Like a Venn diagram where you take only the overlapping centre. If a student has no batch assigned (Meera), or a batch has no matching students, those rows disappear from the result.
📖 Syntax
SQL
SELECT   columns
FROM     table_A
INNER JOIN table_B
  ON table_A.key = table_B.key;
🏫 GUVI Example 1 — Students with their Batch Info
SQL
-- Get student names along with their batch name
SELECT   s.student_id,
         s.name           AS student_name,
         s.city,
         b.batch_name,
         b.start_date
FROM     students s
INNER JOIN batches b
  ON s.batch_id = b.batch_id;

🏫 GUVI Example 2 — 3-Table INNER JOIN
SQL
-- Student → Batch → Course: what course is each student studying?
SELECT   s.name          AS student,
         b.batch_name    AS batch,
         c.course_name   AS course,
         c.duration_weeks,
         i.name          AS instructor
FROM     students s
INNER JOIN batches b     ON s.batch_id      = b.batch_id
INNER JOIN courses c     ON b.course_id     = c.course_id
INNER JOIN instructors i ON b.instructor_id = i.instructor_id
ORDER BY s.name;
🔀 ON vs USING Clause

When the column name is the same in both tables, you can use USING as a shorthand for ON.

SQL
-- Using ON (verbose but clear)
FROM students s
INNER JOIN batches b ON s.batch_id = b.batch_id

-- Using USING (shorthand when column names match)
FROM students s
INNER JOIN batches b USING (batch_id)
💡
USING vs ON
Use USING(col) when both tables have the exact same column name and you join on equality. Use ON for everything else — different column names, non-equi joins, complex conditions.

🟢 LEFT JOIN & 🟣 RIGHT JOIN

Outer joins include non-matching rows from one or both tables, filling missing values with NULL.

🟢 LEFT JOIN (LEFT OUTER JOIN)

Returns all rows from the left table, and matching rows from the right table. Where there's no match, right-side columns are NULL.

🟣 RIGHT JOIN (RIGHT OUTER JOIN)

Returns all rows from the right table, and matching rows from the left table. Where there's no match, left-side columns are NULL.

🟢 LEFT JOIN — All Students, Even Without a Batch
SQL
-- Show ALL students, include batch info if available
SELECT   s.name          AS student,
         s.city,
         b.batch_name,
         b.start_date
FROM     students s
LEFT JOIN batches b
  ON s.batch_id = b.batch_id;

🔷 FULL OUTER JOIN — Everything

FULL OUTER JOIN returns all rows from both tables. Where there's a match, columns are filled. Where there's no match, the missing side gets NULL. It's LEFT + RIGHT combined.

📖 FULL OUTER JOIN Syntax
PostgreSQL / SQL Server
SELECT   s.name, b.batch_name
FROM     students s
FULL OUTER JOIN batches b
  ON s.batch_id = b.batch_id;

MySQL workaround
SELECT s.name, b.batch_name

UNION

SELECT s.name, b.batch_name
FROM   students s
RIGHT JOIN batches b ON s.batch_id = b.batch_id;

✖️ CROSS JOIN — Cartesian Product

CROSS JOIN produces every possible combination of rows from both tables. If Table A has 5 rows and Table B has 4 rows, result = 5 × 4 = 20 rows. No ON clause needed.

🚨
Use with caution!CROSS JOIN on large tables creates an enormous result. 1,000 rows × 1,000 rows = 1,000,000 rows. Always know your table sizes before using CROSS JOIN.
📖 CROSS JOIN Syntax
SQL
-- No ON clause needed — every row pairs with every row
SELECT  c.course_name,
        i.name AS instructor
FROM    courses c
CROSS JOIN instructors i;
🏫 Real Use Case — Generate All Possible Course-Instructor Pairs
SQL
-- For scheduling: generate all possible (batch × course) combinations
SELECT  b.batch_name,
        c.course_name,
        c.fee
FROM    batches b
CROSS JOIN courses c
ORDER BY b.batch_name, c.course_name;

Result: 4 batches × 3 courses = 12 rows, showing every possible assignment. Useful for scheduling tools or test data generation.

🎯 When is CROSS JOIN actually useful?
  • 1
    Generating test data

    Quickly create all combinations — e.g., all (batch × slot) pairs for a timetable generator.

  • 2
    Creating date series

    Cross join a number table with a month table to generate a full calendar grid.

  • 3
    Recommendation grids

    Generate all (student × course) pairs, then filter for courses they haven't enrolled in yet.

🔄 SELF JOIN — Table Joining Itself

A SELF JOIN is when a table is joined to itself. This sounds strange, but it's very useful for hierarchical or comparative data — like finding students in the same batch, or employees and their managers.

💡
When do you need a SELF JOIN?When relationships exist within the same table. For example — an employees table where each employee has a manager_id that points to another employee in the same table. Or finding students who share the same city.
🏫 Example 1 — Find Students in the Same Batch
SQL
-- Find all pairs of students in the same batch (buddy system!)
SELECT   s1.name AS student_1,
         s2.name AS student_2,
         s1.batch_id
FROM     students s1
INNER JOIN students s2
  ON  s1.batch_id = s2.batch_id
  AND s1.student_id < s2.student_id  -- avoid duplicates & self-pairs
ORDER BY s1.batch_id;
💡
The condition s1.student_id < s2.student_id prevents showing the same pair twice (Priya-Ravi AND Ravi-Priya) and also prevents a student pairing with themselves (Priya-Priya).
👔 Example 2 — Employee Manager Hierarchy (classic)
SQL
-- Table: employees (emp_id, name, manager_id)
-- manager_id points to another emp_id in the SAME table
SELECT   e.name   AS employee,
         m.name   AS their_manager
FROM     employees e
LEFT JOIN employees m          -- same table, different alias
  ON e.manager_id = m.emp_id;  -- e's manager_id = m's emp_id

⚡ Advanced Join Patterns

These patterns come up in real interviews and production queries. Master them and you'll be ahead of 90% of SQL users.

🔗 1. Multi-Table Joins (3+ tables)

Chain multiple JOINs to combine 3 or more tables. Each JOIN adds one more table to the result.

SQL — Full Student Report
-- Complete student info: name → batch → course → instructor
SELECT
  s.name            AS student,
  s.city,
  b.batch_name,
  c.course_name,
  c.fee             AS course_fee,
  i.name            AS instructor,
  i.specialization
FROM       students    s
INNER JOIN batches     b  ON s.batch_id      = b.batch_id
INNER JOIN courses     c  ON b.course_id     = c.course_id
INNER JOIN instructors i  ON b.instructor_id = i.instructor_id
ORDER BY   s.name;
📐 2. Non-Equi Join — Join on a Range

Joins don't have to use =. You can join on <, >, BETWEEN, or any condition. Useful for grading, tier assignment, date ranges.

SQL — Assign Grade Based on Score Range
-- grades table: (grade, min_score, max_score)
-- Join student scores to grade ranges using BETWEEN
SELECT
  s.name,
  s.quiz_score,
  g.grade
FROM       students s
INNER JOIN grade_bands g
  ON s.quiz_score BETWEEN g.min_score AND g.max_score;

-- grade_bands table:
-- (A, 90, 100), (B, 75, 89), (C, 60, 74), (D, 0, 59)
📊 3. JOIN with Aggregates — Powerful Combo
SQL — Learner count & avg score per course
-- How many students per course? What's their average score?
SELECT
  c.course_name,
  i.name                      AS instructor,
  COUNT(s.student_id)         AS total_students,
  ROUND(AVG(s.quiz_score), 1) AS avg_score,
  MAX(s.quiz_score)           AS top_score
FROM       courses     c
INNER JOIN batches     b  ON c.course_id     = b.course_id
INNER JOIN students    s  ON b.batch_id      = s.batch_id
INNER JOIN instructors i  ON b.instructor_id = i.instructor_id
GROUP BY   c.course_name, i.name
HAVING     COUNT(s.student_id) >= 5
ORDER BY   avg_score DESC;
∅ 4. Handling NULLs in Outer Joins
SQL — Check if students have fees pending
-- Find students who have NOT made a payment yet
-- LEFT JOIN + IS NULL on right table = Anti-join pattern
SELECT   s.name,
         s.email,
         s.city
FROM     students s
LEFT JOIN payments p
  ON s.student_id = p.student_id
WHERE    p.payment_id IS NULL;
-- WHERE IS NULL on the right-table PK = "no match found"
✅
Anti-Join Pattern
LEFT JOIN + WHERE right_table.id IS NULL = "give me rows from left that have NO matching row in right." This is how you find students with no payments, courses with no enrollments, batches with no instructors, etc.
🚫 5. Anti-Join — "What's Missing?"
SQL
-- Pattern 1: LEFT JOIN + IS NULL (most common)
SELECT s.name
FROM   students s
LEFT JOIN enrollments e ON s.student_id = e.student_id
WHERE  e.student_id IS NULL;
-- "Students who have NOT enrolled in any course"

-- Pattern 2: NOT EXISTS (sometimes cleaner)
SELECT s.name
FROM   students s
WHERE NOT EXISTS (
  SELECT 1 FROM enrollments e
  WHERE  e.student_id = s.student_id
);

-- Pattern 3: NOT IN (use with caution — fails if NULLs in subquery!)
SELECT name FROM students
WHERE  student_id NOT IN (SELECT student_id FROM enrollments);
⚠️
NOT IN danger!If the subquery returns even one NULL, NOT IN returns no results at all. Always prefer LEFT JOIN + IS NULL or NOT EXISTS over NOT IN.
