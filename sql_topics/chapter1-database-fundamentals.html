Topic 1

What is a Database?

A database is an organized collection of data that is stored and accessed electronically. Think of it as a super-powered filing system — but instead of paper folders in a cabinet, data is stored digitally, structured in a way that makes it easy to search, update, and retrieve in milliseconds.

💡
Simple Definition
A database is like a digital Excel sheet on steroids — it stores millions of rows, handles multiple users at once, keeps data safe, and lets you ask complex questions (queries) about your data instantly.
🇮🇳 Real-World Example
Zomato's database at work
Every time you order biryani on Zomato, their database is doing all of this simultaneously:

✅ Storing your order details (restaurant, items, total)
✅ Updating the restaurant's order count
✅ Saving your delivery address
✅ Recording payment status
✅ Notifying the delivery partner
✅ Updating your order history

All of this happens in under 1 second, for millions of users at the same time. That's the power of a proper database.

Why not just use Excel?

📊 Excel / Spreadsheet
Max ~1 million rows (performance degrades fast)
One person edits at a time
No user permissions or security
Data can be easily corrupted or overwritten
Hard to link data across sheets
Manual, slow for large data operations
🗄️ Database
Billions of rows with lightning speed
Thousands of users simultaneously
Role-based access control (admin, reader, etc.)
ACID transactions protect data integrity
Relationships between tables (JOINs)
Automated, optimized with indexes

Topic 2

DBMS vs RDBMS

A DBMS (Database Management System) is software that stores, retrieves, and manages data. An RDBMS (Relational DBMS) is a special type of DBMS where data is stored in related tables and follows strict rules. SQL works with RDBMS.

DBMS — Database Management System

A DBMS stores data in various formats — files, hierarchies, or networks. It doesn't enforce relationships between data. Think of it as a simple filing system.

📁
Examples of DBMS (non-relational)
File system (your computer folders), Microsoft Access (basic), early IBM IMS (hierarchical), MongoDB (document-based NoSQL)
DBMS Characteristics
Data stored in files or simple structures
No strict relationships between data
Limited support for multiple users
No ACID transaction support
Less data integrity enforcement
Examples: File system, XML stores, early databases

RDBMS — Relational Database Management System

An RDBMS stores data in tables (relations) and enforces relationships between them using keys. It follows E.F. Codd's 12 rules for relational databases. SQL is the language used to interact with RDBMS.

🔑
Key Concept: "Relational" means related tables
A Students table and a Courses table are related — a student can enroll in many courses. RDBMS enforces and manages this relationship automatically.
RDBMS Characteristics
Data stored in structured tables (rows + columns)
Relationships enforced via Primary & Foreign Keys
Supports thousands of concurrent users
Full ACID transaction support
SQL as standard query language
Strong data integrity and constraints

Popular RDBMS in Industry

MySQL
Open-source, widely used in web apps. Powers Facebook, YouTube, Twitter.
Free & beginner-friendly
PostgreSQL
Advanced open-source RDBMS. Supports JSON, full-text search, and more.
Used in data engineering
SQLite
Lightweight, file-based. Used in mobile apps, Python projects.
Best for learning & prototyping
Oracle DB
Enterprise-grade, used in banking, telecom, large corporations.
HDFC, Airtel, etc.

Microsoft's enterprise RDBMS, tightly integrated with Windows ecosystem.
Used in IT companies

Used for data analytics
🎯
Which should you learn?
The core SQL you learn is the same across all databases (95% identical). We'll use MySQL syntax with notes for PostgreSQL/SQLite differences where they vary. Once you know SQL, switching databases takes just days.

Topic 3

Structured vs Unstructured vs Semi-Structured Data

Not all data looks the same. As a data scientist, you'll work with all three types. SQL primarily deals with structured data — but knowing the difference helps you choose the right tool for the right job.

🟩 Structured Data SQL's domain

Data that fits neatly into rows and columns — like a table. Each column has a defined data type and every row follows the same format.

Examples:
📋 Customer records (name, phone, city)
💰 Bank transactions (date, amount, type)
🎓 Student marks (subject, score, grade)
📦 Product inventory (SKU, price, stock)
🟥 Unstructured Data No fixed format

Data with no predefined format — no rows, no columns. Needs special tools to process (NLP, Computer Vision, etc.).

Examples:
📧 Emails, customer reviews, WhatsApp chats
🖼️ Images (Instagram posts, Aadhaar photos)
🎥 Videos (YouTube, CCTV footage)
🎙️ Audio (call recordings, podcasts)
🟨 Semi-Structured In between

Data that has some structure but doesn't fit into a rigid table. Has tags or markers to separate elements.

Examples:
📄 JSON (API responses, MongoDB)
📃 XML (older web services, config files)
📊 CSV (exports from Excel/tools)
🌐 HTML (web pages with structure)
🇮🇳 Real-World Example
Swiggy's data — all three types
Structured: Order table — order_id, customer_id, restaurant_id, total_amount, status
Unstructured: Customer reviews — "The biryani was amazing but delivery was late 😤"
Semi-structured: The API response from payment gateway in JSON format

A data scientist at Swiggy uses SQL for the structured data, NLP for reviews, and JSON parsers for API data. SQL is always the foundation.
📌
90% of business data is structured or can be structured
SQL handles the vast majority of real-world data tasks in companies. Whether you go into data engineering, analytics, or ML — SQL knowledge is always needed. Modern databases like PostgreSQL even support JSON columns, bringing semi-structured data into the SQL world.

Topic 5

Keys & Relationships

Keys are what make the "Relational" in RDBMS meaningful. They uniquely identify rows and establish connections between tables. This is the core concept that separates databases from flat files.

🔑 Primary Key (PK)

A column (or combination of columns) that uniquely identifies each row in a table. No two rows can have the same Primary Key value, and it cannot be NULL.

📌
Rules for Primary Keys
✅ Must be UNIQUE — no two rows can share the same PK value
✅ Must be NOT NULL — every row must have a PK
✅ Should never change — once assigned, PK stays fixed
✅ Ideally a simple integer (student_id, order_id)
SQL — Primary KeyMySQL
CREATE TABLE students ( student_id INT PRIMARY KEY, -- This is the PK name VARCHAR(100) NOT NULL, email VARCHAR(150) UNIQUE, city VARCHAR(50) );
🔗 Foreign Key (FK)

A column in one table that references the Primary Key of another table. This creates a link (relationship) between the two tables. Foreign Keys enforce referential integrity — you can't reference a row that doesn't exist.

⚠️
Referential Integrity in action
If you try to add an enrollment with student_id = 99 but student 99 doesn't exist in the students table — the database will REJECT that insert. This is referential integrity — the database protects you from bad data automatically.
🗝️ Other Key Types
Candidate Key
Any column that could be a Primary Key (unique + not null). A table can have multiple candidates — only one is chosen as PK.
e.g., both student_id and email could be PKs — we choose student_id
Composite Key
A Primary Key made of two or more columns together. Used when no single column is unique enough.
e.g., (student_id + course_id) together is unique in an enrollments table
Unique Key
Like PK but allows one NULL. Ensures uniqueness but isn't the primary identifier.
e.g., email column — must be unique but can be null if not provided
Surrogate Key
An auto-generated numeric key (like AUTO_INCREMENT) with no business meaning. Created purely to serve as PK.
e.g., student_id = 1, 2, 3… (not the Aadhaar number)

Topic 6

Data Types in SQL

Every column in a SQL table has a data type — it tells the database what kind of data to expect and how to store it. Choosing the right data type is important: it affects storage space, speed, and what operations you can do.

💡
Rule of thumb
Always use the smallest data type that covers your need. Don't use VARCHAR(1000) for a 2-letter country code. Don't use BIGINT for an age. The right type = less storage + faster queries.

Topic 7

Constraints — Rules that Protect Your Data

Constraints are rules you set on columns to control what data can be inserted. They prevent bad data from entering your database — the database enforces these rules automatically at all times.

NOT NULL

Column must always have a value. Cannot be left empty/null.

SQL
name VARCHAR(100) NOT NULL
Use: Any column that must always have data — name, email, order_amount
UNIQUE

No two rows can have the same value. Allows one NULL.

SQL
email VARCHAR(150) UNIQUE
Use: email, phone number, username, Aadhaar number
DEFAULT

Automatically fills a value if none is provided during INSERT.

SQL
is_active BOOLEAN DEFAULT TRUE
Use: created_at, is_active, status columns
CHECK

Ensures column value passes a condition. Rejects if condition fails.

SQL
age INT CHECK (age >= 18)
Use: age >= 18, rating BETWEEN 1 AND 5, salary > 0
PRIMARY KEY

Combines NOT NULL + UNIQUE. Identifies each row uniquely.

SQL
id INT PRIMARY KEY AUTO_INCREMENT
Use: Every table should have exactly one PK
FOREIGN KEY

Enforces that values in this column must exist in another table.

SQL
FOREIGN KEY (sid) REFERENCES students(id)
Use: Any column that references another table's PK

Topic 8

Entity Relationship Diagrams (ERD)

Before you write any SQL, database designers create an ER Diagram — a visual blueprint that shows all the tables, their columns, and how they relate to each other. Think of it as the architect's plan before building a house.

ERD Symbols
Rectangle — Entity (Table)
Oval/Ellipse — Attribute (Column)
Diamond — Relationship between entities
Line — Connection between entity & attribute
Double oval — Multi-valued attribute
Underlined — Primary Key attribute
Relationship Types
1:1 — One-to-One: Each person has one Aadhaar
1:N — One-to-Many: One teacher teaches many students
M:N — Many-to-Many: Many students take many courses
Cardinality — How many of each side
Crow's foot — Modern notation for cardinality
