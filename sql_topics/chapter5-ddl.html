DDL — Data Definition Language

DDL is where you build the skeleton of your database — creating tables, defining columns, setting constraints, adding indexes, and modifying structure. Every data engineer, DBA, and backend developer writes DDL daily. Master this chapter and you'll be designing production-grade schemas from scratch.

Topic 1

DDL Overview — What & Why

DDL (Data Definition Language) is the set of SQL commands used to define and manage the structure of a database — tables, columns, indexes, constraints, and databases themselves. DDL doesn't touch the data — it builds the container that holds data.

🏗️
CREATE
Build new databases, tables, indexes
✏️
ALTER
Modify existing table structure
🗑️
DROP
Delete tables, databases, indexes
🔄
TRUNCATE
Empty all rows, keep structure
📝
RENAME
Rename tables or columns
📌
DDL is Auto-Committed
Unlike DML (INSERT/UPDATE/DELETE) which you can ROLLBACK, most DDL commands in MySQL are auto-committed immediately — there's no "undo." DROP TABLE? Gone forever. This is why careful DDL usage is critical. Always test on staging environments before production!
🇮🇳 Real-World Scenario
When do Data Scientists write DDL?
✅ Creating tables to store scraped data from e-commerce sites
✅ Setting up staging tables for ETL pipelines
✅ Adding columns to store ML prediction scores
✅ Creating index on a slow-performing column after profiling
✅ Modifying table structure as requirements evolve
✅ Building the GUVI LMS schema from scratch for a new batch

Topic 2

CREATE DATABASE & SCHEMA

Before creating tables, you need a database to hold them. A database is the top-level container — it groups related tables together. In MySQL, DATABASE and SCHEMA are synonymous.

CREATE DATABASE — Full SyntaxMySQL
-- Create a new database CREATE DATABASE guvi_lms; -- Safe version: only creates if it doesn't already exist CREATE DATABASE IF NOT EXISTS guvi_lms; -- With character set (UTF-8 supports all Indian languages!) CREATE DATABASE guvi_lms CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci; -- Switch to using this database USE guvi_lms; -- See all databases SHOW DATABASES; -- See current database SELECT DATABASE(); -- Drop a database (⚠️ IRREVERSIBLE!) DROP DATABASE IF EXISTS guvi_lms;
💡
Always use utf8mb4 for Indian applications
utf8mb4 supports all Unicode characters including Hindi, Tamil, Telugu, and even emojis! The older utf8 in MySQL only supports up to 3-byte characters. For any app that stores Indian language text, always specify CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci.

💰 Always use DECIMAL for money — FLOAT causes rounding errors (0.1 + 0.2 ≠ 0.3 in binary!)

📝 VARCHAR vs TEXT → Use VARCHAR (< 255 chars) for indexed columns, TEXT for large content

Topic 5

Constraints — Enforcing Data Rules

Constraints are rules applied to columns that the database enforces automatically — every INSERT and UPDATE is checked against them. They're your first line of defense against bad data.

🔶 NOT NULL
Column must always have a value — cannot be left empty (NULL).
Use for columns that are required in every row.
name VARCHAR(100) NOT NULL, email VARCHAR(150) NOT NULL
🔷 UNIQUE
No two rows can have the same value. Allows one NULL (since NULL ≠ NULL).
email VARCHAR(150) UNIQUE, -- or table-level: UNIQUE (email)
🟢 DEFAULT
Automatically fills this value when none is provided on INSERT.
is_active BOOLEAN DEFAULT TRUE, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, status ENUM(...) DEFAULT 'active'
🔮 CHECK
Row is only accepted if the condition evaluates to TRUE. Rejects invalid values.
age TINYINT CHECK (age BETWEEN 18 AND 100), score DECIMAL CHECK (score >= 0 AND score <= 100), fee DECIMAL CHECK (fee > 0)
🔑 PRIMARY KEY
Uniquely identifies each row. Combines NOT NULL + UNIQUE. One per table.
id INT PRIMARY KEY AUTO_INCREMENT, -- or composite: PRIMARY KEY (student_id, course_id)
🔗 FOREIGN KEY
Links to another table's PK. Enforces referential integrity — referenced row must exist.
FOREIGN KEY (student_id) REFERENCES students(student_id) ON DELETE CASCADE

Topic 6

PRIMARY KEY — Single & Composite

The Primary Key (PK) is the most important constraint — it uniquely identifies every row in a table. A table can have only ONE PK, but that PK can span multiple columns (Composite PK).

Primary Key — All VariationsMySQL
-- Method 1: Inline (single column PK) CREATE TABLE students ( student_id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(100) ); -- Method 2: Table-level (single or composite) CREATE TABLE enrollments ( student_id INT NOT NULL, course_id INT NOT NULL, enrolled_on DATE, PRIMARY KEY (student_id, course_id) -- Composite PK ); -- AUTO_INCREMENT: MySQL auto-generates sequential numbers -- PostgreSQL uses SERIAL or GENERATED ALWAYS AS IDENTITY -- SQLite uses INTEGER PRIMARY KEY AUTOINCREMENT -- Natural Key vs Surrogate Key: -- Natural: email, Aadhaar, PAN (real-world identifier) -- Surrogate: id INT AUTO_INCREMENT (DB-generated, no real meaning) -- Best practice: Use SURROGATE KEYS as PK — they never change! -- UUID as PK (used in distributed systems): -- id CHAR(36) PRIMARY KEY DEFAULT (UUID())
💡
When to use Composite PK?
Use composite PK for junction/link tables where the combination of two FKs is unique:
• enrollments: (student_id, course_id) — a student enrolls in a course once
• order_items: (order_id, product_id) — one product per order line
• friends: (user_id, friend_id) — user A friends user B once

If you find yourself needing a surrogate PK on a junction table anyway (for referencing from other tables), add one as enrollment_id INT AUTO_INCREMENT.

Topic 7

FOREIGN KEY — Referential Integrity

A Foreign Key (FK) is a column that references the Primary Key of another table. It enforces referential integrity — you cannot insert a FK value that doesn't exist in the referenced table, and you cannot delete a referenced row without handling the children first.

📋 students (Parent)
🔑 student_id (PK)
name
email
student_id = 1,2,3...
→
FK enforces
this link
📋 enrollments (Child)
🔑 enrollment_id (PK)
🔗 student_id (FK)
course_id
score
student_id must exist in students!
FOREIGN KEY — Full SyntaxMySQL
CREATE TABLE enrollments ( enrollment_id INT PRIMARY KEY AUTO_INCREMENT, student_id INT NOT NULL, course_id INT NOT NULL, -- Inline FK (simple) -- student_id INT REFERENCES students(student_id) -- Table-level FK (recommended — allows naming + ON DELETE) CONSTRAINT fk_enroll_student FOREIGN KEY (student_id) REFERENCES students(student_id) ON DELETE CASCADE ON UPDATE CASCADE, CONSTRAINT fk_enroll_course FOREIGN KEY (course_id) REFERENCES courses(course_id) ON DELETE RESTRICT ); -- Check existing foreign keys on a table: SHOW CREATE TABLE enrollments; -- Drop a specific FK constraint: ALTER TABLE enrollments DROP FOREIGN KEY fk_enroll_student;
⚠️
FK Rules to Remember
1. Parent table must be created BEFORE child table
2. Referenced column must be PK or UNIQUE in parent table
3. Child column data type must match parent column data type EXACTLY
4. In MySQL, FK only works with InnoDB engine (not MyISAM!)
5. Always name your FK constraints — makes debugging and altering easier

Topic 8

ON DELETE / ON UPDATE Actions

When you try to DELETE a parent row or UPDATE a parent's PK that has child rows referencing it, the database needs to know what to do. This is controlled by ON DELETE and ON UPDATE actions.

🎮 Interactive — Which action to use when?
DELETE student_id=1 from students → CASCADE → ALL enrollments WHERE student_id=1 are also deleted automatically

ON DELETE CASCADE automatically deletes child rows when the parent row is deleted.

✅ Use when: Child rows have no meaning without the parent. If you delete a student, their enrollments should also disappear.
❌ Avoid when: Child data must be preserved for audit trails, history, or compliance. Deleting an instructor should NOT delete the courses — you'd lose the course catalog!

FOREIGN KEY (student_id) REFERENCES students(student_id) ON DELETE CASCADE
DELETE instructor_id=5 from instructors → SET NULL → courses.instructor_id = NULL (course still exists, but unassigned)

ON DELETE SET NULL sets the FK column to NULL when the parent row is deleted.

✅ Use when: The child can exist without the parent but should show the relationship is gone. If an instructor leaves GUVI, courses still exist — they just need a new instructor.
⚠️ Requirement: The FK column must be NULLable (not NOT NULL).

FOREIGN KEY (instructor_id) REFERENCES instructors(instructor_id) ON DELETE SET NULL
DELETE course_id=3 from courses → RESTRICT → ERROR! Cannot delete — active enrollments reference this course

ON DELETE RESTRICT blocks the DELETE if any child rows reference the parent. This is the SAFEST option — prevents accidental data loss.

✅ Use when: You should never delete a parent that has children. A course with active enrollments should NEVER be deleted — admin must manually resolve enrollments first.
📌 In MySQL, RESTRICT and NO ACTION behave identically.

FOREIGN KEY (course_id) REFERENCES courses(course_id) ON DELETE RESTRICT
DELETE parent row → NO ACTION → Same as RESTRICT in MySQL (deferred check in some DBs)

ON DELETE NO ACTION in MySQL behaves identically to RESTRICT — the delete is blocked if children exist. In standard SQL (Oracle, PostgreSQL), NO ACTION defers the check to end-of-transaction, while RESTRICT checks immediately.

📌 Default when no action is specified: NO ACTION / RESTRICT.

DELETE parent row → SET DEFAULT → FK column set to its DEFAULT value

ON DELETE SET DEFAULT sets the FK column to its default value when parent is deleted. Rarely used in MySQL (not fully supported in InnoDB). More common in PostgreSQL.

✅ Use when: You have a "catch-all" parent record (e.g., a default instructor ID = 0) to assign unassigned children to.

🎯
Quick Decision Guide — Which action to choose?
"If parent is deleted, what should happen to children?"
→ Children should die too? → CASCADE
→ Children should remain but unlinked? → SET NULL
→ Prevent deletion until children are removed manually? → RESTRICT

Topic 9

ALTER TABLE — Modifying Existing Tables

ALTER TABLE lets you modify a table's structure after it's been created — adding columns, changing data types, dropping columns, renaming, and modifying constraints. This is used constantly as requirements evolve.

ADD COLUMN — Adding new columns to existing tableMySQL
-- Add a single column ALTER TABLE students ADD COLUMN linkedin_url VARCHAR(200); -- Add column at a specific position ALTER TABLE students ADD COLUMN middle_name VARCHAR(50) AFTER name; -- Add column as the FIRST column ALTER TABLE students ADD COLUMN student_code VARCHAR(20) FIRST; -- Add multiple columns at once ALTER TABLE students ADD COLUMN github_url VARCHAR(200), ADD COLUMN portfolio_url VARCHAR(200), ADD COLUMN updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP;
MODIFY COLUMN — Change data type or constraintsMySQL
-- Change data type (e.g., expand VARCHAR size) ALTER TABLE students MODIFY COLUMN name VARCHAR(200) NOT NULL; -- Add NOT NULL to existing column ALTER TABLE students MODIFY COLUMN phone VARCHAR(15) NOT NULL; -- Change ENUM options ALTER TABLE courses MODIFY COLUMN level ENUM('Beginner','Intermediate','Advanced','Expert') DEFAULT 'Beginner'; -- PostgreSQL uses CHANGE differently: -- ALTER TABLE students ALTER COLUMN name TYPE VARCHAR(200); -- ALTER TABLE students ALTER COLUMN name SET NOT NULL;
DROP COLUMN — Remove columnsMySQL
-- Drop a single column (⚠️ DATA IS LOST FOREVER) ALTER TABLE students DROP COLUMN linkedin_url; -- Drop multiple columns at once ALTER TABLE students DROP COLUMN github_url, DROP COLUMN portfolio_url; -- ⚠️ CANNOT drop a column that is a Foreign Key -- Must drop the FK constraint FIRST, then drop the column ALTER TABLE enrollments DROP FOREIGN KEY fk_enroll_student, -- drop FK first DROP COLUMN student_id; -- then drop column
ADD / DROP Constraints with ALTER TABLEMySQL
-- Add a UNIQUE constraint to existing column ALTER TABLE students ADD CONSTRAINT uq_student_phone UNIQUE (phone); -- Add a CHECK constraint ALTER TABLE students ADD CONSTRAINT chk_age CHECK (age BETWEEN 16 AND 80); -- Add a FOREIGN KEY ALTER TABLE courses ADD CONSTRAINT fk_courses_instructor FOREIGN KEY (instructor_id) REFERENCES instructors(instructor_id); -- Drop a UNIQUE constraint ALTER TABLE students DROP INDEX uq_student_phone; -- UNIQUE constraints are indexes in MySQL -- Drop a CHECK constraint (MySQL 8+) ALTER TABLE students DROP CHECK chk_age; -- Drop a FOREIGN KEY ALTER TABLE courses DROP FOREIGN KEY fk_courses_instructor;
RENAME Column (MySQL 8.0+)MySQL 8+
-- Rename a column (MySQL 8.0+) ALTER TABLE students RENAME COLUMN dob TO date_of_birth; -- For older MySQL: use CHANGE (must re-specify data type) ALTER TABLE students CHANGE dob date_of_birth DATE; -- old_name new_name datatype -- Rename entire table ALTER TABLE students RENAME TO learners; -- Or use RENAME TABLE (moves between databases too!) RENAME TABLE students TO learners; RENAME TABLE guvi_lms.students TO guvi_prod.learners;

Topic 10

DROP vs TRUNCATE vs DELETE — Know the Difference

Three commands can remove data — but they work very differently. This is one of the most commonly asked interview questions. Know these differences cold.

PropertyDROP TABLETRUNCATE TABLEDELETE FROM
What it removesEntire table (structure + data)All rows (data only, keeps structure)Specific rows (or all if no WHERE)
Table structure after❌ Gone — table doesn't exist✅ Table still exists, empty✅ Table still exists
Can be rolled back?❌ No (auto-commit DDL)❌ No in MySQL (DDL)✅ Yes (DML — can ROLLBACK)
Triggers fire?❌ No❌ No✅ Yes (DELETE triggers run)
WHERE clause?❌ N/A❌ Not allowed✅ Required for partial delete
AUTO_INCREMENT reset?N/A (table gone)✅ Resets to 1❌ Continues from last value
Performance on large tablesFast (single operation)Very fast (no row-by-row)Slow (logs every row deleted)
Honors FK constraints?❌ May fail if referenced❌ May fail if referenced✅ Yes, honored
CategoryDDLDDL (MySQL), DML (PostgreSQL)DML
💀 DANGER ZONE — Irreversible Commands
DROP TABLE students; → Table is gone. All data, constraints, indexes — permanently deleted. No undo.
TRUNCATE TABLE students; → All rows gone. Structure remains but data is irrecoverable in MySQL.

Best practices:
✅ Always backup before DROP or TRUNCATE in production
✅ Use DROP TABLE IF EXISTS to avoid errors in scripts
✅ In production, prefer DELETE WHERE ... + ROLLBACK capability
✅ If you must TRUNCATE, disable FK checks first: SET FOREIGN_KEY_CHECKS = 0;
DROP, TRUNCATE, DELETE — Syntax & UsageMySQL
-- DROP TABLE (complete removal) DROP TABLE students; -- errors if doesn't exist DROP TABLE IF EXISTS students; -- safe version DROP TABLE IF EXISTS enrollments, students; -- drop multiple -- TRUNCATE (empty all rows, keep structure) TRUNCATE TABLE students; -- removes all rows fast -- To truncate a table with FK children: SET FOREIGN_KEY_CHECKS = 0; TRUNCATE TABLE students; SET FOREIGN_KEY_CHECKS = 1; -- DELETE (selective row removal — DML, covered in Ch 6) DELETE FROM students WHERE student_id = 5; -- specific row DELETE FROM students WHERE city = 'Delhi'; -- multiple rows DELETE FROM students; -- all rows (recoverable)

RENAME — Tables & Columns

Renaming keeps data intact while changing the name. It's common during schema refactoring, migrations, and feature updates.

RENAME — All VariationsMySQL / PostgreSQL
── MySQL ────────────────────────────────────────── -- Rename a TABLE (Method 1 — MySQL specific) RENAME TABLE students TO learners; -- Rename multiple tables atomically RENAME TABLE students TO learners, courses TO programs, instructors TO mentors; -- Rename a table (Method 2 — using ALTER TABLE) ALTER TABLE students RENAME TO learners; -- Rename a COLUMN (MySQL 8.0+) ALTER TABLE students RENAME COLUMN dob TO date_of_birth; -- Rename column (older MySQL — must repeat data type!) ALTER TABLE students CHANGE dob date_of_birth DATE; ── PostgreSQL ──────────────────────────────────── -- Rename table ALTER TABLE students RENAME TO learners; -- Rename column ALTER TABLE students RENAME COLUMN dob TO date_of_birth; -- Rename database (PostgreSQL) ALTER DATABASE guvi_lms RENAME TO guvi_production;
⚠️
Rename ripple effects
Renaming a table/column doesn't automatically update: stored procedures, views, application code, BI reports, or foreign key references in other tools. Always search your codebase for the old name after renaming in the DB!

Topic 12

CREATE INDEX / DROP INDEX

An index is a separate data structure that the database maintains to speed up queries. Without indexes, every query scans every row (full table scan). With indexes, the database jumps directly to the relevant rows — like using a book's index instead of reading every page.

❌ Without Index — Full Table Scan

Query: WHERE city = 'Chennai'

🔍 Row 1: Mumbai — not Chennai, skip
🔍 Row 2: Delhi — not Chennai, skip
✅ Row 3: Chennai — MATCH!
🔍 Row 4: Bangalore — skip
🔍 Row 5: Hyderabad — skip
Checks ALL 1,000,000 rows even for 10 matches. O(n) — very slow!
✅ With Index on city — Index Seek

B-tree index: cities sorted alphabetically

📖 Index: Ahmedabad → rows [45,890]
📖 Index: Bangalore → rows [12,67,234]
⚡ Chennai → rows [3,88,299,401] → Jump!
📖 Delhi → rows [2,15,789]
Jumps directly to Chennai rows using B-tree. O(log n) — lightning fast!
CREATE INDEX — All Types & SyntaxMySQL
-- Basic single-column index CREATE INDEX idx_student_city ON students(city); -- UNIQUE index (enforces uniqueness + speeds up lookup) CREATE UNIQUE INDEX idx_student_email ON students(email); -- Composite index (multiple columns) CREATE INDEX idx_city_course ON students(city, course); -- Useful for: WHERE city = 'Chennai' AND course = 'SQL' -- ⚠️ Column order matters! Index on (city, course) helps: -- WHERE city = ? ✅ -- WHERE city = ? AND course = ? ✅ -- WHERE course = ? ❌ (skips left prefix) -- FULLTEXT index (for text search) CREATE FULLTEXT INDEX idx_ft_description ON courses(description); -- Use with: MATCH(description) AGAINST('machine learning') -- Create index inline during table creation (recommended) CREATE TABLE orders ( order_id INT PRIMARY KEY AUTO_INCREMENT, customer_id INT NOT NULL, status VARCHAR(20), created_at TIMESTAMP, INDEX idx_customer (customer_id), -- FK index INDEX idx_status_date (status, created_at) -- composite ); -- View indexes on a table SHOW INDEX FROM students; SHOW INDEXES FROM students; -- Drop an index DROP INDEX idx_student_city ON students; ALTER TABLE students DROP INDEX idx_student_city;
⚠️
Index Tradeoffs — Don't over-index!
Every index: ✅ Speeds up SELECT with that column in WHERE/JOIN/ORDER BY
Every index: ❌ Slows down INSERT/UPDATE/DELETE (index must be maintained)
Every index: ❌ Uses disk space

Rule of thumb: Index columns you filter or join on regularly. Don't index every column. A table with 10 indexes is a sign of over-engineering — aim for 3–5 carefully chosen indexes per table.
