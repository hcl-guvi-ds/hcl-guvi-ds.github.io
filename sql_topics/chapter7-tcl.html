TCL — Transaction Control Language

When ₹50,000 moves from one bank account to another, what happens if the power cuts out halfway? Does the money vanish? TCL prevents disasters like this. Transactions, ACID properties, and isolation levels are the foundation of every reliable database system — from HDFC to Flipkart to IRCTC. Understand them deeply here.

Topic 1

TCL Overview — Controlling Data Changes

TCL (Transaction Control Language) manages the execution of transactions — groups of SQL statements that should either all succeed or all fail together. TCL gives you control over when changes become permanent and the power to undo them if something goes wrong.

🚀
BEGIN
Start a transaction
✅
COMMIT
Save changes permanently
↩️
ROLLBACK
Undo all changes
📍
SAVEPOINT
Create a checkpoint
🗂️
RELEASE
Remove a savepoint
📌
Auto-commit mode — MySQL's default behaviour
By default in MySQL, every single DML statement (INSERT/UPDATE/DELETE) is automatically committed immediately — you don't need to type COMMIT. This is autocommit mode.

To use explicit transactions, either:
• Start with START TRANSACTION; (disables autocommit for that block)
• Or set globally: SET autocommit = 0; (then EVERY statement needs COMMIT)
🇮🇳 Why Transactions Matter — Real Indian Examples
Scenarios where transactions are non-negotiable
🏦 HDFC Net Banking transfer: Debit ₹50,000 from sender + Credit ₹50,000 to receiver = must happen together or not at all
🎫 IRCTC ticket booking: Reduce available seats + charge payment + send confirmation — all or nothing
📦 Flipkart order placement: Reduce stock quantity + create order + deduct wallet balance — atomic
🎓 GUVI batch enrollment: Create enrollment record + send welcome email trigger + update batch count — together

Topic 2

BEGIN, COMMIT & ROLLBACK

These three commands form the core lifecycle of every transaction. Think of them as Start, Save, and Undo.

Transaction Lifecycle — Complete SyntaxMySQL / PostgreSQL
── Starting a transaction ───────────────────────────────── START TRANSACTION; -- MySQL preferred BEGIN; -- PostgreSQL / works in MySQL too BEGIN WORK; -- SQL standard form ── Inside the transaction ───────────────────────────────── UPDATE accounts SET balance = balance - 50000 WHERE account_id = 101; UPDATE accounts SET balance = balance + 50000 WHERE account_id = 202; INSERT INTO transactions_log (from_acc, to_acc, amount, txn_date) VALUES (101, 202, 50000, NOW()); ── Path 1: Everything went well → COMMIT ────────────────── COMMIT; -- All 3 statements now permanent, visible to all COMMIT WORK; -- SQL standard form ── Path 2: Something went wrong → ROLLBACK ──────────────── ROLLBACK; -- All 3 statements undone, DB back to original state ROLLBACK WORK; -- SQL standard form ── Check autocommit status ──────────────────────────────── SHOW VARIABLES LIKE 'autocommit'; SET autocommit = 0; -- disable autocommit (manual COMMIT needed) SET autocommit = 1; -- re-enable autocommit (default)
✏️ Try It — Commit vs Rollback Paths Exercise 1
💡 A ROLLBACK undoes EVERYTHING since the last START TRANSACTION — even multiple statements across multiple tables
Transaction Flow
🚀 START TRANSACTION — autocommit suspended
✅ INSERT INTO scores_archive — rows staged (not yet committed)
✅ UPDATE students — scores bumped by 5 (not yet committed)
📋 SELECT verifies: all B2024 scores increased by 5
✅ COMMIT — both INSERT and UPDATE are now permanent and visible to all connections
↩️ OR: ROLLBACK — both operations are reversed; DB returns to pre-transaction state
⚠️
What automatically causes a ROLLBACK?
If a session disconnects unexpectedly (crash, timeout, kill), MySQL automatically rolls back any open uncommitted transaction — protecting data integrity.

DDL statements (CREATE/ALTER/DROP/TRUNCATE) in MySQL cause an implicit COMMIT — they commit all pending DML before executing. This is why you should never mix DDL inside a DML transaction in MySQL.

Topic 3

SAVEPOINT — Partial Rollback Checkpoints

A SAVEPOINT creates a named checkpoint inside a transaction. You can then ROLLBACK TO that checkpoint, undoing only the operations after it — while keeping earlier changes. Think of it as a video game's "save progress" point.

SAVEPOINT — Full Syntax & UsageMySQL
START TRANSACTION; -- Step 1: Insert basic student record INSERT INTO students (name, city, course, score, fee_paid) VALUES ('Arjun Mehta', 'Pune', 'SQL', 85, 12000); SAVEPOINT after_student_insert; -- checkpoint 1: student row exists -- Step 2: Try to create enrollment INSERT INTO enrollments (student_id, course_id, fee_paid) VALUES (LAST_INSERT_ID(), 1, 12000); SAVEPOINT after_enrollment; -- checkpoint 2: enrollment also exists -- Step 3: Try to process payment (this fails!) UPDATE payments SET status = 'confirmed' WHERE student_id = LAST_INSERT_ID() AND amount = 12000; -- Oops! Payment record doesn't exist yet. Roll back only the payment step. ROLLBACK TO SAVEPOINT after_enrollment; -- Now: student row ✅ | enrollment ✅ | payment ❌ (rolled back) -- Re-try payment with correct data INSERT INTO payments (student_id, amount, status, created_at) VALUES (LAST_INSERT_ID(), 12000, 'pending', NOW()); -- All good now → COMMIT everything COMMIT; ── Cleanup savepoints ────────────────────────────────────── -- RELEASE SAVEPOINT removes a savepoint (frees memory) -- No data is committed or rolled back RELEASE SAVEPOINT after_student_insert; -- optional cleanup

💡 SAVEPOINTs let you recover from intermediate failures without restarting the entire multi-step process
SAVEPOINT Flow Visualized
🚀 START TRANSACTION — begin
✅ Phase 1: INSERT staging data → SAVEPOINT sp_after_staging created
✅ Phase 2: CLEAN/TRANSFORM → SAVEPOINT sp_after_cleaning created

Topic 5

ACID Properties — The 4 Pillars of Reliable Databases

ACID is the set of properties that guarantee database transactions are processed reliably. Every production database system (MySQL, PostgreSQL, Oracle) follows ACID. These properties are why you can trust your bank balance, your flight booking, and your exam score.

⚛️
A
Atomicity
A transaction is an all-or-nothing unit. Either ALL statements succeed and commit, or NONE of them apply. There is no "half-done" state.
Bank transfer: Debit succeeds but credit fails → Atomicity rolls back the debit too. Money never disappears.
🎯
C
Consistency
A transaction takes the database from one valid state to another valid state. All constraints, rules, and triggers are enforced throughout.
Enrolling 1 student reduces available_seats by 1 AND updates enrollment count — total never inconsistent.
🔒
I
Isolation
Concurrent transactions are isolated from each other. While a transaction is in progress, other transactions cannot see its uncommitted changes.
Two people book the last seat simultaneously — only one succeeds; the other sees the seat as taken.
🏔️
D
Durability
Once committed, a transaction survives permanently — even if the system crashes, loses power, or has hardware failure.
After you see "Payment Successful" on IRCTC, a power cut cannot un-book your ticket. It's on disk forever.
✏️ Try It — ACID in Code: Seat Booking Exercise 3
💡 FOR UPDATE creates an exclusive lock — prevents two people from booking the same last seat simultaneously
ACID Properties in Action
⚛️ ATOMICITY: All 4 operations (check, update seats, insert ticket, deduct wallet) form one atomic unit
🎯 CONSISTENCY: Before = 1 seat available. After = 0 seats available + 1 confirmed ticket. Sum is preserved.
🔒 ISOLATION: FOR UPDATE lock prevents Session B from seeing 1 available seat while Session A is booking it
🏔️ DURABILITY: COMMIT writes to WAL (Write-Ahead Log) — survives power failure, server restart, or crash

Topic 6

Isolation Levels — Controlling Concurrent Access

Perfect isolation (every transaction sees a completely private copy of the database) would eliminate all concurrency issues — but it would be incredibly slow. Isolation levels let you choose a balance between safety and performance.

Isolation Level Dirty Read Non-Repeatable Read Phantom Read Performance Use Case
LOWEST READ UNCOMMITTED ✗ Possible ✗ Possible ✗ Possible ⚡⚡⚡ Fastest Real-time analytics where approximate data is OK
DEFAULT READ COMMITTED ✓ Protected ✗ Possible ✗ Possible ⚡⚡ Fast Most web applications, PostgreSQL default
MySQL DEFAULT REPEATABLE READ ✓ Protected ✓ Protected ✗ Possible* ⚡ Moderate Banking, e-commerce, inventory management
STRICTEST SERIALIZABLE ✓ Protected ✓ Protected ✓ Protected 🐢 Slowest Financial settlements, regulatory reporting

* MySQL's InnoDB prevents phantom reads in REPEATABLE READ using gap locks — making it effectively similar to SERIALIZABLE in many cases

Setting Isolation LevelMySQL
-- Set for the current session only SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED; SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ; SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE; SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED; -- Set globally (affects all new connections) SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ; -- Set for just the next transaction SET TRANSACTION ISOLATION LEVEL SERIALIZABLE; START TRANSACTION; -- this transaction runs with SERIALIZABLE -- Check current isolation level SELECT @@transaction_isolation; -- MySQL 8+ SELECT @@tx_isolation; -- MySQL 5.7 SHOW VARIABLES LIKE 'transaction_isolation';
💡
Which isolation level should you use?
Most applications: READ COMMITTED (PostgreSQL default) or REPEATABLE READ (MySQL default) — good balance
Financial transactions: SERIALIZABLE — maximum safety, accept the performance cost
Analytics / reporting: READ UNCOMMITTED — fastest, approximate data is acceptable
Never change MySQL's default (REPEATABLE READ) without understanding the implications first

Topic 8

Deadlocks — When Transactions Block Each Other

A deadlock occurs when two or more transactions are each waiting for the other to release a lock — creating a circular dependency that no one can break. MySQL detects deadlocks automatically and kills one transaction (the "victim") to break the cycle.

💀 Classic Deadlock Scenario
Session A
🔒 HOLDS lock on Row 1
⏳ WANTS lock on Row 2
→ Deadlock! ←
Session B
🔒 HOLDS lock on Row 2
⏳ WANTS lock on Row 1

Session A holds Row 1 and waits for Row 2. Session B holds Row 2 and waits for Row 1. Neither can proceed. MySQL's deadlock detector breaks this by rolling back the transaction with the least "work" done (the victim). The victim gets an error: ERROR 1213: Deadlock found when trying to get lock.

Deadlock Example — Bank Transfers Gone WrongMySQL
── Session A ────────────────────────────────────────────── START TRANSACTION; UPDATE accounts SET balance = balance - 100 WHERE account_id = 1; -- LOCKS row 1 -- (waiting for row 2...) UPDATE accounts SET balance = balance + 100 WHERE account_id = 2; -- WAITS — B has row 2 locked! ── Session B (concurrent) ───────────────────────────────── START TRANSACTION; UPDATE accounts SET balance = balance - 200 WHERE account_id = 2; -- LOCKS row 2 -- (waiting for row 1...) UPDATE accounts SET balance = balance + 200 WHERE account_id = 1; -- WAITS — A has row 1 locked! DEADLOCK! ── MySQL detects after innodb_lock_wait_timeout (~50 sec) ─ ── Kills one session: ERROR 1213 Deadlock found ── The other session proceeds normally ── ✅ Prevention: Always acquire locks in the SAME order! ── -- Both sessions: lock row 1 FIRST, then row 2 -- This ensures they never create circular dependency -- Check deadlock info: SHOW ENGINE InnoDB STATUS\G
🛡️
5 Strategies to Prevent Deadlocks
1. Consistent lock ordering — Always acquire locks in the same order (e.g., always lock lower account_id first)
2. Keep transactions short — Long transactions hold locks longer, increasing collision chance
3. Index your WHERE columns — Full table scans lock too many rows; indexes narrow the lock scope
4. Use SELECT FOR UPDATE only when needed — Reduce the number of rows you lock
5. Retry on deadlock — Catch ERROR 1213 in application code and retry the transaction
✏️ Try It — Deadlock-Safe Transfer Pattern Exercise 4
💡 Consistent lock ordering is the single most effective deadlock prevention technique
Deadlock-Safe Pattern Analysis
✅ Transfer A→B: Locks 1001 first, then 2002. Executes, commits.
✅ Transfer B→A: Also locks 1001 first, then 2002. Waits if A is in progress, then executes.
🛡️ No deadlock possible! Both sessions acquire locks in the SAME ORDER — no circular dependency can form.
💡 Without consistent ordering: Session A (1001→2002) concurrent with Session B (2002→1001) = deadlock!

Optimistic vs Pessimistic Locking

When multiple users may try to modify the same data, you need a strategy. Pessimistic locking assumes conflicts will happen and locks data upfront. Optimistic locking assumes they won't and checks at the end.

🔒 Pessimistic Locking

Lock the row as soon as you read it. No one else can modify it until you release. Uses SELECT ... FOR UPDATE.

✅ When to use: High conflict scenarios — inventory management, seat booking, bank transfers. Conflicts are expected and costly.

❌ Downside: Locks rows for the duration of the transaction — others must wait. Reduces concurrency. Risk of deadlocks.

START TRANSACTION; -- LOCK the row immediately on read SELECT * FROM seats WHERE seat_id = 'B2-34' FOR UPDATE; -- exclusive lock -- Modify it (you're safe — locked) UPDATE seats SET is_booked = TRUE WHERE seat_id = 'B2-34'; COMMIT; -- lock released
🤝 Optimistic Locking

Read without locking. Add a version column. On update, check the version matches what you read. If someone else changed it, abort and retry.

✅ When to use: Low conflict scenarios — product reviews, user profiles, blog posts. Conflicts are rare. High concurrency is needed.

✅ Benefit: No locks during read. Much higher concurrency. No deadlocks possible.

-- Step 1: Read with version SELECT id, name, stock, version FROM products WHERE id = 101; -- Gets: stock=10, version=5 -- Step 2: Update ONLY if version matches UPDATE products SET stock = stock - 1, version = version + 1 WHERE id = 101 AND version = 5; -- optimistic check! -- Step 3: Check rows affected -- IF ROW_COUNT() = 0: -- Someone else updated it! -- RETRY the whole operation
AspectPessimisticOptimistic
Lock timingAt READ time (SELECT FOR UPDATE)At WRITE time (check version)
ConcurrencyLow (others wait for lock)High (no waiting during read)
Deadlock riskYes (if locking order inconsistent)None (no database locks)
Retry needed?No (lock prevents conflict)Yes (if version mismatch)
Best forHigh contention, critical dataLow contention, read-heavy
ExamplesBank transfer, seat booking, flash saleUser profile update, blog post edit

Transaction Safety in Python

As a data scientist, you'll often run DML from Python scripts — for ETL pipelines, ML data prep, or automation. Proper transaction handling in Python is critical for production reliability.

mysql-connector-python — Transaction PatternPython 3
import mysql.connector from mysql.connector import Error def bank_transfer(from_acc, to_acc, amount): conn = None try: conn = mysql.connector.connect( host='localhost', database='hdfc_bank', user='app_user', password='***', autocommit=False # Disable autocommit — we control commits! ) cursor = conn.cursor(dictionary=True) # Step 1: Lock and check sender balance cursor.execute( "SELECT balance FROM accounts WHERE account_id = %s FOR UPDATE", (from_acc,) ) sender = cursor.fetchone() if not sender or sender['balance'] < amount: raise ValueError(f"Insufficient balance. Available: {sender['balance']}") # Step 2: Debit sender cursor.execute( "UPDATE accounts SET balance = balance - %s WHERE account_id = %s", (amount, from_acc) ) # Step 3: Credit receiver cursor.execute( "UPDATE accounts SET balance = balance + %s WHERE account_id = %s", (amount, to_acc) ) # Step 4: Log transaction cursor.execute( """INSERT INTO transfer_log (from_acc, to_acc, amount, status) VALUES (%s, %s, %s, 'success')""", (from_acc, to_acc, amount) ) # All steps succeeded → commit permanently conn.commit() print(f"✅ Transfer of ₹{amount} from {from_acc} to {to_acc} successful") return True except (Error, ValueError) as e: if conn: conn.rollback() # Undo everything — atomicity preserved print(f"❌ Transfer failed, rolled back: {e}") return False finally: if conn and conn.is_connected(): cursor.close() conn.close() # Always close connection! # Usage bank_transfer(1001, 2002, 25000) bank_transfer(9999, 2002, 99999999) # will rollback (insufficient)
SQLAlchemy — Context Manager Pattern (Recommended)Python 3 + SQLAlchemy
from sqlalchemy import create_engine, text from sqlalchemy.orm import sessionmaker engine = create_engine("mysql+mysqlconnector://user:pass@localhost/guvi_lms") Session = sessionmaker(bind=engine) # Method 1: with statement — CLEANEST pattern with Session() as session: with session.begin(): # auto-commits on exit, rollbacks on exception session.execute(text( "UPDATE students SET score = score + 5 WHERE batch = :batch" ), {"batch": "B2024"}) session.execute(text( "INSERT INTO audit_log (action, table_name) VALUES (:a, :t)" ), {"a": "score_update", "t": "students"}) # ↑ Automatically COMMITs here (or ROLLBACKs if exception) # Method 2: pandas with transaction import pandas as pd with engine.begin() as conn: # auto-commit on success, rollback on error df = pd.read_sql("SELECT * FROM students WHERE batch='B2024'", conn) df['grade'] = df['score'].apply(lambda x: 'A' if x>=90 else 'B') df[['student_id', 'grade']].to_sql( 'student_grades', conn, if_exists='replace', index=False )
